## 现状分析

- 属性计算散落三处且口径不一的风险已经存在：`fitModule`（PG/CPU × 工程技能）、FittingWindow（同公式复算 + 直接读船体静态 defense/mobility）、BattleScene（`gunneryMult`/`navigationMult` 在 render 期算好，武器伤害与速度各自乘）。
- 引擎对被动模块**零支持**：BattleScene 把全部已装配模块塞进 `modulesStateRef`（战斗模块架 UI + 处理循环），被动件会出现在模块架上却无任何行为。
- v0.4 引入的运行时 `playerSigMult`（MWD）与本版的静态 sig 修正（护盾扩展）是两层概念，必须区分：静态修正进 `getEffectiveStats`，运行时倍率保持在 step 闭包里逐帧计算。
- 存档层无需改动：模块以 id 序列化，v2 格式自动覆盖新模块。

## 设计决策

### 1. `src/lib/shipStats.js` — 单一属性口径

```js
getEffectiveStats(ship, fittedModules, skills) → {
  resources: { pg, cpu, cap_capacity, cap_recharge },   // 含工程技能 + Co-Processor
  defense:   { shield/armor/hull: { hp, em, th, kin, exp }, sig_radius },
  mobility:  { base_speed /* 含导航技能 */, agility, mass },
  damageMult: { hybrid_weapon, missile_weapon }          // 含炮术技能
}
```

纯函数、无 store 依赖。求值规则（**实现与文档都按此表述**）：

```
final = (base + Σ add 修正) × Π mult 修正 × 相关技能 skillMult
```

- 全部 `add` 先于全部 `mult`；mult 之间连乘（v0.5 无堆叠惩罚，PRD 已确认）。
- 抗性按百分比点数 `add`，结果 `min(90)` 封顶；不支持对抗性的 `mult`（文档写明）。
- 技能映射：engineering→resources.pg/cpu，navigation→mobility.base_speed，gunnery→damageMult.*（基线 1）。

### 2. 修正声明格式（`modules.js`）

```js
passive: true,
modifiers: [
  { stat: 'defense.armor.hp',     op: 'add',  value: 150 },
  { stat: 'defense.armor.resists', op: 'add',  value: 12 },   // 组简写 → em/th/kin/exp 各 +12
  { stat: 'damage.hybrid_weapon', op: 'mult', value: 1.10 },
]
```

- 路径是相对有效属性形状的点记法；`<layer>.resists` 是组简写（省去 EANM/DC 写 4×3 条）。
- 支持的路径白名单（实现时集中定义，未知路径 console.warn 跳过而非崩溃）：三层 hp、三层 resists、sig_radius、base_speed、agility、resources.pg/cpu、damage.hybrid_weapon/missile_weapon。
- 限装：模块可声明 `maxPerShip: 1`（Damage Control），`fitModule` 按 **module id** 计数校验（未来 DC II 与 DC I 互斥需要 group 字段——记入 v0.7 注意事项，本版不做）。

### 3. 三处接入（消灭平行公式）

1. **`fitModule`**：装配校验改为"假定装入后"调用 `getEffectiveStats`（含候选模块）——`usedPG ≤ eff.resources.pg && usedCPU ≤ eff.resources.cpu`。含候选是关键：Co-Processor 提升的是容量自身，先装它再装重件的顺序必须可行。同时加 `maxPerShip` 校验。
2. **`unfitModule`**：卸载前用"移除后"的有效属性复查，若剩余装配超限（卸 Co-Processor 的场景）→ `alert` 并拒绝卸载（PRD 开放问题的落地决策：阻止优于自动连锁卸载，行为可预期）。
3. **FittingWindow**：删除本地 PG/CPU 公式与 `skillMult` 导入，全面改读 `getEffectiveStats`；Defense 组显示有效 HP + 每层抗性紧凑串（如 `50/35/25/10`）；Navigation 显示有效速度与敏捷；新增 Offense 组显示每武器类型 DPS 倍率（×1.21 这类）。**装配即时反馈是本版的核心体验**。
4. **BattleScene**：
   - render 期 `useMemo` 计算 `eff`；删除 `gunneryMult`/`navigationMult`（防止双重应用——这是本版最大的回归风险点，见风险 1）。
   - 初始化改用 eff：`playerHp`/HUD max/电容容量与回充/基础速度/基础 sig。
   - 武器伤害系数 = `eff.damageMult[mod.type]`。
   - 修理量封顶改为 eff 的最大 HP（装甲板加的 HP 可被修理回去）。
   - **`modulesStateRef` 只收非 passive 模块**：被动件不进战斗模块架、不进处理循环，战斗侧对被动模块零分支（本版验收核心承诺）。
   - 有效 sig = `eff.defense.sig_radius（静态，含扩展罩惩罚） × playerSigMult（运行时，MWD）`。

## 验证计划

**A. Node 纯函数测试（shipStats.js 直接导入）**：
1. add→mult 顺序：板(+150) + 纳米(hull ×0.9) 组合的精确值；2×MagStab → damageMult 1.21。
2. 抗性组简写展开正确；DC+EANM 叠加后抗性封顶 90 生效（构造极端值）。
3. 技能等价回归：仅技能无被动件时，eff 输出与旧公式（`base × skillMult`）逐字段一致——保证迁移不改变现有数值。
4. Co-Processor：cpu 容量 ×1.10；未知 stat 路径 → warn 不崩。
**B. store 集成测试（sed 副本）**：DC 第二件被拒；CPU 恰好不足 → 装 Co-Processor 后可装 → 卸 Co-Processor 被阻止（alert 捕获）；装/卸往返后 inventory 守恒；存档 v2 回归（被动件 id 往返）。
**C. SSR 测试**：FittingWindow 装板后 Armor HP 显示 base+150、装 2×MagStab 显示 ×1.21、扩展罩后 sig 显示 +8；被动件**不出现**在战斗模块架（BattleScene 不可 SSR，用"modulesStateRef 构造逻辑抽出后单测"或降级为 FittingWindow 侧断言——实现时定，倾向把 rack 过滤写成可导出的纯函数 `activeModules(fittedModules)` 顺带测掉）。
**D.** `npm run lint`、`npm run build`；浏览器实机（装板进战斗看初始装甲、修理封顶、扩展罩 sig 挨打体感）待分类器恢复。

## 风险

1. **双重应用**：gunnery/navigation 技能迁入 eff 后，BattleScene 里任何残留的 `gunneryMult`/`navigationMult` 乘法都是平方级放大——验证 A-3 的等价回归 + 全文检索这两个标识符必须清零。
2. `fitModule` 是被 UI 高频调用的核心 action，改造后必须跑 v0.3 全部 store 阶段回归（不只新增用例）。
3. 修理封顶从船体静态值切到 eff 后，若 BattleScene 仍持有旧 `activeShip.defense.*.hp` 引用会出现"修不满/修超"——检索 `activeShip.defense` 在 BattleScene 的全部出现点逐一替换。
4. 无堆叠惩罚下 4×低槽全输出可能成唯一解——按 PRD 结论先观察（用敌人多样性对冲），不预防性实装惩罚。
