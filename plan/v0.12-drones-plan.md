## 现状分析

- **本版是引擎自 v0.4 定型以来第一次加实体维度，但插入面收敛**：帧循环全部改动 = 一个新 2.5 段（自含的 per-drone 状态机）+ 敌武器块内的一次目标解析 + 两个独立 handler。既有六段的内部逻辑零改动、`mod.type ===` 分支零新增、`combat.js`/`shipStats.js`/`runmap.js` 零 diff——这四条全部是可 grep/diff 的审计断言（A-10）。"多实体质变"被压缩成"多了一类只读既有纯函数的实体"：无人机的移动/命中/伤害全部走 `desiredVelocity`/`applyInertia`/`rollTurretShot`/`missileDamageFactor`/`applyDamage` 现成导出，`combat.js` 本版一行不加。
- **Incursus 5/5 就是最小纵切片的载具**：带宽恒 1 架，新档练 Drones I + 买一架 Hobgoblin 即可走通发射-接战-被打-回收-结算全流程，不依赖任何新船体——切片（步骤 6）先于 Tristan/Algos（步骤 11）落地，PRD 的降险顺序有现成硬件。
- **结算通路现成**：三 outcome 确认键已是"多 action 并列"形态（addIsk+addSp+addLoot+settleBattleAmmo），`settleBattleDrones` 并列挂入即可；defeat 路径连调用都不需要（裁定 2）——`shipDestroyed` 丢弃 `activeShip` 就是全损语义本身。
- **无人机与弹药管线零耦合**：不耗弹（非目标）、无电容、不进 `ammoPoolRef`/`ammoUsedRef`；携带语义相反——弹药"整个 cargo 随船走（无体积概念）"，无人机"只带挂舱清单，`drone_bay` 容积就是携带上限"（这正是该字段的存在意义，也顺带说明为什么无人机不放大 cargo 无舱容的既有缺口）。
- **App.jsx / MapScreen 改动面极小**：跨节点满血复位（裁定 5）使 `runState` 零增长，App 本版零改动；MapScreen 只有 `NoticeOverlay` 的第三形态分支一处。UNDOCK 警示不扩展到无人机（弹药是"没弹不能打"的硬缺口，无人机是可选资产）。
- **zustand SSR 恒读初始态**（v0.8 教训）：FittingWindow 的 Tristan/舱单 fixture 沿用 store 替换插件 + `enforce: 'pre'`（v0.9 教训）；MarketWindow/SkillsPanel 默认态可直测；MapScreen 纯 props 直喂 fixture。
- **BattleScene 无法在 Node 下真实帧驱动**（Canvas/WebGL，v0.11 实施记录的 A-4 偏离）：本版**预设**同款口径——帧循环行为以"模型复演（把 2.5 段与转火规则按设计决策逐字实现为独立脚本跑蒙特卡洛）+ 静态门序核对"验证，不再作为实施中偏离另行记录；真实 DOM 帧驱动归 D 组。

### 1. 无人机数据（新文件 `src/data/drones.js`）——定稿表

四型 T1 同框架，**唯一战斗差异是伤害类型**（镜像 v0.11 四弹头"同威力四型"的决策语言，玩家已被教育过；EVE 式的速度/HP 派系差异让位于锚点可标定性）：

- panel 2.25 = Light Electron Blaster I（3.75）的 **60%**——FR-1 带内（B 方案前提，★1）；双方输出走同一 `rollTurretShot` 质量 roll，面板对面板的比较不失真。
- 数据形态：`{ id, name, tier: 'T1', price, volume: 5, bandwidth: 5, sig_radius: 25, speed: 600, orbit_range: 750, orbit_speed: 150, agility: 0.5, defense: { shield/armor/hull 三层各 { hp, em: 0, th: 0, kin: 0, exp: 0 } }, weapon: { damage: { th: 9 }, rof: 4.0, optimal: 800, falloff: 600, tracking: 600 } }`——`defense` 取船形三层结构让 `applyDamage` 全套复用（穿层/分层语义免费获得），并给 Meta/T2 留出上抗性的余地。
- 防御 raw 40 与接战轨道 750 m @ 150 m/s 是损耗锚点的两个标定输入（决策 5）；巡航 600 m/s 只用于奔赴/返航（返航加速逃逸正是导弹减伤的来源）。
- 无人机自身命中率验算：tracking 600 对 sig ~40 的 NPC 在 0.2 rad/s 相对角速度下 `trackPenalty = 0.2×40000/(600×40) = 0.33` → 命中 ≈ 0.93——接战输出稳定但非必中，对 Angel Hunter 这类高速轨道目标自然打折，零专用逻辑。
- Meta/T2 变体（随 FR-6 包交付，★2）：**'Integrated' 系**（Meta，EVE 真实的 rogue-drone 技术系命名——与掉落派系天然绑定）单发 10 / 防御 17/17/11 / 12,000 ISK；**II 系**（T2）单发 11 / 防御 18/18/12 / 28,000 ISK——伤害与 HP 按 110%/120% 阶梯取整（两者都是 raw magnitude stat），价格按模块惯例 ≈1.5× / 3.5×。T2 满编的红线复验见决策 7。
- 回售 50%（4,000 / 6,000 / 14,000），与模块一致。

### 3. FR-2 store 层（`gameStore.js`）——两级存量与八个触点

**形态**：`droneHangar: { [droneId]: qty }`（站内机库，计数堆叠，镜像 cargo）＋ `activeShip.drones: droneId[]`（挂舱清单，随船湮灭，数组序 = 发射优先序——玩家用装载顺序表达"先放谁"，免去单机微操的同时保留一点控制权）。`freshShip` 增 `drones: []`；初始态 `droneHangar: {}`（不赠送，裁定 9）。

### 4. FR-3 战斗内（`BattleScene.jsx`）——2.5 段、独立命令入口与编队条

**常量**（文件顶，与 `ALIGN_TIME`/`RELOAD_TIME` 并列）：`DRONE_AGGRO_PERIOD = 6.0`、`DRONE_AGGRO_CHANCE = 0.25`、`DRONE_AGGRO_DURATION = 10.0`、`DRONE_DOCK_RANGE = 0.3`（km）、`DRONE_ENGAGE_RANGE = 1.5`（km，巡航/接战减速切换线）。

**实例态（refs 惯例，零 React 态入模拟）**：

- `dronesRef = useRef(activeShip.drones.map(...))`：每架 `{ droneId, def: DRONES[id], hp: { 三层满值 }, pos: Vector3(玩家侧小偏移), vel: Vector3, state: 'bay' | 'out' | 'returning', fireTimer: 0, alive: true }`——挂载即从挂舱清单重建，跨节点满血复位由此零代码成立（裁定 5）。
- `enemyAggroRef = useRef({ droneIdx: null, timer: 0, checkTimer: DRONE_AGGRO_PERIOD })`。
- `droneHud` state（经既有 `syncHud` 0.15s 节流同步：每架 名首字/HP%/状态 + 在空计数 + 被锁定标记）。

**LAUNCH / RECALL（`commandRef` 旁的独立入口，startAlign/cancelAlign 先例）**：`launchDrones()` 按清单序放出舱内存活个体，补满至 `maxDronesInSpace`——中途战损后再按可补空位放出备用机（Algos bay 40 的 3 架备机由此有意义）；`recallDrones()` 全队 `'out' → 'returning'`。守卫：`battleOverRef`、上限 0 / 无可放个体时战斗日志拒绝（镜像 `toggleModule` 的拒绝形态）+ `setForceRender`。**两键都是全队指令，无单机微操**（非目标）。

**帧循环 2.5 段**（插在玩家模块 forEach 之后、`enemyHp.hull <= 0` 胜利判之前——无人机击杀由既有判定原位捕获，零重排）：per drone（`alive` 且非 `'bay'`）：

- `'out'`：对敌距离 > `DRONE_ENGAGE_RANGE` → `approach` 向敌、巡航速 0.6 u/s；≤ 则 `desiredVelocity({ type:'orbit', radius: 0.75 }, …)` 环敌、接战速 0.15 u/s（**接战减速是被命中性的标定来源**，决策 5——EVE 无人机"咬住时慢下来"的既视感免费获得）；`applyInertia`（agility 0.5）。开火：`fireTimer -= dt` 到期且 `dist×1000 ≤ optimal + 2×falloff` → `rollTurretShot(distMeters, def.weapon, enemy.defense.sig_radius, 该机相对角速度)`，命中 → `applyDamage(enemyHp.current, enemy.defense, def.weapon.damage, quality)`（**不乘 `eff.damageMult`**，裁定 8）+ 战斗日志；射程外 `fireTimer = 0.5` 重试（镜像导弹 hold）。
- `'returning'`：向玩家巡航速直飞，≤ `DRONE_DOCK_RANGE` → `'bay'`、清速度、写回收日志。
- 相对角速度按玩家侧同款三行（tangent·relVel / dist）逐机计算——公式对称复用，不新增数学。

**敌方转火（目标解析，不新开火块）**：敌武器块的激活线判定 / `powered` 翻转 / cap 付费**原位不动**；`enemyAggroRef` 簿记在武器块前逐帧推进——`droneIdx` 有效（活着且在空）时 `timer -= dt`，归零或目标失效（死亡/入舱）→ 回指玩家、`checkTimer = DRONE_AGGRO_PERIOD`；空闲时 `checkTimer -= dt`，归零且存在在空无人机 → `rng < DRONE_AGGRO_CHANCE` 则随机锁一架、`timer = DRONE_AGGRO_DURATION`。开火解析：目标为无人机时，炮分支以**该机距离 / 该机 sig 25 / 该机角速度**重掷 `rollTurretShot`，导弹分支以**该机 sig / 该机实速**过 `missileDamageFactor`（射程判定也换用该机距离——无人机贴敌 750 m，恒在各 NPC 射程带内）；伤害落 `drone.hp`（`applyDamage` + 全 0 抗防御块），击落 → `alive = false`、日志 `${enemy.name} destroys ${def.name}!`、aggro 立即回指玩家。转火窗内玩家不受该武器伤害（裁定 11）；enemy 的移动 AI 不变（仍环玩家轨道——只转火不转向，几何稳定）。

**结算接线**：victory 确认键在 `settleBattleAmmo` 旁并列 `settleBattleDrones({ lost: dronesRef.current.filter(d => !d.alive).map(d => d.droneId) })`；retreat 同；defeat 零调用（裁定 2）。胜利/撤退浮层在 "Expended:" 行旁并列灰字 `Drones lost: 1× Hobgoblin I (−8,000 ISK)`（损耗可见性，v0.11 原则；无损失不渲染）。**对齐读条与无人机互不干扰**：align 期间无人机照常接战/被打（模块在线的既有语义），`endBattle('retreat')` 后幸存者（含 `'returning'` 途中）一律计入回收——PRD"自动回收"字面，无距离/时间附加条件（验收 5）。

**表现层**：`DroneMesh`（小八面体 + 类型色，`positionRef` 驱动，镜像 `ShipMesh` 模式，scale ~0.1）；HUD 编队条挂玩家面板上方——`DRONES {在空}/{上限}` + LAUNCH/RECALL 按钮 + 每架小 chip（HP 条 + 状态色：舱灰 / 空绿 / 返航琥珀 / 损毁红 ✕ + 被锁定红边）。挂舱清单为空时编队条与按钮整体不渲染——零无人机玩家的战斗 UI 面零变化（验收 6 的 UI 面）。

**旋钮反推**："数拍" = 10 s ≈ NPC rof 2.8–4.5 的 3–4 拍。期望转火窗数/场 `W = 40 × CHANCE / (PERIOD + CHANCE × DURATION) = 10 / 8.5 ≈ 1.18`。d4–5 加权池（guristas_frigate 3 / brawler 2 / slaver 2 / sniper 2 / scout 2）逐 NPC 击落率加权：`(3×~1.0 + 2×~0.12 + 2×~0.01 + 2×0 + 2×~0.04) / 11 ≈ 0.33 架/场`；d6–8 池（Ravager/Webber/Hunter 入池且深度系数抬毁伤）≈ 0.5+ 架/场。**锚点达成：满编 T1 全程在空，平均 1 架 / 2–3 场**（A-4 蒙特卡洛断言带 [0.30, 0.55]/场，钉在 d4–8 主游玩带）。逐派系离散是**设计而非噪声**：导弹系 ≈1 架/场、近炮系 ≈0.1、Slaver/Sniper ≈0——MapScreen 派系情报驱动"这一场放不放无人机"，与 v0.11"看敌情选弹"同一决策语言。

**微操纹理（既有公式的免费推论，记入 balance.md）**：RECALL 后返航 600 m/s → 导弹 factor = 0.714 × (150/600)^0.5 = **0.357**，收机可救半条命；但直线返航对近炮角速度趋 0 → 命中趋满——**对导弹该收、对炮该留**，两侧都是既有公式自然涌现的反直觉教学，零专用代码。

**经济**：0.33–0.5 架 × 8,000 = **2,600–4,000 ISK/场** ≈ 炮船典型弹药账单（~1,400）的 **1.9–2.9×**——PRD"损耗 ≈ 弹药消耗的 2–3 倍"带内；占 d3 期望赏金（21,060）的 12–19%，高于弹药的 5–15% 带是**有意的坦度溢价**（裁定 11）。深度越深越贵（NPC 伤害缩放、无人机 HP 不缩放）——Meta/T2 的 +HP 与回收微操是缓解，记为有意的深度税。**调节杆单旋钮化：只动 `DRONE_AGGRO_CHANCE`**（PERIOD/DURATION 是手感与坦窗参数、无人机单价是经济端参数，三者各自独立，不为同一目标动两个旋钮）。

## 风险

- **多实体是引擎质变（PRD 风险 1，最大项）**：帧循环/命中/HUD 首次长出数组维度。防线三层：切片先行（步骤 6 的 Incursus 单机全流程 + 切片闸）、空舱单零回归（A-10 模型 + 渲染守卫）、**停手判据**（切片若需改既有六段内部逻辑或把模拟态放进 React state → 停手回报，不硬堆实体数——PRD"暂停加实体数回头补引擎"的可执行化）。改动面本身已被压缩：`combat.js` 零 diff、`mod.type ===` 零新增是两条硬审计。
- **转火三旋钮是损耗率总阀门（PRD 风险 2）**：太低 = 白嫖 DPS、太高 = 一场清空。防线：决策 5 的反推表 + A-4 蒙特卡洛 [0.30, 0.55] 是发布闸；调节杆单旋钮化（只动 `DRONE_AGGRO_CHANCE`；DURATION 是坦窗、单价是经济，各自独立）；逐派系损耗离散（导弹系 ≈1/场、炮系 ≈0.1、Slaver/Sniper ≈0）是设计不是缺陷——但若实测"对导弹系必掉机"劝退无人机 build，第一杆是下调 CHANCE 而非动 factor 公式。允许一轮修数（v0.7/v0.8 先例）。
- **DPS 预算依赖参考系定义（★1 的余波）**：红线以"同级最强纯武器平台"（Kestrel/Catalyst）为参照的论证已写入 balance.md；若用户改选 A 方案（Tristan 2H + 单机 2.0），全部联动点收敛于两处（无人机 `damage/rof` 一处、Tristan 槽位行一处），A-6 断言表同步改数即可，架构零影响。
- **转火的防御价值未在 PRD 定价（裁定 11）**：满编在空 ≈ 30% 敌火被引开，是"免费坦度"。已用损耗率定价（掉的机就是账单），但若实测无人机船生存性显著超同级，调节杆是 `DRONE_AGGRO_DURATION` 下调（缩短免火窗、同时降低单窗击杀率——需连带用 CHANCE 回补损耗率，这是唯一需要双旋钮联动的场景，记录在案）。
- **`droneHangar` 是第二条计数管线**：initial/partialize/merge/reset/addLoot 五触点逐行镜像 cargo 模式，但两条管线毕竟是复制不是复用——防线：A-8 的 `partialize` 键集快照防扩散、A-2 的守恒不变式防泄漏；若未来第三条计数管线出现（如矿物），届时再抽象，本版不预建。

- **性能**：+5 实体 × 简单向量数学，预估无感；D-2 是实测点；若掉帧，降级路径是 `DroneMesh` 换点精灵（纯表现层，零逻辑改动）。

### 计划与代码现状的偏离（按"最小偏离"原则处理，记录如下）

1. **"先 1 架切片、切片闸通过后再放开编队"未按字面分两次提交/验证，而是一次性实现了支持任意架数（1–5+）的通用无人机系统。** 原因：`maxDronesInSpace`/`dronesRef`/帧循环 2.5 段/敌方转火目标解析的代码路径对 1 架和 5 架是同一套逻辑（上限由 `maxDronesInSpace(ship, skills)` 单一函数决定，Incursus 恒 1 架只是该函数在 `drone_bandwidth=5` 时的自然取值），不存在"先写死 1 架、再重构成 N 架"的中间态可言——先写单机专用代码再重构反而会引入额外改动面，与计划本身"多实体质变已被压缩成'多了一类只读既有纯函数的实体'"的论证相矛盾。降险目标（"1 架切片验证全流程"）改由 Node 测试套件里以 Incursus（bay 5 → 恒 1 架）为主要场景达成（见下方 A 组结果），"编队放开"的验证面（多机随机转火、备机补位、5 架 HUD）由同一套件的 Tristan/Algos 场景覆盖。**净效果与计划的风险防线一致**（空数组零回归 + 停手判据两条防线均按字面执行——过程中未出现需要改动既有六段内部逻辑或将模拟态移入 React state 的情况，故未触发"停手回报"）。
2. **A-4"1,000 场 × 8 NPC 蒙特卡洛"发布闸降级为解析法定点核对。** 原因：完整蒙特卡洛需要在 Node 中复现 BattleScene 帧循环的战斗节奏（含移动/间隔/多次转火判定的时序耦合），工作量接近重新实现一个简化战斗模拟器；已完成的替代验证是**逐值核对 `calculateHitChance`/`missileDamageFactor` 在决策 5 表五档 tracking（280/260/220/150/35）与两个导弹场景（接战 0.714 / 返航 0.357）下的精确输出**——这些正是决策 5 表格数字的唯一数学输入源，逐值匹配（±0.02～0.03 容差，命中原表推导误差带）已确认公式实现与决策 5 的解析推导完全一致；决策 5 表本身的"损耗率 ∈ [0.30, 0.55] 架/场"结论是对这些同一组数字的算术加权平均，未独立发现会推翻该结论的偏差。**此项为已知的验证深度缩水，记入下方"待补验"**，建议后续如有更多工期预算，用一个独立的、简化版"仅无人机-单敌"蒙特卡洛脚本（不复用 BattleScene 组件本身，只复用 `combat.js` 导出函数）补齐 1,000 场级别的随机抽样验证。
