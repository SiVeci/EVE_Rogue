## 代码核对结论（动笔前逐项核实）

**PRD 写于 v0.9/v0.10 实施之前，以下按代码现状与用户指示裁定（无 ★ 阻塞项；贴线裁定汇总于文末答复）**：

1. **PRD 未写三 outcome 的消耗结算**：v0.9 立下"撤退零 store 写入"的不变式，但打出去的弹收不回来。按用户指示裁定：**胜利/阵亡/撤退三种 outcome 均在确认键结算消耗**——撤退对该不变式的修订钉死为"只结弹药，不结其他"（赏金/SP/掉落照旧放弃），CLAUDE.md 同步改写。刷新语义与 v0.9 逐字一致：战斗中刷新 = 本场从未结算（无收益、无消耗、无指定变更），已清节点的消耗随 per-node 落账早已入档。
2. **PRD 未涉 v0.9 和平残骸路径与两个掉落显示端**：`App.jsx:55` 对无伏击 wreck 直接 `addLoot(node.loot)`；`MapScreen` 的 `NoticeOverlay` 与 BattleScene 胜利浮层都以 `MODULES[id]` 渲染条目——弹药束对象会让 NoticeOverlay 渲染出对象子节点直接崩溃。三处全部纳入路由/显示改造（决策 6）。
3. **`applyAmmoToWeapon` 落位 `src/lib/ammo.js` 而非 PRD 草拟的 `combat.js`**：combat.js 顶部 `import * as THREE`；而 gameStore（migrate 自动指定、fitModule 默认弹）、App（UNDOCK 警示）、FittingWindow（适配族列表）都需要弹药 helper，把 three.js 拖进 store 依赖图不洁。同层 lib 纯函数、rng 无关，测试惯例不受影响。
4. **"标准装药"命名为 Lead Charge S**：游戏全部内容用 EVE 真实命名（Antimatter/Lead/Iridium Charge S；Mjolnir/Inferno/Scourge/Nova Rocket 与 Light Missile），PRD 的"标准"即 Lead（EVE 里 ×1.0 基准档），赠弹与文档以此名书写。
5. **`fitModule` 自动指定默认弹**（超 PRD 字面、FR-6 精神）：装上武器即自动指定该族标准弹（混合炮→Lead，火箭→Scourge Rocket，轻导→Scourge Light Missile），玩家可改。"忘带子弹"的主要成因（装了武器忘了指定）从源头消除；PRD 的"必须指定才能开火"字面保留在开火门上。
6. **无弹匣模型**：弹药指定是指针不是弹夹——弹只在开火瞬间离开 cargo，换弹是纯 10 秒时间成本（不退弹不扣弹）。共享弹池（同弹种多门武器扣同一计数）因此天然成立，免去 per-磁匣簿记。
7. **弹药可按堆回售（50%，与模块一致）**：PRD 未提回售。裁定跟随既有先例——初始赠送的 7 件模块今天就能按 50% 卖掉，弹药不特殊化；迁移赠弹全数套现 ≈ 10,500 ISK 一次性泄漏，量级同今天卖光初始模块，接受。MarketWindow 右栏加 cargo 堆条目。
8. **balance.md v0.9 段的一行修订义务**："ISK-priced repair is deliberately deferred to the v0.11 ammo-economy pass"——v0.11 PRD（范围唯一权威）未纳入维修计价，该行到期失实。随本版 balance.md 增补顺手改为 "revisit after the ammo economy (v0.11) has been observed in play"，维修计价维持带外。
9. **换弹计时须防"重开归零"漏洞**：现有 `toggleModule` 在激活时 `mod.timer = 0`（立即触发）——若换弹复用 `timer`，玩家可"换弹→关→开"清零读条。裁定：换弹用独立的 `mod.reload` 字段，在帧循环**最顶部无条件递减**（不受 active 影响），开火门检查 `reload > 0` 则 0.5s 重试；`toggleModule` 零改动。副作用是关着的武器也在换弹（EVE 语义，更正确）。

## 现状分析

- **消耗品是第一条非模块库存管线**：模块 `inventory` 是逐件对象数组（同型多件靠重复引用），弹药是 `{ id: qty }` 堆叠计数——两者存档形态、市场交易粒度（逐件 vs 一手 100 发）、入库路由都不同，PRD "与既有管线分离、不动模块侧"的判断与代码一致。
- **战斗内改动面收敛在玩家武器两分支内部**：开火流程现状为"timer 到期 → 射程 hold（missile/neut）→ 电容门 → 付费置 timer → 结算"。弹药插入为"射程 hold 之后、电容门之前"的一个弹药门 + 结算时的 stats 替换 + 每发递减弹池 ref。`mod.type ===` 分支集合零新增（弹药不是模块类型），v0.7 立的管线审计口径继续成立。
- **结算通路**：三 outcome 均为"`endBattle` 冻结 → 浮层 → 确认键写 store + 导航回调"。消耗结算并列挂在确认键上，与 addIsk/addSp/addLoot 同一时序，刷新语义自动继承 v0.9。
- **`activeShip` 每武器指定的持久化形态**：`fittedModules.high` 数组内是共享的 `MODULES` 对象引用，不能在对象上挂弹药——指定必须是与 high 栏**按下标平行的数组** `activeShip.ammo`，由 fitModule/unfitModule/switchShip/merge 四处共同维护"长度恒等于 high 栏长度"的不变式。
- **blueprints 现状**：住在 `initialState`、不入 `partialize`——迁往数据文件是零存档影响的纯重构；`manufacture(bp)` 以 bp 对象为参，签名可保持。
- **zustand SSR 恒读初始态**（v0.8 教训）：`STARTER_CARGO` 进 `initialState` 后，MarketWindow 货架计数、FittingWindow cargo 摘要的默认态 SSR 直接可测；装配态 fixture 沿用 v0.8 store 替换插件 + v0.9 的 `enforce: 'pre'` 修正。
- **runmap 的 rng 顺序协议不受触碰**：`generateRunMap` 零改动；但 `rollLoot` 每表 roll 次数随弹药条目增加而变，同种子输出跨版本不稳定——既有事实（v0.7 起每版重生成期望），不构成协议破坏。

## 设计决策

### 1. 弹药数据（新文件 `src/data/ammo.js`）——修正矩阵与价格定稿

**混合炮装药**（`family: 'hybrid_s'`，适配全部 6 件 `hybrid_weapon`；修正乘在武器基础面板上，falloff/tracking/rof/cap 不动）：

| id | 名称 | 伤害 | optimal | 单价 | 一手（100 发） |
|----|------|------|---------|------|----------------|
| `antimatter_charge_s` | Antimatter Charge S | ×1.2 | ×0.6 | 60 | 6,000 |
| `lead_charge_s` | Lead Charge S（标准） | ×1.0 | ×1.0 | 35 | 3,500 |
| `iridium_charge_s` | Iridium Charge S | ×0.85 | ×1.6 | 45 | 4,500 |

**导弹弹头**（同威力四型 × 两族；弹头**替换**伤害对象为单一类型、总量守恒，并自带爆炸参数——绝不踩 v0.10 公式的缺省回退）：

| 族 | 适配 | explosion_radius / velocity | 四型 id（伤害类型） | 单价 | 一手 |
|----|------|------------------------------|---------------------|------|------|
| `rocket_s` | rocket_launcher_i/ii | **20 / 150**（= 现发射器字段，Scourge 逐字节等价现状） | `mjolnir_rocket`(EM) `inferno_rocket`(TH) `scourge_rocket`(KIN) `nova_rocket`(EXP) | 35 | 3,500 |
| `light_missile` | light_missile_launcher_i/ii | **40 / 170**（同上） | `mjolnir_light_missile` `inferno_light_missile` `scourge_light_missile` `nova_light_missile` | 35 | 3,500 |

全部 11 件 `tier: 'T1'`。数据形态：装药带 `charge: { damage_mult, optimal_mult }`，弹头带 `warhead: { damage_type, explosion_radius, explosion_velocity }`（不带绝对伤害——总量从发射器面板求和而来，T1/T2 发射器自动各得其值）。同文件导出 `AMMO_LOT = 100`（市场一手）、`DEFAULT_AMMO = { hybrid_s: 'lead_charge_s', rocket_s: 'scourge_rocket', light_missile: 'scourge_light_missile' }`、`STARTER_CARGO`、`AMMO_SAFETY_QTY = 100`（见决策 5）。

**武器侧**（`src/data/modules.js`）：10 件武器各加一行 `ammoFamily`（6 hybrid → `'hybrid_s'`，火箭 ×2 → `'rocket_s'`，轻导 ×2 → `'light_missile'`）；`energy_neut` 与其余主动件不加——弹药选择器只作用于 `hybrid_weapon`/`missile_weapon`，其他模块零触碰。发射器保留既有爆炸字段作防御性回退（正常路径永远被弹头值覆盖）。

### 2. `src/lib/ammo.js` —— 修正应用与四个纯 helper

```js
// 弹药修正应用于 getEffectiveStats 之后、开火结算之时 —— 不进被动件 modifiers
// 管线，因此结构性豁免 v0.10 堆叠惩罚（同 stat 的模块 mult 分组排位对它不可见）。
// 输出仍是米制面板（optimal 米、damage 原形状），BattleScene 的 dist×1000
// 比较约定零改动。null/未知弹药恒等返回（开火门另行拦截，本函数不管资格）。
export function applyAmmoToWeapon(weaponStats, ammo) {
  if (!ammo) return weaponStats;
  if (ammo.charge) return { ...weaponStats,
    optimal: weaponStats.optimal * ammo.charge.optimal_mult,
    damage: Object.fromEntries(Object.entries(weaponStats.damage)
      .map(([t, v]) => [t, v * ammo.charge.damage_mult])) };
  if (ammo.warhead) {
    const total = Object.values(weaponStats.damage).reduce((a, b) => a + b, 0);
    return { ...weaponStats,
      damage: { em: 0, th: 0, kin: 0, exp: 0, [ammo.warhead.damage_type]: total },
      explosion_radius: ammo.warhead.explosion_radius,
      explosion_velocity: ammo.warhead.explosion_velocity };
  }
  return weaponStats;
}
```

另导出：`compatibleAmmo(weaponModule)`（按 `ammoFamily` 过滤 AMMO 列表，选择器数据源）、`defaultAmmoIdFor(module)`（`DEFAULT_AMMO[module?.ammoFamily] ?? null`，fitModule/migrate/merge 共用）、`ammoReadiness(activeShip, cargo)`（→ `{ weapons, unassigned, dry }`，UNDOCK 警示与 FittingWindow 红字共用）。

**关键验算锚点**（写入 balance.md，A-1/A-2 逐值断言）：Lead 对任意武器 = 恒等（深比较相等，"基准不漂移"验收的字面）；Scourge Rocket 对 Rocket Launcher I 产出 `damage {0,0,18,0} + 20/150` = v0.10 现状逐字节一致；四型弹头对同一发射器伤害总和相等（32 组合守恒）；反物质电子爆能炮 optimal 1200→720、伤害 6/6→7.2/7.2；铱弹 125mm 磁轨 optimal 9,000→14,400——对 15 km 轨道的 Guristas Sniper，命中率从 ~6%（lead，距离惩罚 2²）跳到 ~97%，TTK 差一个数量级（验收 2 的"可测 TTK 差"）。

### 3. FR-2/FR-3 的战术数学（验收 1/2 的解析支撑）

- **Sansha 换 Mjolnir**：Slaver/Ravager 的 EM 抗性 5/5/10 对 KIN 35/30/40——同发射器 EM 弹头对 KIN 弹头的逐层伤害比 ≈ 1.46/1.38/1.50，全程 TTK 缩短 ≈ 30%，A-2 以 ≥20% 为断言阈（沿用 v0.8 弱抗清洁度的 20% 口径）。
- **四型弹头对无抗 dummy**：TTK 完全相等（守恒的行为面）。
- **反物质近轨**：1 km 轨道贴在 720m optimal 外沿、落 falloff 浅区（惩罚 (280/2000)²≈2%），实效 ≈ +19% dps——近战 build 的正确升级；对 9–15 km 风筝目标则命中崩塌，取舍真实。
- **弹药维度的全类型覆盖**：导弹族由此成为唯一能打全四类型的武器线（炮系锁 KIN/TH 是 PRD 明写的定位）——对 Sansha/Angel 的"正确答案"首次从防御端（抗性镀层）延伸到主动端。

### 4. FR-4 战斗内消耗 / 打空 / 换弹（`BattleScene.jsx`）

**实例态与初始化**（refs 惯例，零 React 态入模拟）：

- `modulesStateRef` 初始化改为**带高槽下标的枚举**：先走 `fittedModules.high` 逐下标（非 passive 者入列，武器额外挂 `{ ammoId: activeShip.ammo?.[i] ?? null, highIndex: i, reload: 0 }`），再拼 mid/low 的 `activeModules` 过滤——顺序与现状一致（high→mid→low），rack UI 零变化。
- `ammoPoolRef = useRef({ ...cargo })`（挂载快照全部机库存量——FR-4 "出击自动携带全部 cargo"）；`ammoUsedRef = useRef({})`。
- 常量 `RELOAD_TIME = 10.0`（文件顶，与 `ALIGN_TIME` 并列）。

**帧循环改动**（模块 forEach 内，仅玩家侧）：

1. 循环顶部（`!mod.active` 早退**之前**）：`if (mod.reload > 0) mod.reload -= dt`——换弹在关机时照走（核对结论 9 的防漏洞结构）。
2. 武器开火门（timer 到期、射程 hold 之后、电容门**之前**）：`reload > 0` → `timer = 0.5` 重试；`!ammoId || pool[ammoId] <= 0` → `mod.active = false` + 战斗日志 `[武器] Out of ammunition!` + `setForceRender`（镜像电容耗尽的自动关机路径——"打空自动停火并在 HUD 提示"的字面）。
3. 通过后照常付电容、置 timer，随后 `pool[ammoId]--`、`used[ammoId]++`，结算用 `fireStats = applyAmmoToWeapon(mod.stats, AMMO[mod.ammoId])` 替换两分支里的 `mod.stats`（hybrid 的 `rollTurretShot` 与 `applyDamage`、missile 的 `missileDamageFactor` 与 `applyDamage`；missile 射程 hold 仍读 `mod.stats.range`——弹头不改射程）。命中/未命中均耗弹（打出去的收不回来）；missile/neut 的射程 hold 与 ewar/propulsion 周期不耗弹。
4. `toggleModule` 加前置门：武器且（未指定或弹池为 0）→ 日志拒绝激活（镜像电容不足拒绝）。

**战斗中换弹**：模块 rack 的武器圆钮下方加余弹计数徽标（`×214`，经 `syncHud` 节流同步进新的 `ammoHud` state；0 时红色）与一个紧凑 `<select>`（`compatibleAmmo` 列表 + 各自余量）；选择 → `mod.ammoId = 新id; mod.reload = RELOAD_TIME` + 日志 + `setForceRender`，读条期间圆钮显示倒计时。站内（FittingWindow）切换即时无读条（EVE 泊站换弹语义）。

**结算时序**（三 outcome 确认键，与 addIsk 同位）：

| outcome | 确认键动作 | 依据 |
|---------|-----------|------|
| 胜利 | `settleBattleAmmo({ consumed, assignments })` + 既有 addIsk/addSp/addLoot + `onVictory(hp)` | 消耗扣账 + 战斗中换的弹回写站内指定（下一节点延续） |
| 撤退 | `settleBattleAmmo({ consumed, assignments })` + `onRetreat(hp)` | v0.9 "零写入"修订为"只写弹药结算"（核对结论 1） |
| 阵亡 | `settleBattleAmmo({ consumed })` + `shipDestroyed()` + `onDefeat()` | 消耗照扣（打出去的没了）；指定随船体湮灭不回写；**cargo 本身保留**（验收 6：弹药是机库物资） |

胜利/撤退浮层各加一行灰字消耗清单（`Expended: 34× Lead Charge S · 8× Scourge Rocket (−1,470 ISK)`）——5–15% 锚点要玩家看得见才成立；阵亡浮层不加（保持戏剧时刻干净，扣账静默进行）。

**PRD 开放问题"消耗粒度"落定：按发**。理由：① 经济锚点（决策 8）整套按发计算，按包（首次激活扣一手）会让 30 发的短战斗成本虚高到一整手、5–15% 带失真；② HUD 逐发下跳的计数正是 roguelike 资源张力的载体，"太琐碎"的担忧由呈现方式化解——只有打空/换弹产生战斗日志，逐发变化只走静默的徽标数字（`syncHud` 0.15s 节流），不刷屏。**"无弹降级开火"落定：不做**。理由：验收标准"打空弹药武器停火并提示"已按无降级定向；四重缓解（赠弹/自动指定/保底/UNDOCK 警示）把"忘带"压到极低频；×0.5 降级会把"配弹"从决策降格为增益，动摇本版的存在意义——作为回退阀记录于 balance.md，实测高频翻车再启用。

### 5. FR-6 + 存档 v2→v3（`gameStore.js`）

**新初始态**：`cargo: { ...STARTER_CARGO }`，`STARTER_CARGO = { lead_charge_s: 300, scourge_rocket: 150, scourge_light_missile: 150 }`（总值 21,000 ISK；3 门混合炮 ≈ 7–8 场、单发射器 ≈ 15 场——首个段绰绰有余，"新手第一课不能是忘带子弹"）。`freshShip` 增 `ammo: []`；`initialState` 删除 `blueprints`（迁数据文件，见决策 7）。

**新 actions**：

- `buyAmmo(ammoId)`：一手 `AMMO_LOT`（100 发）整批交易，`isk` 门槛 + `cargo` 累加（镜像 buyModule 静默失败）。
- `sellAmmo(ammoId)`：整堆按 `round(qty × price × 0.5)` 回售（核对结论 7）。
- `setWeaponAmmo(highIndex, ammoId)`：校验该下标是武器且 `AMMO[ammoId].family === module.ammoFamily`，不合法静默拒绝；FittingWindow 选择器唯一入口。
- `settleBattleAmmo({ consumed, assignments })`：`cargo[id] = max(0, qty − used)`（未知 id 忽略、不删零值键）；`assignments` 存在且 `activeShip` 在位时按 highIndex 回写 `ammo` 数组（逐项过 `setWeaponAmmo` 同款校验）。
- `fitModule`：高槽成功装配时 `ammo` 平行数组 push `defaultAmmoIdFor(module)`（武器得默认弹、非武器得 null）；`unfitModule`：高槽卸载时同步 `splice(index, 1)`。**不变式：`ammo.length === fittedModules.high.length`**，四个触点（fit/unfit/freshShip/merge）共同维护。
- `manufacture(bp)`：按 `MODULES[bp.produces]`（→ inventory，现状）/ `AMMO[bp.produces]`（→ `cargo += bp.qty`）路由，其余不动。
- `shipDestroyed`：免费 Incursus 保底分支内追加**弹药保底**——`DEFAULT_AMMO` 三族各补足到 ≥ `AMMO_SAFETY_QTY`（100 发）。取三族而非字面"一手标准弹"的理由：保底船必须配上玩家机库里**任意**幸存武器都打得响（导弹玩家拿 100 发 Lead 仍然哑火）；只在保底触发时生效，正常阵亡 cargo 原样保留。

**`SAVE_VERSION = 3`。`partialize`** 增 `cargo: state.cargo`；`activeShip` 块增 `ammo: state.activeShip.ammo`。

**`migrate` 链式化**（可执行迁移用例）：

```js
migrate: (persisted, version) => {
  if (version !== 1 && version !== 2) return undefined;   // 未知版本仍然丢弃
  const p = { ...persisted };
  if (version === 1) p.ownedShips = [p?.activeShip?.shipId ?? 'incursus'];  // 既有 v1 修复原样
  p.cargo = { ...STARTER_CARGO };                          // v2 档补赠（FR-6）
  if (p.activeShip) p.activeShip = { ...p.activeShip,
    ammo: (p.activeShip.fitted?.high ?? []).map((id) => defaultAmmoIdFor(MODULES[id])) };
  return p;                                                // 已装武器自动指定默认弹
}
```

**`merge` 增补**：

- `cargo`：逐键校验 `AMMO[id] && Number.isFinite(qty) && qty >= 0` → `Math.floor(qty)`，未知弹药 id **静默丢弃**（与 `fromIds` 同容错等级）；非对象整体回退 `{}`。
- `activeShip.ammo` 重对齐：与 `fitted.high` 的**原始 id 序列成对遍历**——模块 id 未知则连同其弹药项一起丢（保持与 `fromIds` 丢弃后的下标对齐，这是平行数组最易踩的坑）；幸存武器的 ammoId 若未知或族不符 → 自愈为 `defaultAmmoIdFor`（自动指定语义的延伸）；非武器 → null；缺 `ammo` 键（手改档）→ 全量按默认重建。
- 免费 Incursus 保底分支（既有 podded+破产判定）内同步三族弹药补足（与 `shipDestroyed` 同款）。
- `insurance` 不变式、损坏档回退全新开局等既有条款零改动。

### 8. FR-5 经济锚点：单场消耗 ≈ 赏金 5–15% 的验算（写入 balance.md，A-3 可执行断言）

**基准场景**：深度 3 巡逻。d3 解锁池 = guristas_frigate(w3) + serpentis_brawler(w2) → 期望基础赏金 `(3×15,000 + 2×18,000)/5 = 16,200`，×depthMult 1.3 = **21,060 ISK**。名义战斗时长 40s（TTK 带 20–60 中值；LML 风筝 build 因 dps 低取 55s）。每武器每场发数 = `ceil(时长/rof)`。

| loadout | 发数 | 弹药 | 消耗 ISK | 占赏金 |
|---------|------|------|----------|--------|
| 3× 爆能炮（rof 3.2） | 39 | Lead 35 | 1,365 | **6.5%** |
| 同上 | 39 | Antimatter 60 | 2,340 | **11.1%** |
| 同上 | 39 | Iridium 45 | 1,755 | **8.3%** |
| 3× 磁轨（rof 4.0） | 30 | Lead 35 | 1,050 | **5.0%**（带下缘，设计位） |
| 4× 火箭 Kestrel（rof 4.0） | 40 | 弹头 35 | 1,400 | **6.6%** |
| 3× 轻导（rof 5.0，55s） | 33 | 弹头 35 | 1,155 | **5.5%** |

敏感性：25s 速胜的 lead 爆能 ≈ 3.9%（快战省弹，容许溢出带下）；60s 苦战的 antimatter ≈ 16.2%（premium 装药慢战边缘越带上限，容许）；elite 赏金 ×2 而时长 ~×1.5 → 占比降至巡逻的 ~75%，深度上升同理（发数不随深度涨、赏金涨）——**汇的相对强度在浅层最大**，与"新手先学会算弹药账"的教学位一致。调节杆：**只动单价**（消耗速度=rof 是战斗手感参数不动；掉落束动 qty/chance 单侧）。

## 风险

- **单价 × 消耗速度是新的经济耦合点（PRD 风险 2，最大项）**：过贵玩家退回赠弹、过贱汇不成立。防线：决策 8 的带验算进 A-3 作发布闸；调节杆单旋钮化（只动单价；束只动 qty/chance；rof 是手感参数绝不为经济而动）；允许一轮修价（v0.7/v0.8 先例）。磁轨/火箭标准弹贴 5.0% 带下缘是已知设计位，若实测偏软先动这两处单价。
- **"必须装弹"新增败因（PRD 风险 1）**：fitModule 自动指定 + 赠弹 + 保底 + UNDOCK 警示四重缓解后，残余理论卡死是"有船有枪、无弹、无 ISK、机库无可卖物"——可卖装备回血 + RESET PROGRESS 兜底，且该形态在现版本（无武器可卖时）同样存在，非本版引入。×0.5 降级开火作为记录在案的回退阀，不预装。
- **BattleScene 武器分支自 v0.4 定型以来首次加实例态**（ammoId/reload/池 ref）：改动面收敛为门序四步 + stats 替换一处，分支集合零新增由 A-9 钉死；reload 独立字段避开 toggle 清零漏洞（核对结论 9）。refs 惯例全程遵守，无新 React 态入模拟。
- **平行数组是本版最易碎的不变式**：`ammo` 与 `fitted.high` 的下标对齐横跨 fit/unfit/switch/migrate/merge 五处，`merge` 的"未知模块成对丢弃"是唯一非显然点——A-7 的随机序列不变式测试与 A-8 的容错用例双向夹击。
- **撤退写 store 修订了 v0.9 文档不变式**：范围钉死"只结弹药"，CLAUDE.md/plan 双处记录；三 outcome 的结算表（决策 4）是语义的唯一权威，防止未来版本误扩。
- **掉落束对火箭新手 ~40% 消耗补贴**：有意的软着陆，若导致"弹药净汇在浅层不成立"，单侧下调 guristas_frigate 束的 qty/chance，不动价格。
- **v3 是首次链式迁移**：v1→v3 双段路径此前不存在，A-8 全路径回归 + 未知版本丢弃 + 损坏档兜底三层防线；`partialize` 键集快照防意外扩散。
- **LML 玩家两版连续承压**：v0.10 nerf 实效、本版再加弹药成本——但四型弹头让导弹成为唯一全伤害类型武器族（对 Sansha/Angel 的唯一主动端答案），净战略地位上升；随发布说明写明，观察后再议。
