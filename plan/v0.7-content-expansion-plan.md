## v0.5 / v0.6 落地后的复核结论

前置版本全部就位，以下按实际代码核对，其中三点是原计划未明确、实施时必须补上的义务：

- **（v0.5）T2 支援件零引擎工作**：全部走 `passive + modifiers` 管线，"数值 = T1 的 115–125%"直接改 `modifiers` 的 value；FittingWindow 属性面板由 `getEffectiveStats` 驱动，新船体（含 8 高槽 Catalyst）的数值展示零额外工作。
- **（v0.6 · 新增义务①）T2 必须同步编入掉落表才可获得**：v0.6 确立了"Meta/T2 仅掉落"，本版新增的 ~10 件 T2 若不进 `lootTable` 就是死内容。分配原则沿用"按 NPC 武器/派系主题"：如 Railgun II → Guristas Sniper、Rocket/LML II → Guristas/Angel 系、盾回 II → Caldari 系敌人（Sansha 表则投甲修/EM 抗性向）……4 个新 NPC 也各需自带 lootTable（A-5 引用完整性测试会强制核对）。
- **（v0.6 · 新增义务②）期望校准联动**：现有四张表经 10k 次蒙特卡洛校准在全解锁深度 [0.37, 0.40]（目标带 0.35–0.45）。向既有表追加 T2 项会抬高深度 7+ 的期望，实施时必须**同表下调既有项 chance 对冲**，并复跑 10k 校准（新旧全部 8 张表）。
- **（v0.6 · 新增义务③）`TIER_COLORS` 该抽共享了**：目前 MarketWindow 与 BattleScene 各持一份重复定义，本版 FittingWindow 的技能门槛提示（T2 徽章红字）是第三个消费点——顺手抽到 `src/lib/tiers.js`，三处改引。
- **（代码核对）requiredSkills 插入点明确**：`fitModule` 现有守卫链（activeShip/槽型 → 槽位数 → maxPerShip → eff 校验）在 maxPerShip 之后插入技能门槛即可；登舰门槛落在 `switchShip` 头部与 `buyShip` 的自动登舰分支，两处共用 `canBoard(ship, skills)`。
- **（代码核对）NPC 缺显式 `faction` 字段**：派系目前只体现在名字字符串里。加权派系池与 MapScreen 派系徽标都需要真实字段——给全部 NPC（含既有 4 个）补 `faction`，`buildEncounter` 的展开运算符自动透传（与 lootTable 同模式，零引擎改动）。MapScreen 已有 `threatLabel`（武器类型/射程/Web），派系徽标 + 主伤害类型是对它的纯扩展。
> 本版是对前三版管线的"内容压力测试"：任何一件新内容如果还需要改 BattleScene，都视为管线缺口，先补管线再落内容。

## 现状分析

- 技能系统：StationHub 内硬编码 `SKILL_DEFS`（3 项）+ 固定 1000 SP 训练价；store 的 `merge` 对 skills 做 `{ ...默认, ...存档 }` 浅合并——**新增技能自动以等级 1 回填，无需存档版本升级**。
- 船体门槛不存在：`buyShip`/`switchShip` 只查余额和归属；`fitModule` 只查槽位与 PG/CPU（v0.5 后经 `getEffectiveStats`）。两处都是加 `requiredSkills` 校验的自然位置。
- FittingWindow 的 `getSlotStyle` 按弧段均分槽位：high 弧段 70°，8 槽时间距 10°——拥挤但可用，需实测视觉效果，必要时把三段弧各拓宽 10°。
- 战斗模块栏是横排 48px 圆钮，8 高槽 + 中低槽在 1280 宽度下仍可容纳，无需改造。
- v0.4 的 NPC 管线（数据对象 + `ai` 参数 + `weapon.type` 分支 + `ewar` 透传 + `lootTable`）已支撑 4 个原型，新派系是纯数据扩充。
- MapScreen 敌情已显示名称/武器类型/射程/Web 标注（v0.4），本版升级为派系 + 伤害类型。

## 设计决策

### 1. 技能数据化（`src/data/skills.js`）

```js
export const SKILLS = {
  engineering: { name: 'Engineering', desc: '+5% CPU and Powergrid per level' },
  gunnery:     { name: 'Gunnery', desc: '+5% weapon damage per level' },
  navigation:  { name: 'Navigation', desc: '+5% base speed per level' },
  // 新增（v0.7 全部为门槛型技能，不给数值加成——避免与 v0.5 属性管线二次纠缠，
  // 数值型收益留给后续版本按需开放）
  destroyers:          { name: 'Destroyers', desc: 'Required to board destroyer hulls' },
  small_hybrid_turret: { name: 'Small Hybrid Turret', desc: 'Level 5 unlocks Tech II hybrid turrets' },
  missiles:            { name: 'Missiles', desc: 'Level 5 unlocks Tech II launchers' },

```

- StationHub SKILLS tab 改为遍历此表（删除硬编码 `SKILL_DEFS`）；训练价改为 `1000 × 当前等级`（线性递增，给 SP 一个长线出口；胜利加 SP 的来源不在本版范围，现有 SP 存量即门槛节奏）。
- 存档：零版本升级（merge 自动回填），但需要一个用例验证旧档新技能默认 1。

## 风险

- 内容量 ×2 后平衡回归面大：balance.md 锚点先行 + A 组数值测试是防线，但"好不好玩"仍需实机，本版验收允许数值二轮修正。
- Angel Hunter 若引入 `ai.command` 泛化，须同步检查敌方 `desiredVelocity` 调用的 approach 分支在 dist<0.2 时的停车行为是否适合追击 AI（贴脸后应保持 orbit 0.5km 而非停死——实现时若发现别扭，改为 `command: 'orbit', radius: 0.5` 即可，不动管线）。
- 训练价递增改变 SP 经济：现有存档的大额 SP 存量会瞬间贬值/升值不对称——发布说明里写明，不做迁移补偿。
