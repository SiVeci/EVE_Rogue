## 现状分析（读完 BattleScene 后的关键结论）

- `calculateHitChance` / `applyDamage` 等已是模块级纯函数，抽取到共享库零风险。
- 敌人武器是硬编码分支：计时器到点 + 射程内 → 必中伤害（导弹语义）。改为按 `weapon.type` 分支即可与玩家武器同构。
- 玩家 web 敌人：模块循环算出 `enemyWebbed` → 敌人速度 ×0.5。反向管线（敌 web 玩家）不存在，但结构对称，加法一致。
- 相对角速度 `angularVelocity` 对双方对称（|切向相对速度|/距离），敌方炮台命中可直接复用。
- 玩家命中/暴击共用一次 roll（roll < 0.01 → 3× wrecking）——抽成 `rollTurretShot` 时必须保持该语义。
- 遭遇在 BattleScene 内即时生成（`getEncounter`），MapScreen 又独立调用一次——两处可能不一致，正好借 FR-4（敌情预生成）根治。

## 设计决策

### 1. 共享战斗数学库 `src/lib/combat.js`

从 BattleScene 平移 5 个纯函数：`getCapacitorRecharge`、`calculateHitChance`、`applyDamage`、`desiredVelocity`、`applyInertia`；新增：

```js
// 一次炮台射击判定。miss 返回 null，命中返回伤害系数（1% 概率 3× wrecking）
rollTurretShot(distMeters, stats, targetSig, angularVel, rng = Math.random)
```

玩家 hybrid 分支与敌方炮台共用它 → 敌方炮台自动获得命中曲线、伤害浮动与 wrecking（NPC 数值按有效期望 ~0.77× 标定）。导弹敌人保持必中平伤。`rng` 参数是为 Node 测试注入确定性。

### 2. NPC 原型与遭遇池（`npcs.js` 重构）

新增 3 个 NPC（数值要点）：

| NPC | 武器 | AI/机动 | 抗性倾向 | 解锁深度 |
|-----|------|---------|----------|----------|
| Serpentis Brawler | 爆能炮台：optimal 1.5km / tracking 280 / th9+kin9 / 3.0s | orbit 1.0km，350 m/s | 甲抗（Serpentis） | 3 |
| Guristas Sniper | 磁轨炮台：optimal 15km / falloff 5km / tracking 35 / th7+kin7 / 4.5s | orbit 15km，220 m/s | 盾抗 | 5 |
| Angel Webber | 火箭：range 9km / kin6+exp10 / 3.5s + **ewar**（10km 内减速 50%） | orbit 2.5km，330 m/s | 均衡 | 7 |

数值意图：Sniper 对静止目标全中、对 1km 高速绕轨（~0.3 rad/s，trackPenalty≈8）几乎全失；开 MWD 直冲时 sig ×5 抵消 tracking 劣势 → "冲脸有代价"成立。

API 重构：`getEncounter` 拆为
- `ENCOUNTER_UNLOCKS`（深度→NPC 解锁表）+ `availableNpcs(depth)`；
- `buildEncounter(npc, depth, nodeType)`（现有缩放逻辑参数化，ewar 字段透传，不缩放减速百分比）；
- `rollEncounter(depth, nodeType, rng = Math.random)`（从池中随机抽取）。

### 3. 敌情预生成（FR-4，"看到的就是打到的"）

- App.jsx：新增 `goToMap()`——用 `useGameStore.getState().deadspaceDepth`（避开闭包陈旧值）为两个节点各 roll 一份遭遇存入 state，再切视图。UNDOCK 与 onVictory 都走它。
- MapScreen：删除自身的 `getEncounter` 调用，改收 `encounters` prop；节点上显示敌人名、武器类型提示（Turret/Missile + 射程）、有 ewar 标注 Stasis Web、赏金。
- BattleScene：新增 `encounter` prop 直接使用；保留 `?? rollEncounter(...)` 兜底。

### 4. 敌方武器泛化 + 敌方 EWAR（FR-1/2）

BattleScene 敌人开火分支改为按 `enemy.weapon.type`：
- `hybrid_weapon`：射击门限 `distMeters ≤ optimal + 2×falloff`，`rollTurretShot` 判定，目标 sig 用**玩家有效 sig**；
- `missile_weapon`：现行为不变（射程内必中）。

敌方 EWAR：`enemy.ewar && distMeters ≤ enemy.ewar.optimal` → `playerMaxSpeed ×= (1 − pct/100)`，在玩家模块循环之后、移动积分之前应用（与推进加成乘法叠加）。

### 5. MWD 与信号半径惩罚（FR-5/6）

- `modules.js` 新增 `mn1_mwd_i`（propulsion / mid / T1 / 12,000 ISK）：`speed_multiplier 5.0`、**`sig_multiplier 5.0`**、cap 30/10s、pg 12 cpu 18（Incursus 40PG 下装 MWD 是真实取舍）。
- BattleScene 玩家模块循环：propulsion 同时累乘 `playerSigMult ×= stats.sig_multiplier ?? 1`；敌方炮台命中以 `玩家 sig × playerSigMult` 计算。AB 无该字段 → 不受罚（符合 EVE 设定）。

### 6. CLAUDE.md 同步更新

存档系统、市场、遭遇池已使 CLAUDE.md 多处失实（"无存档"、"getEncounter"、"唯一 NPC"等），本版顺手把 v0.2–v0.4 的架构变化更新进去。

## 验证计划

**A. Node 纯函数测试**（combat.js 与 npcs.js 不再依赖组件，直接导入）：
1. 命中公式：optimal 内静止 → 1.0；optimal+falloff → 0.5；Sniper 参数 vs 0.3 rad/s → ≈0；同参数下 sig ×5 → 命中显著回升（MWD 惩罚方向正确）。
2. `rollTurretShot`：注入 rng 验证 miss / 普通命中系数域 [0.5,1.0) / roll<0.01 → 3.0（wrecking 语义保持）。
3. `applyDamage`：已知抗性下伤害精确值；层级顺序盾→甲→壳；无穿层。
4. 遭遇池：availableNpcs 在深度 1/3/5/7 分别返回 1/2/3/4 个；rollEncounter 注入 rng 可命中每个原型；elite ×1.5/赏金 ×2 与深度缩放数值；Angel 的 ewar 字段透传且减速值不被缩放；buildEncounter 不污染 NPCS 原始数据。
5. 存档回归：v0.4 无 store 结构变更，跑 v0.3 的 p1/p7 两个代表阶段确认无回归。

**B. SSR 渲染测试**：MapScreen 注入两份构造遭遇 → 两节点分别显示敌名、武器提示、Stasis Web 标注与赏金。

**C.** `npm run lint`、`npm run build`。

**D.** 浏览器实战验证（Sniper 绕轨 miss 率、MWD 冲锋被命中率、web 减速体感）仍受分类器故障阻塞——数学层已由 A-1 覆盖方向正确性，真机体感调参留待工具恢复。

## 风险

- 敌方炮台引入伤害浮动（×0.5–1.0 + wrecking），同深度体感 DPS 低于面板值——NPC 面板伤害已按期望上调补偿，实玩后可再调。
- BattleScene 是"每帧重建闭包"模式，新增的 `playerSigMult`/敌 ewar 必须放在 step 闭包内计算（不能提升为 render 期常量），否则 MWD 开关不生效——实现时注意。
- MapScreen/BattleScene 的遭遇改为对象传递后，刷新页面丢失 map 态遭遇（React state 不入档）——当前视图本就不持久化（刷新回 station），行为一致，不处理。
