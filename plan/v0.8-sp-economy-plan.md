## 代码核对结论（动笔前逐项核实）

**依赖确认**：README 表中 v0.8 唯一依赖 v0.7，已按代码核实全部在位——`src/data/skills.js`（10 项技能 + `meetsRequiredSkills`/`describeRequiredSkills`）、`gameStore.canBoard` 守卫 `switchShip`/`buyShip` 两条登舰路径、8 个带 `faction`/`lootTable` 的 NPC 与加权池 `pickWeightedNpc`、8 件带 `requiredSkills` 的 v0.7 T2、Catalyst 门槛、FittingWindow 锁定提示。不阻塞。

**PRD 引用逐项核实（与代码一致）**：

**PRD / 文档与代码不一致，按代码现状裁定（计划已按裁定编写）**：

1. **CLAUDE.md 的训练价公式文本过时**：写的是 `1000 × currentLevel`，代码 / balance.md / 本 PRD 均为 `1000 × (currentLevel + 1)`（v0.7 实施记录里明确修正过）。以代码为准；实施步骤 8 顺手改 CLAUDE.md。
2. **两件 v0.6 T2 缺 `requiredSkills`**：`light_neutron_blaster_ii` 与 `ballistic_control_system_ii` 没有技能门槛字段，8 件 v0.7 T2 都有。这与 CLAUDE.md"每件 T2 都需对应技能 4/5 级"的声明、`skills.js` 技能描述文本（"Level 5 unlocks Tech II small hybrid turrets **and damage amplifiers**"）、以及 PRD FR-2 的示例（LNB II 出现在 Small Hybrid Turret 5 的解锁清单里）三处矛盾——v0.7 给新 T2 加门槛时漏掉了这两件存量。**决定：本版补上**（LNB II → `{ small_hybrid_turret: 5 }`，BCS II → `{ missiles: 5 }`），否则 FR-2 的反查清单与技能描述文本自相矛盾。对旧档的影响见"存档影响"。
3. **PRD FR-4 的"15 km 磁轨 build"现有内容撑不起**：玩家侧唯一磁轨是 125mm（optimal 9 km，raw 2.25 dps/门），既到不了 15 km 档射程，DPS 数学上也进不了 20–60 s 验收带（详算见设计决策 4）。**决定：补一件 150mm Railgun I（T1 长程磁轨）**，让"远程磁轨驱逐"成为真 build 方向——这是对 PRD FR 清单的一处小扩充，需用户确认（见计划外拍板项）。
4. **玩家侧现无 EM/EXP 伤害来源**：全部玩家武器只输出 TH/KIN（混合炮）或纯 KIN（导弹）。验收第 4 条"四派系同理"对 Sansha（弱 EM）/ Angel（弱 EXP）**当下只能在纯函数数学层验证**，实机可玩的只有 Guristas（弱 KIN）/ Serpentis（弱 TH）两路。这正是 PRD 动机"FR-3 是 v0.11 弹药的硬前置"的另一面：签名先立、投放口后到。验证计划按此拆层，不算缺陷。

## 现状分析

- **结算通路**：`endBattle('victory')` 冻结模拟并 roll 掉落 → 浮层展示赏金/掉落 → 确认键一次性 `addIsk + addLoot + advanceDepth + onVictory`。SP 完全平行于 ISK：`spReward` 随 encounter 透传进浮层展示，确认键内并列 `addSp`。BattleScene 改动面 = 解构 +1、浮层文案 +1 行、按钮 +1 行，零战斗循环改动（延续 v0.7"新内容不碰引擎"的合规口径）。

- **`buildEncounter` 展开透传**：`spReward` 加进返回对象即达 MapScreen 与 BattleScene 的 props，与 `lootTable`/`faction` 同模式。
- **抗性签名是纯数据**：改 `npcs.js` 四个 NPC 的 `defense` 三层数值，敌我双方伤害结算都走 `applyDamage` 按层查抗性，无任何引擎分支。

## 设计决策

### 1. FR-1 SP 战斗产出

- `src/data/npcs.js`：文件顶部加常量 `SP_REWARD_BASE = 200`（与 balance.md 锚点同源），`buildEncounter` 返回对象增加：

  ```js
  spReward: Math.round(SP_REWARD_BASE * depthMult * (nodeType === 'elite' ? 2 : 1))
  ```

  与 `reward` 完全同构：吃 `depthMult`（不吃 elite 的 ×1.5 statMult），elite ×2。基线不随 NPC 原型变化（ISK 赏金按原型分档体现"目标价值"，SP 表达"战斗经验"，只随深度/节点走）。深度 1 巡逻 = 200、elite = 400，验收第 1 条的字面数值。
- `src/store/gameStore.js`：新 action，与 `addIsk` 逐字对称：

  ```js
  addSp: (amount) => set(state => ({ sp: state.sp + amount })),
  ```

- `src/components/BattleScene.jsx`：`useGameStore()` 解构加 `addSp`；胜利浮层赏金行改为并列展示（`Bounty: {enemy.reward.toLocaleString()} ISK · +{enemy.spReward.toLocaleString()} SP`，SP 用技能页同款琥珀色 `#e8a838` 强调）；确认键内 `addIsk(enemy.reward)` 旁加 `addSp(enemy.spReward)`。
- **阵亡不给 SP**（PRD 开放问题 2 的落定）：保持 roguelike 损失锐度，也免去"送死刷 SP"的口子；数学上 25% 安慰奖（深度 1 约 50 SP）对死亡螺旋的缓解量级本就无感，等实测观察再议。`shipDestroyed` 零改动。
- **MapScreen 不显示 SP**（范围裁定）：节点上的赏金因 NPC 原型而异、是巡逻 vs elite 的决策信息；`spReward` 在同深度下只有恒定 ×2 关系，无增量决策信息。PRD FR-1 也只要求结算浮层。`spReward` 已随 encounter 透传到 MapScreen props，将来要显示是一行的事。

**节奏推演**（定稿 200 的依据，写入 balance.md）：单 run 巡逻连胜的累计 SP = `200n + 15n(n−1)`，5 胜 = 1,300 ≥ Destroyers I 的 1,000（验收第 2 条前半；开新档时初始 1,500 SP 本就够直接点，"5 场"覆盖的是把初始 SP 花去别处的玩家）。第一个 T2 武器门槛 15,000 SP：纯巡逻浅深度（均值深度 3，260 SP/胜）约 55 胜；中深度混 30% elite（约 390 SP/胜）约 38 胜；深潜 elite 为主约 30 胜——带宽 30–55 胜，覆盖 PRD 的 30–50 目标带。产出端唯一调节杆是 200 这个基数。

### 2. FR-2 SKILLS 页签目标导向

- `src/data/skills.js` 新增纯函数（引入 `SHIPS`/`MODULES`，无循环依赖）：

  ```js
  // 反查一个技能解锁的全部内容，按所需等级分组升序。
  // 数值技能天然返回 []（没有内容以它作 requiredSkills）。
  export function skillUnlocks(skillKey) {
    const byLevel = new Map();
    for (const item of [...Object.values(SHIPS), ...Object.values(MODULES)]) {
      const lvl = item.requiredSkills?.[skillKey];
      if (lvl === undefined) continue;
      if (!byLevel.has(lvl)) byLevel.set(lvl, []);
      byLevel.get(lvl).push(item.name);
    }
    return [...byLevel.entries()].sort((a, b) => a[0] - b[0])
      .map(([level, names]) => ({ level, names }));
  }
  ```

- `src/components/StationHub.jsx`：SKILLS 页签主体抽成同文件导出的 `SkillsPanel` 组件（行为零变化，纯粹让 B 组 SSR 测试能直接渲染它——`activeTab` 默认 'fitting'，整组件渲染够不到技能页；v0.6 对掉落列表做过同样的"抽出以便测"）。每个技能行 desc 下方渲染 `skillUnlocks(key)`：

  ```
  Lv 1 → Catalyst, Cormorant            （当前等级 ≥ level：绿色 #2cd67c 前缀 ✓)
  Lv 5 → Light Neutron Blaster II, …    （未达标：链接蓝 #5a96ff）
  ```

  数值技能 `skillUnlocks` 为空数组 → 不渲染任何附加行，即 PRD 的"属性技能行维持现状"。
- **前提数据修复**（核对结论第 2 条的落地）：`modules.js` 给 `light_neutron_blaster_ii` 加 `requiredSkills: { small_hybrid_turret: 5 }`、给 `ballistic_control_system_ii` 加 `requiredSkills: { missiles: 5 }`。补齐后各门槛技能的解锁清单：

  | 技能 | 解锁（level → 内容） |
  |------|----------------------|
  | destroyers | 1 → Catalyst、Cormorant |
  | small_hybrid_turret | 5 → Light Neutron Blaster II、125mm Railgun II、Magnetic Field Stabilizer II |
  | missiles | 5 → Rocket Launcher II、Light Missile Launcher II、Ballistic Control System II |
  | shield_operation | 4 → Small Shield Booster II |
  | repair_systems | 4 → Small Armor Repairer II |
  | high_speed_maneuvering | 4 → 1MN Afterburner II |
  | electronic_warfare | 4 → Stasis Webifier II |

  （不变式：每个门槛技能解锁清单非空——进 A 组测试。）

## 风险

- **SP 速率是本版唯一敏感数值**：产出过快 T2 失去长线意义、过慢闭环等于没修。防线：A-2 节奏模拟给出三画像带宽 + balance.md 单旋钮约定（只动 200 基数）；上线后按实际节奏只调锚点不动公式。
- **两件 v0.6 T2 补门槛是对存量玩家的追溯 nerf**：机库里已掉落的 LNB II/BCS II 会突然要求技能 5。缓解：已装的不受影响（merge 不重验）、FittingWindow 自动出红字说明、发布说明写明（v0.7 训练价改动同款处理）。若用户否决此项，则 FR-2 清单照代码现状渲染（这两件不出现在任何技能下），并需同步修 CLAUDE.md 与 skills.js 的描述文本自洽——两条路都通，默认走补授。
- **抗性收窄对错误伤害类型玩家是隐性增强敌人**：量化上限 +13%（纯导弹打 Webber），A-5 以 ×1.15 为回归闸；当下玩家武器只有 TH/KIN，Guristas/Serpentis 两派系反而对现有 fit 更顺，真正的"选错惩罚"要到 v0.11 弹药提供 EM/EXP 才完整兑现——届时若体感过硬，调 NPC 弱抗与其余抗性的差值（20–30 带内），不动公式。
- **150mm 上架的护卫外溢**：护卫 3 门 150mm 的远程 fit 数学上 dps 低于爆能且 tracking 70 贴脸即瞎，不构成优势策略；但 12 km optimal 与 Guristas Sniper 15 km 轨道的交互（互相 falloff 边缘对射）是新几何，D-3 实机确认无"互瞎僵局"体感（模型值：15 km 处玩家命中 68%，无僵局）。
- **Cormorant 平衡是二手推断**：TTK 表建立在命中率 0.95/质量期望 0.7725 的解析模型上，实机的轨道几何（敌方 2.5 km 收距 vs 玩家 keepAtRange）可能压低有效命中；D-3 是最终裁判，工具不可用时此项与 v0.4–v0.7 的积压实机项一并顺延，数值允许二轮修正（v0.7 先例）。
