**实施中的一处数值调整**（计划内已预留的校准空间）：`guristas_sniper` 和 `angel_webber` 两张掉落表的理论期望值原设计恰好卡在 0.35 的下边界，10,000 次蒙特卡洛验证时因抽样噪声两次都略跌破边界（0.346、0.349）。这不是实现 bug，是数值设计本身没有留安全边际——按计划验证步骤 A-4 的校准意图，把两张表里各一件的 chance 上调 0.02（railgun_125mm_i 0.15→0.17、fleeting_compact_stasis_webifier 0.15→0.17），复测后四张表全部落在 [0.37, 0.40] 区间，不再压边界。

验证结果：A 组纯函数测试 27/27 通过（tier 门控三深度、elite 双轮、rng 序列确定性逐条验证、10,000 次期望校准、lootTable→MODULES 引用完整性、未知 id 容错、空表边界）；B 组 store 测试 7/7 通过（addLoot 追加/混合容错/空数组、掉落 T2 件走正常装配与阵亡销毁流程）+ 跨进程存档往返 5/5 通过（含被动 T2 件的 tier/passive/modifiers 字段完整保留）；C 组 SSR 测试 12/12 通过（购买目录排除 Meta/T2、出售区三档 tier 徽章与售价）；lint 与 build 均通过，无新增告警。浏览器实机验证（掉落浮层交互、elite 双倍掉落体感）仍受预览工具故障阻塞。

## v0.5 落地后的复核结论

原计划中依赖 v0.5 的条件分支全部消除，以下几点已按实际代码确认：

- **掉落表引用完整性可满足**：下方 4 张 lootTable 引用的全部 T1/Meta 模块 id（`ballistic_control_system_i`、`magnetic_field_stabilizer_i`、`co_processor_i`、`overdrive_injector_system_i` 等）现已真实存在于 `modules.js`，验证 A-5 的静态完整性测试可以直接落地。
- **BCS II 的数据形态可以写死**（见下文第 3 节）：v0.5 的 `passive: true + modifiers` 管线已实装，T2 被动件是纯数据。
- **市场 UI 零额外工作**：v0.5 给 `MarketWindow.keyStats` 加的被动件摘要分支（`MODIFIER_LABELS` + `formatModifier`）对 T2 被动件自动生效；T1 过滤同样只影响购买目录，v0.5 新增的 9 件 T1 被动件不受影响。
- **掉落的被动件零引擎工作**：入库后经装配即走 `getEffectiveStats`，战斗中被 `activeModules()` 过滤，与 v0.5 验收一致，本版无需再验被动件行为，只验"掉落→入库"链路。

## 现状分析

- 胜利流程：`endBattle('victory')` 冻结模拟 → 胜利浮层按钮一次性执行 `addIsk(enemy.reward) + advanceDepth() + onVictory()`（onVictory 即 App 的 `goToMap`，会为下一层重新 roll 遭遇）。掉落的结算点应在 `endBattle` 时 roll（战斗结果确定的瞬间），按钮点击时入库——两个时机分离，防止刷新浮层重 roll。
- `npcs.js` 经 v0.4 重构后每个 NPC 是独立数据对象，`buildEncounter` 用扩展运算符透传未知字段——**`lootTable` 加在 NPC 上会自动出现在 encounter 里**，BattleScene 无需知道池子怎么配。
- 库存管线：store 的 `inventory` 存模块对象、存档存 id（v2 格式），新增"入库"action 走 `MODULES[id]` 映射即可，存档零改动。
- MarketWindow 目录来自 `Object.values(MODULES)` 按槽位过滤——收紧为 T1 only 是一行过滤；机库出售侧不区分品级，天然支持 Meta/T2 变现。
- `tier` 字段已就位（'T1' | 'Meta'），v0.6 引入 'T2'。

## 设计决策

### 1. `src/lib/loot.js` — 纯函数掉落 roll

```js
// 品级解锁：T1 恒可掉，Meta 需 depth ≥ 4，T2 需 depth ≥ 7
export const TIER_UNLOCKS = { T1: 1, Meta: 4, T2: 7 };

// encounter 携带 lootTable；elite 对全表 roll 两轮（PRD 开放问题的落地选择：实现最简）
export function rollLoot(encounter, depth, nodeType, rng = Math.random) => moduleId[]
```

- 逐条独立 roll：`rng() < entry.chance` → 掉落。
- 按品级过滤：entry 对应模块的 `tier` 未到解锁深度 → 本次跳过（同一张表随深度"逐渐变肥"，不用为每个深度段维护多张表）。
- 未知 moduleId 的表项跳过并 `console.warn`（数据容错与存档 merge 同强度）。

### 2. 掉落表数据（`npcs.js` 每个 NPC 加 `lootTable`）

设计原则：普通节点期望 ~0.4 件/场（PRD FR-1）；**向 Meta 倾斜**（PRD 风险条：T1 掉太多会蚕食市场）；每个 NPC 掉"符合其武器/派系"的模块，让"想要某件装备"变成"去找某种敌人"：

| NPC | 表项（moduleId × chance）建议 |
|-----|-------------------------------|
| guristas_frigate | rocket_launcher_i ×0.10；light_missile_launcher_i ×0.10；ballistic_control_system_i ×0.12（Meta 层前空转，深度 4+ 生效） |
| serpentis_brawler | light_ion_blaster_i ×0.12；fleeting_compact_stasis_webifier ×0.10；magnetic_field_stabilizer_i ×0.10；light_neutron_blaster_ii ×0.08 |
| guristas_sniper | railgun_125mm_i ×0.15；co_processor_i ×0.10；light_neutron_blaster_ii ×0.08 |
| angel_webber | fleeting_compact_stasis_webifier ×0.15；mn1_monopropellant_afterburner ×0.10；overdrive_injector_system_i ×0.10 |

（chance 为首版建议值，落地后按 A-4 的期望测试校准到 0.35–0.45 区间。）

### 3. T2 样例（`modules.js`，`tier: 'T2'`，仅掉落获取）

- **Light Neutron Blaster II**（hybrid/high）：Ion 的 ~118% 伤害（th 9 + kin 9）、optimal 1400 / falloff 2400 / tracking 300 / rof 3.2、pg 7 cpu 18、price 30,000（price 在 v0.6 只用于出售定价 = 15,000 入袋）。
- **Ballistic Control System II**（passive/low）：沿用 v0.5 已实装的被动件形态，纯数据——

  ```js
  ballistic_control_system_ii: {
    id: 'ballistic_control_system_ii',
    name: 'Ballistic Control System II',
    type: 'damage_amp', slot: 'low', tier: 'T2', price: 28000,
    passive: true,
    description: '...',
    cost: { pg: 4, cpu: 24, cap: 0 },
    modifiers: [{ stat: 'damage.missile_weapon', op: 'mult', value: 1.13 }]
  }
  ```

技能门槛（PRD v0.7 FR-4）本版**不做**——T2 在 v0.6 只是"更强的稀有件"，门槛留给 v0.7 加 `requiredSkills` 时统一处理。

### 4. 结算流与 UI

- BattleScene：`endBattle('victory')` 时 `rollLoot(enemy, deadspaceDepth, nodeType)` 存入 ref/state；胜利浮层在赏金下方列出掉落（tier 着色：T1 灰 / Meta `#e8a838` / T2 蓝），空掉落显示 "No salvageable modules."；按钮点击时 `addIsk + addLoot(lootIds) + advanceDepth + onVictory`。
- store 新增 `addLoot: (moduleIds) => inventory 追加`（复用 fromIds 语义：未知 id 静默丢弃）。
- MarketWindow：购买目录过滤 `tier === 'T1'`（Meta 从货架消失——v0.3 上架的 Meta 变体自此转为掉落专属，与 PRD v0.6 FR-4 一致）；出售区加 tier 着色以突出高价值变现。

### 5. 存档与兼容

- 存档格式零改动（掉落物是普通模块 id）。已持有 Meta 件的旧档不受市场收紧影响（东西还在，只是买不到第二件了）。

## 验证计划

**A. Node 纯函数测试（loot.js + npcs.js）**：
1. 品级门控：同一张表在 depth 2 / 5 / 8 下可掉集合分别为 T1 / T1+Meta / 全部。
2. elite 双轮：注入全命中 rng → 掉落数恰为普通节点两倍。
3. 确定性：注入 rng 序列逐条验证命中/未命中边界（`rng() < chance` 语义）。
4. 期望校准：以真随机跑 10,000 次模拟，各 NPC 普通节点期望落在 0.35–0.45 件/场（校准 chance 值的回归锚）。
5. 表数据完整性：全部 lootTable 的 moduleId 都存在于 MODULES（防拼写错误的静态检查式测试）。
**B. store 集成测试（sed 副本）**：`addLoot` 入库 + 含未知 id 的混合列表容错；掉落 Meta 件 → 存档往返 → 出售价正确（price×0.5）。
**C. SSR 测试**：MarketWindow 购买目录不含 Meta/T2（收紧生效）、出售区 tier 着色存在；胜利浮层属 BattleScene（Canvas 不可 SSR）——把"掉落列表渲染"所需数据整形抽成可导出纯函数（如 `describeLoot(ids)`）单测，浮层本体留浏览器验证。
**D.** lint、build；浏览器实机（打 elite 看双轮掉落、结算浮层交互）待预览工具恢复。

## 风险

- **经济平衡**：Meta 从市场下架后，v0.3 时代"攒钱买 Meta"的路径消失，短期可能感觉变穷——观察后可用"掉落率上调"或"部分 Meta 保留市场但高溢价"调节，本版先按 PRD 执行。
- 掉落 roll 放 `endBattle` 而非按钮点击：若玩家在浮层期刷新页面，掉落丢失（战斗本就不入档，行为一致）——不处理，但写入 CLAUDE.md 已知行为。
- lootTable 与 MODULES 的引用完整性靠 A-5 测试兜底，数据增删时必跑。
