**PRD 与代码/自洽性不一致，按裁定编写（★ = 需用户拍板，末节汇总）**：

1. **★ PRD 的导弹公式满足不了它自己的 MWD 验收**：PRD 写 `min(1, sig/expR, ((sig/expR)·(expV/v))^drf)`——MWD 的 `sig_multiplier` 与 `speed_multiplier` 同为 5.0，在乘积项里**精确抵消**，开 MWD 前后第三项不变，验收第 1 条"开 MWD 导弹伤害回到满额"数学上不可能成立。裁定改用等价性更好的简化形：`factor = min(1, (sig/expR) × min(1, expV/v)^drf)`（drf 只作用于速度比；原式的裸 `sig/expR` 中间项在该形下恒不束紧，省去）。该形下四条行为全对：静止=1、快船显著<1、被 web 回升、MWD bloom 满额（验算见设计决策 1）。
2. **★ NPC 导弹武器零电容消耗**：玩家导弹发射器 `cost.cap` 全部为 0（EVE 惯例），若 NPC 火箭耗电则双侧不对称。裁定 NPC 导弹武器 `cap_use: 0`——代价是**中和对纯火箭船（Guristas Pirate、Angel Hunter）无武器压制效果**，对 Angel Webber 只能停它的 web。这把 FR-2"武器每次激活消耗电容"收窄为"炮台与 EWAR 耗电、导弹免疫"，换来 EVE 保真 + 玩家对称 + "按敌情选模块"的克制纹理（MapScreen 派系情报本就显示武器类型，出击前可判断带不带中和）。
3. **★ "炮停转"需要一条激活线，而非玩家式的逐次成本判定**：结构性矛盾——玩家语义（cap < cost 才跳过）下，NPC 武器 cap_use（2–12 GJ）远小于"被中和压空后每个中和周期间隙的回充积累"（该曲线在低电量区速率 ≈ `0.9–1.7×max/T`，5 秒间隙可积 5–15 GJ），武器会在中和周期间隙不断"漏火"，只能降频、永远停不下来；把 cap_use 拉大到能停的量级，则电容稳定标定所需的回充峰值同步膨胀，间隙积累更大——不等式两头无解（推导见设计决策 2）。裁定：NPC 武器/EWAR 的激活判定线为 `max(cap_use, 10% × cap_capacity)`（滞回带）。低于线 → 全停；回充过线 → 恢复——恰是 PRD FR-3"回充过**激活线**后恢复"的字面。对"与玩家模块语义一致"的偏离理由：NPC 没有手动重开模块的驾驶员，滞回带防止贴线抖动；且 `T > 90s`（全部 NPC 满足）时"中和周期间隙回充 < 10% 线"结构性成立，"停转"从调参问题变成不变式。
4. **★ 堆叠惩罚会造出"卸不掉的超限 fit"，unfitModule 守卫需放宽一格**：Co-Processor（CPU ×1.10 mult）被罚后，旧档 4× Co-Processor 的 Incursus 满配 fit 容量从 197.6 掉到 175.4 tf（−22.2）。现守卫"卸后仍超限即拒绝"会把这种档**每一件都卸不下来**（若无单件 CPU ≥ 22 tf 的模块），PRD 自己的测试要求"超限旧档可载入、**卸下后**装不回去"都无法执行。裁定：`unfitModule` 增加一格豁免——**当前已超限时，只要卸载不使超限恶化即放行**（合法 fit 的原守卫原样保留）。
5. **PRD 写于 v0.9 之前**：文中默认 BattleScene 每胜推进深度、每场满血开局。现状是三 outcome、HP 跨节点、`depth` prop 按节点深度——对本版 FR 挂点无影响（都在模拟循环内），但 TTK 复核必须按**段级损耗**建模（一段 5 层的累积磨血），敌方电容也要明确"每场满电开局"（BattleScene 每场重挂载，电容 ref 挂载时初始化，天然成立且不入 HP 快照——与玩家电容同语义；8 秒对齐撤退期间敌方电容照常消耗/回充，撤退后该 encounter 节点已封闭不可再入，无状态残留问题）。
6. **PRD 的 drf 调参方向写反了**：PRD 风险节说"过强则 drf 上调（减伤变浅）"——drf 在其公式里是指数，`x^drf`（x≤1）随 drf 上调而**变小**，即减伤变**深**。以指数语义为准：减伤过强应**下调** drf。锚点注释按正确方向写。
7. **PRD 未提的 UI 触点**：MarketWindow 的 `keyStats` 按 `m.type` switch 生成目录行——`energy_neut` 不加 case 会渲染空行（T1 中和要上市场）。FittingWindow 无任何类型分支（grep 核实），MapScreen `threatLabel` 只读 `weapon.type`（不变）——均零改动。
8. **敌方电容不随深度/elite 缩放（裁定）**：`buildEncounter` 只缩放 HP/伤害/赏金/SP，沿用"EWAR strength is not scaled"先例——中和 build 的有效性不随深度贬值，正如 web 克制不随深度贬值。

## 现状分析

- **本版主体在纯函数层**（PRD 判断与代码一致）：`missileDamageFactor` 新增、`applyDamage` 改造都在 `combat.js`；堆叠惩罚在 `shipStats.js` 的 mult 合并处。BattleScene 改动收敛为：两个导弹分支各传一个 factor、敌方电容三件套（ref + 回充 + 激活线判定）、`energy_neut` 一个新分支、敌方 HUD 加一条电容条。
- **敌我导弹侧的输入都现成**：敌方侧 `playerVel.current.length()×1000` 是玩家实速（m/s）、`effectiveSig = eff.defense.sig_radius × playerSigMult` 已含 MWD bloom 与盾扩静态 sig（炮台分支现用同一变量）——导弹 factor 直接复用，PRD"MWD 信号惩罚对导弹也成立 / 静态 sig 修正参与"零额外接线。玩家侧对称：`enemyVel` + `enemy.defense.sig_radius`（NPC 无 sig 修正）。
- **零差异回归的证明结构**：本版对无中和战斗的唯一行为变更点是"电容不足跳过激活"这一个新判定。若标定保证任意时长下 NPC 电容恒在激活线上（站立消耗 < 回充峰值、且均衡点远高于线），该判定永不触发 → 逐拍一致自动成立。测试据此设计（A-3/A-4）：证明"判定不触发"，而非逐帧比对两套引擎。
- **堆叠惩罚 × v0.9 交界（用户特别标注项，逐一核实）**：`getEffectiveStats` 喂 `resumeHp` 的上限——全游戏 HP 类 mult 只有 Nanofiber 的 hull ×0.90（负向），受罚后堆叠负向反而**变轻**（0.81 → 0.8218），上限只增不减，`resumeHp` 的夹紧防御照旧；盾/甲全是 add（盾扩/甲板），不受罚。CPU/PG 超限旧档：`merge` 按 id 重建 fit 不重验（核实），载入照常、BattleScene 不校验 `used`（核实）照打，仅装/卸时把关——容忍语义不破，唯一缺口是核对结论 4 的卸载死锁，已裁定修法。
- **中和的自残端**：玩家侧 T1 中和 5 GJ/s 自耗（25 GJ/5s）对 Incursus（回充峰值 5.35 GJ/s）几乎吃满全部续航——开中和≈放弃主动修的电容预算，PRD"自身电容开销高"的落点。Catalyst（8 高槽、450 GJ）双中和 + 6 炮是天然的中和压制船。

## 设计决策

### 1. FR-1 导弹伤害公式（`src/lib/combat.js`）

```js
export const MISSILE_DRF = 0.5;

// EVE 简化版导弹伤害系数。返回 (0,1]，静止或大信号目标 = 1。
// 速度比先夹到 ≤1 再取 drf 次方（目标比爆炸速度还慢时不产生 >1 的增伤），
// 信号比在幂外线性参与 —— MWD 的 sig 与 speed 同倍率放大时信号项胜出，
// bloom 把速度减伤吐回去（炮台侧语义的镜像）。
export function missileDamageFactor(targetSig, targetSpeed, explosionRadius, explosionVelocity, drf = MISSILE_DRF) {
  if (!explosionRadius || !explosionVelocity) return 1; // 字段缺失回退满额（向后兼容）
  const sigRatio = targetSig / explosionRadius;
  const velTerm = Math.pow(Math.min(1, explosionVelocity / Math.max(targetSpeed, 1)), drf);
  return Math.min(1, sigRatio * velTerm);
}
```

**drf = 0.5 定稿**（PRD 开放问题 2）。行为验算（NPC 火箭 expR 35 / expV 150 对各态玩家）：

| 目标状态 | sig | v (m/s) | factor | 判据 |
|----------|-----|---------|--------|------|
| 任意护卫静止 | ≥35 | 0 | **1.000** | 验收 2：静止=面板，全船体精确成立（expR=35=最小 sig） |
| Atron 全速 | 35 | 400 | **0.612** | 验收 1：显著减伤（−39%） |
| Atron 被 web 50% | 35 | 200 | **0.866** | 验收 1：web 后回升 |
| Atron 开 MWD | 175 | 2000 | **1.000** | 验收 1：bloom 满额（5×0.274=1.37→夹 1） |
| Incursus 全速 | 42 | 315 | 0.828 | 中速中信号只得浅减伤（正确形状） |
| Catalyst 全速 | 65 | 250 | 1.000 | 驱逐吃满导弹（sig 1.86 抵掉速度项）——大船该怕导弹 |

**NPC 导弹参数定稿**（三个火箭 NPC 统一，"对静止护卫 factor≈1"标定即 expR ≤ 全场最小 sig 35）：`guristas_frigate`、`angel_webber`、`angel_hunter` 的 `weapon.stats` 增加 `explosion_radius: 35, explosion_velocity: 150`（与玩家火箭 expV 同值；expR 取 35 而非玩家的 20，因为 NPC 火箭的标定对象是玩家船 sig 带 35–65，玩家火箭的标定对象是 NPC sig 带 38–42——各自贴住自己目标带的下缘）。

**玩家侧实效变化（PRD 未算的账，如实记录）**：玩家**火箭**（expR 20 ≪ NPC sig 38–42）sigRatio ≥1.9，对全部现有 NPC 恒 factor=1——零变化。玩家**轻导**（expR 40 / expV 170）对高速轨道 NPC 掉输出：对 Brawler（350 m/s）0.662、Slaver（260）0.768、Sniper（220）0.835、Hunter（390）0.627——LML 风筝 build 实效 −17%~−37%，**web 是解**（Meta web 后 Brawler 140 m/s → factor 1.0，导弹船带 web 是 EVE 正统的伤害投送技）。发布说明写明；TTK 带复核覆盖（A-7）。

**接线**：敌方导弹分支（BattleScene 敌 AI 段）`applyDamage(..., missileDamageFactor(effectiveSig, playerVel×1000, w.explosion_radius, w.explosion_velocity))`；玩家导弹分支 `factor × eff.damageMult.missile_weapon` 作 mult。命中即中语义不变（非目标：无飞行时间）。

### 2. FR-2 敌方电容（`src/data/npcs.js` + BattleScene）

**数据形态**：NPC 顶层加 `cap_capacity` / `cap_recharge`（GJ / 秒充满，与玩家船 `resources` 同名同位纲）；武器 `stats.cap_use`（每次开火，导弹 0）；`ewar` 块加 `activation_time: 5.0` 与 `cap_use`（镜像玩家 web 周期语义：电力充足时每周期付费、效果仍按帧检查射程包络——零差异由"付费永远成功"保证）。`buildEncounter` 零改动（顶层展开 + `weapon.stats` 展开自动透传，`ewar` 对象原样携带；伏击 encounter 同路，A-13 断言）。

**标定数学**（写入 balance.md）：站立消耗 `C = cap_use/rof + ewar_cap_use/activation_time`；回充峰值 `2.5 × cap_capacity / cap_recharge`；**稳定不变式：峰值 ≥ 1.45×C**（有余量的均衡点落在 55%–95% 电量区，远高于 10% 激活线 → 无中和时"跳过激活"永不触发 → 逐拍零差异）。**可中和不变式：T1 中和 10 GJ/s > 全场最大回充峰值 5.63**（压制在数学上必然收敛）。**停转不变式：`cap_recharge > 90s`** 时中和周期（5s）间隙的低电量区回充积累 `< 10% 线`（激活线设计成立的结构条件）。

**sim 实现**：`enemyCapRef = useRef(enemy.cap_capacity ?? Infinity)`（挂载即满电——"每场满电开局"由 BattleScene 每场重挂载天然保证，与玩家电容同语义、同样不入 HP 快照）；帧循环第 5 段与玩家并列 `getCapacitorRecharge` 回充；武器开火前与 ewar 周期付费前判 `cap ≥ npcActivationFloor(...)`，不足则武器 0.5s 重试（镜像射程外 hold）、ewar 断电（powered ref 置 false，效果分支多一个 `&& powered` 条件）；`combat.js` 新增纯函数：

```js
export const NPC_CAP_FLOOR_RATIO = 0.10;
export function npcActivationFloor(capCapacity, capUse) {
  return Math.max(capUse || 0, capCapacity * NPC_CAP_FLOOR_RATIO);
}
```

导弹武器（cap_use 0）**不进此判定**（免疫分支即"cap_use > 0 才检查"）。电量首次跌破/回穿激活线各写一条 combatLog（`powers down — capacitor drained!` / `systems back online`，powered ref 翻转驱动，非新类型分支）。无 cap 字段的 NPC（`Infinity` 回退）整段电容逻辑跳过——PRD 的向后兼容条款，实装后 8 个 NPC 全有字段，回退仅存于直入 fallback 的防御路径。

### 4. FR-4 伤害穿层（`src/lib/combat.js` applyDamage 本体）

**算法**：命中最外完整层 L 结算后，若结算伤害 > 该层余量：按"耗用比例"折出剩余 volley 份额 `frac = 1 − hp[L]/dmg`，L 清零，剩余份额带着**下一层的抗性**重新结算并落在下一层；**穿一层封顶**（第二层若又破，残余丢弃——现有 alpha 量级 wrecking 磁轨 ×3 也到不了穿两层，PRD 判断与数值核实一致）。船毁（hull 触底）溢出照旧丢弃。

**返回形状不变**：仍 `{ layer, dmg }`——`layer` = 首击层、`dmg` = 跨层合计。4 个调用点**零改动**（战斗日志"hits your shield for N"的 N 含穿透部分，语义微调可接受）；无溢出的 volley 与现行为**逐字节一致**（A-6 断言）。

**量级预估**（A-7/A-8 验证目标）：穿层只在破层帧生效。玩家承受侧：NPC volley 6–18 raw × 深度系数，破层溢出均值约半发 → 每场 +5~15 HP、**一段 5 层累计 +25~75 HP ≈ 玩家 EHP 池的 2–6%**（v0.9 交界的段级损耗视角）；玩家输出侧：滴灌型（爆能/导弹）几乎无感，alpha 型（磁轨、wrecking ×3）每场省 1–2 发 → TTK −3~−8%，验收 4"磁轨 fit 总 TTK 缩短"方向成立。

### 5. FR-5 堆叠惩罚（`src/lib/shipStats.js` mult 合并处）

**惩罚序列**（EVE 原式，写入锚点）：第 i 件（0 起）同 stat 模块 mult 的实效系数

```js
export const stackingPenalty = (i) => Math.pow(0.5, Math.pow(i / 2.22292081, 2));
// 1.0000, 0.8691, 0.5706, 0.2830, 0.1060, ...
```

**规则定稿**：只罚**模块 mult**；技能倍率（skillMult）、一切 `add`、resist（本就禁 mult）豁免。同 stat 的模块 mult 按**正向组（>1）/负向组（<1）分组**，各组内按 |value−1| 降序排位，第 i 位实效 `1 + (value−1) × stackingPenalty(i)`，组内连乘后再与技能因子相乘。分组理由：正负同罚但不互相占位（EVE 同向分栈语义；当前数据无正负混堆的 stat，规则是防御性的）。实现：`multFactors` 单因子累积改为 per-stat 数组收集 + 上述折算，应用循环不变。

**数值影响表**（验收 5 + PRD 开放问题"序列过陡"的回答）：

| 堆叠 | 旧组合值 | 新组合值 | 实效损失 |
|------|----------|----------|----------|
| 1× MFS I | 1.100 | **1.100** | 0（单件逐字不变，验收 5 后半） |
| 2× MFS I | 1.210 | **1.1956** | −1.2%（第 2 件收益 10%→8.69%，即 ×0.869） |
| 3× MFS I | 1.331 | **1.2638** | −5.0%（第 3 件 ×0.571） |
| MFS II + MFS I | 1.243 | 1.2295 | II 排首位全额，I 吃 0.869 |
| 2× Co-Processor（CPU） | 1.210 | 1.1956 | Incursus 135 tf：−1.94 tf |
| 4× Co-Processor | 1.4641 | 1.2996 | −22.2 tf（卸载死锁场景，核对结论 4） |
| 2× Nanofiber（hull ×0.90 负向） | 0.810 | 0.8218 | 负向堆叠变轻（drawback 减震） |

序列**不陡**：第 2 件仍拿 87% 收益，双增幅从"+21%"到"+19.6%"——PRD 担心的"从最优跌到不可用"不会发生，第 3 件（57%）才是真拐点，恰是"低槽机会成本"的设计意图。

**旧档容忍语义**（用户标注的 v0.9 交界项，逐点落实）：`merge` 重建 fit 不重验 → 超限档照常载入；BattleScene 不校验 `used` → 照常战斗、`resumeHp` 上限照常（HP 类 mult 无正向堆叠，上限不降）；`fitModule` 候选制校验 → 超限档装不进新件（"装不回去"）；`unfitModule` 按核对结论 4 放宽（已超限时卸载不恶化即放行）→ "可卸下"成立。

## 风险
