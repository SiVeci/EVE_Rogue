# 实施计划：v0.3 经济闭环 — 市场、首批模块变体、Kestrel 解锁

> 对应 PRD：[PRD/v0.3-economy-market.md](../PRD/v0.3-economy-market.md)
> 日期：2026-07-14。前置：v0.2 存档系统（已完成，存档 version 1）。

## 现状分析

- 软锁确认存在：蓝图只覆盖 2 种模块，阵亡销毁装配 → 爆能炮等 5 种模块永久不可再得。
- `activeShip` 永远存在（船体免费不死）；UNDOCK 按钮在 `App.jsx` 头部；战败由 BattleScene 调 `shipDestroyed()` 后 `onDefeat` 回空间站。
- 价格数据不存在；现有蓝图成本（10000/15000）高于 PRD 定价带，需要下调以满足 FR-9（制造价 < 市场价 20–30%）。
- BattleScene 按模块 `type` 分支消费 stats —— 7 件新变体（全部落在已有 6 类型内）自动可用，零引擎改动。

## 设计决策

### 1. 价格落点：数据文件加 `price` 字段

`modules.js` / `ships.js` 每个条目加 `price`（ISK）。市场/出售/保底判断都读它，不另建价格表。建议值：

| 条目 | price | | 条目 | price |
|------|-------|-|------|-------|
| Light Electron Blaster I | 5,000 | | Small Shield Booster I | 8,000 |
| Light Ion Blaster I ⊕ | 7,500 | | Medium Shield Booster I ⊕ | 12,000 |
| 125mm Railgun I ⊕ | 7,000 | | Small Armor Repairer I | 8,000 |
| Rocket Launcher I | 6,000 | | Small ACM Compact Armor Repairer ⊕ | 11,000 |
| Light Missile Launcher I ⊕ | 7,000 | | 1MN Afterburner I | 6,000 |
| Stasis Webifier I | 5,000 | | 1MN Monopropellant Enduring AB ⊕ | 9,000 |
| Fleeting Compact Stasis Webifier ⊕ | 8,000 | | **Incursus** | **60,000** |
| | | | **Kestrel** | **75,000** |

（⊕ = 本版新增，共 7 件；Meta 件 `tier: 'Meta'`，为 v0.6 "市场只卖 T1" 预留过滤字段。）
蓝图成本下调：bp_rocket 10000→4,500，bp_booster 15000→6,000（约为市场价 75%）。蓝图未入档，改动直接生效。

### 2. 新模块数值（DPS 锚定现有件）

- **Light Ion Blaster I**：optimal 1300 / falloff 2200 / tracking 320 / rof 3.2 / 伤害 th7.5+kin7.5（电子炮 +25%），pg5 cpu15。
- **125mm Railgun I**：optimal 9000 / falloff 3000 / tracking 90 / rof 4.0 / th4.5+kin4.5（DPS ≈ 电子炮 60%），pg6 cpu18。
- **Light Missile Launcher I**：range 22000 / rof 5.0 / kin15（DPS ≈ 火箭 65%），pg4 cpu22。
- **Medium Shield Booster I**：回量 60 / 2s / cap45，pg14 cpu28（超规格：强但吃装配）。
- **Small ACM Compact Armor Repairer**（Meta）：回量 40 / cap12（−25%）、pg6 cpu12。
- **Fleeting Compact Stasis Webifier**（Meta）：减速 60%，cpu32。
- **1MN Monopropellant Enduring Afterburner**（Meta）：×2.35，cap4。

### 3. 船体系统与"无船状态"

- store 新增 `ownedShips: ['incursus']`；**`activeShip` 允许为 `null`**（阵亡后、选船前）。
- `switchShip(shipId)`：当前装配全部卸回机库 → 载入目标船（空装配）。
- `buyShip(shipId)`：扣款入库；**若当前无活动船则自动激活**（减少一步操作）。
- `shipDestroyed()` 扩展：装配销毁、深度归 1（原逻辑）+ 从 `ownedShips` 移除当前船、`activeShip = null`；随后保底判定：`ownedShips` 为空且 ISK < 最低船价（60,000）→ 免费补一艘 Incursus 并激活（FR-7）。
- 无船门控：App.jsx UNDOCK 按钮在 `!activeShip` 时禁用；FittingWindow 在无船时显示占位面板（提示去市场买船/从机库激活）。`fitModule`/`unfitModule` 加 `!activeShip` 早退守卫。

### 4. UI

- **MarketWindow.jsx**（新组件）：左侧市场目录（SHIPS / HIGH / MID / LOW 分组，含关键属性与价格，ISK 不足禁用），右侧"你的机库"（模块列表 + 50% 售价卖出按钮）。风格沿用 panel + inline style。
- **StationHub**：新增 MARKET tab（放 FITTING 之后）。
- **FittingWindow**：中央环顶部加已拥有船体切换条（当前船高亮，点击其他船调 `switchShip`）。
- 船体不可出售（v0.3 范围外，防止"卖唯一船"边界）。

### 5. 存档版本 2 与迁移

- `partialize` 增加 `ownedShips`；`activeShip` 为 null 时序列化为 null。
- `SAVE_VERSION = 2`；`migrate` 改为版本感知：`version === 1` → 补 `ownedShips: [activeShip.shipId]` 升级；其他未知版本 → 丢档（维持 v0.2 行为）。
- `merge` 扩展：校验 `ownedShips`（过滤未知 id，空则回退 `['incursus']`……仅当 activeShip 也为空时才强制回退，避免推翻合法的"无船有钱"状态——实现时以"activeShip 非空但不在 ownedShips → 补录"兜底）。

## 验证计划

沿用 v0.2 的两层无浏览器验证（浏览器工具恢复则补真机冒烟）：

**A. Node 跨进程测试**（文件模拟 localStorage，真实 store 代码）：
1. 买卖闭环：买模块扣款入库 → 卖回 50%；ISK 不足购买被拒。
2. 买船 + 换船：Kestrel 入库；切换时装配全部回机库、无凭空消失/复制。
3. 阵亡链路：有余款 → 船体移除、activeShip=null、装配销毁、深度归 1；身无分文 → 免费 Incursus 激活（保底）。
4. 软锁根治：唯一爆能炮随船损毁后，市场可重新购得。
5. 存档迁移：手写 version 1 存档 → 加载后 `ownedShips` 自动补录，其余字段保留。
6. 持久化回归：v2 全字段（含 null activeShip、ownedShips）跨进程往返一致。

**B. SSR 渲染测试**：MarketWindow 渲染出商品与价格、余额不足禁用态；FittingWindow 无船占位分支、有船时切换条渲染。

**C.** `npm run lint`、`npm run build` 通过。

## 风险

- 定价/死亡惩罚强度需实玩校准，本版只保证量级合理（一次阵亡 ≈ 3–5 场巡逻收益）。
- `activeShip` 可空是本版最大不变量变化，所有直接消费点（FittingWindow、App、BattleScene 入口）必须门控覆盖；BattleScene 本身靠"无船禁止 UNDOCK"上游保证，不改其内部。
- merge 对 v2 新字段的容错要与 v0.2 同强度（未知船 id 过滤、类型校验）。
