**裁定清单**（PRD 未覆盖 / 与现状冲突的决策点；★ = 需用户拍板，文末汇总）：

1. **单成员"编队"不套用角色限制——支援型仍可 solo 出现**。若 size 1 也强制主位为 dps，Sniper/Scout/Webber 将从全部单敌遭遇中消失——与 FR-3"单成员编队与 v0.12 现状逐拍零回归"直接矛盾。裁定：size 1 → 全池加权（与 v0.12 逐字节同分布）；size ≥ 2 → 主位从 dps 子池抽、二三位从全池抽。FR-1 的"主位始终是输出型"字面限定于多敌编队。
2. **`rollGroupSize` 在 d1–5 零 rng 消耗**。d1–5 规模恒 1 是确定性结论，不掷即得——第一个段（d1–5）的 runmap 种子流与 v0.12 完全一致；d6+ 每次 `rollEncounter` 多消耗 1–3 个 draw（规模 1 + 席位 0–2），跨版本种子输出不稳定为既有事实（v0.7 起每版重生成期望），不构成协议破坏。rng 顺序钉为锁定协议：① 规模（仅 d≥6）② 主位 pick ③ 二位 pick ④ 三位 pick。
3. **出现权重定稿**（PRD"解锁深度上多敌占比 ~25–35%"取"多敌总占比"读法）：d6–10 双敌 30%（落带）；d11+ 三敌 15% + 双敌 20% = 多敌 35%（落带上缘，双敌份额被三敌部分替换）。深度门控与段边界天然对齐：段起点 1/6/11/16…，第二段（d6–10）恰是双敌首段、第三段（d11–15）恰是三敌首段——整段形态跃迁，可读性强。
4. **成员允许重复、允许跨派系混编**。PRD 未约束；独立抽取是最简形态且保留"双 Brawler 贴脸"这类极端组合作为深层威胁。跨派系混编的风味违和记为观察项（若刺眼，候选阀是二三位抽取对主位派系加权 ×2——记录不预装）。
5. **★ wreck 伏击沿用编队掷：d6+ 的伏击可以是双敌**。PRD 未提伏击。推荐**允许**：① 语义统一（伏击就是 `rollEncounter(depth,'patrol')`，特判单敌要新增参数）；② 伏击本就是"所见即所遇"的既有例外（25% 概率预掷但不可见），"sensor ghosts"文案已示警；③ 风险已由 25% 概率与 patrol 档定价。备选：`runmap.js:116` 的伏击调用强制单敌（需要 `rollEncounter` 加 `forceSingle` 参数，runmap 产生一行 diff）。本计划按"允许"编写，若拍板改判仅联动两处（npcs.js 参数 + runmap 一行）。
6. **elite 节点（含段末终层）同吃编队门控**。PRD 未排除；d10 段末可出 elite 双敌（statMult = 2.35×1.5×0.65 ≈ 2.29/成员）。保留：v0.14 Boss 将替换里程碑段末位，此前的"段末编队"是有价值的难度先行数据点。
7. **玩家 web / neut 只作用于主目标**。"唯一主目标"（非目标 3）的自然延伸：web 的每帧距离判定与 neut 的每周期扣蓝全部对主目标结算，切换目标即转移。已知代价：无法"web 追击者、打狙击手"——记为单目标模型的固有限制，写入 balance.md。
8. **敌方多 web 乘法叠加**。二三位可同时抽中两个 EWAR 支援（裁定 4 允许重复），两个 45–50% web 同时命中 → 玩家速度 ×(0.55×0.5)=0.275。乘法是 per-member 循环的自然代码形态；EVE 式 EWAR 堆叠惩罚不引入（引擎无先例，非目标"协同 AI"的邻接区）。写入 balance.md。
9. **成员开火 stagger：`weaponTimer` 初值 = 1.5 + 0.5×i**。三成员首轮齐射的 alpha 同步会瞬间打穿浅层护卫的盾层——0.5s 错峰是唯一新增的手感参数。i=0 时恒 1.5，零回归成立。
10. **掉落不吃席位系数**（FR-2 字面：系数明列"HP/伤害/赏金/SP"，掉落只说"求和"）。双敌节点模块 EV = 两表全额之和 ∈ [0.70, 0.90]/场（既有单表带 [0.35, 0.45] 的两倍）——多敌节点是**有意的富矿**：赏金/SP 按 1.3× 定价了强度，salvage 溢价补偿的是"必须全灭才能拿"的约束。观察项 + 候选阀（对多敌成员的掉落 chance 乘席位系数）记录于风险节，不预装。
11. **intel 卡逐成员行 + 总赏金行；`ROW_HEIGHT` 120 → 132**。FR-5 要求 per-member 派系/伤害徽标/射程/赏金全量展示，三成员在 120px 内放不下。布局常量是卡片与 SVG 连线的共同单一来源（`MapScreen.jsx:50-56`），改常量即全局安全。单成员卡保持现状 markup 分支（零视觉回归）。
12. **victory 用聚合总和字段结算，retreat 用 `killedRef`**。聚合 `reward`/`spReward` 定义为 Σ(逐成员 rounded 值)，与 killedRef 全灭累积值恒等（同一组数字的和）——victory 走总和字段避免依赖运行时累积，两者相等性入 A 组断言。
13. **多敌 intro 文案**：size>1 时默认开场日志改为 `"Gate activation complete. ${n} hostiles on scan."`；`introLog` prop（伏击文案）优先级不变。
14. **单成员战斗的 UI 面像素回归**：敌方面板在 size 1 时不渲染目标高亮描边与点选热区提示（成员卡形态统一但视觉与现状等价）；MapScreen 单成员卡走原 markup。"逐拍零回归"的模拟面由 A-8 钉死，UI 面按 v0.12"空舱单像素一致"同一口径处理。

## 现状分析

- **改动面收敛在三个文件**：`npcs.js`（生成端）、`BattleScene.jsx`（战斗端）、`MapScreen.jsx`（intel 端）。以下**八个文件零 diff 是本版的审计断言**：`combat.js`（全部数学复用——per-member 命中/伤害/电容/地板/轨道全走既有导出）、`shipStats.js`、`runmap.js`（聚合对象透明流过）、`loot.js`（逐成员各调一次 `rollLoot`，函数本身不动）、`lib/ammo.js`、`lib/drones.js`、`gameStore.js`（killedRef 结算复用 `addIsk`/`addSp`，无新 action、无新 key）、`App.jsx`（encounter 纯透传，三 outcome 回调签名不变）。
- **v0.12 攒下的先例逐一对位**：数组迭代（`dronesRef` → `enemiesRef`）、目标解析不新开火块（敌打无人机 → 玩家/无人机打成员）、转火簿记（`enemyAggroRef` → 每成员 `aggro`）、编队 HUD（drone chips → 成员卡列表）、"死亡由既有判定原位捕获"（2.5 段插位论证 → 死亡清扫段插位论证）。PRD"敌侧扩展不需要发明任何新原语"与代码现状核实一致——本版没有任何一处需要新数学或新状态形态。
- **最小纵切片的载具是"聚合形态迁移 + 单成员"**：先把全部调用点迁到 `{ members }` 聚合上、d1–5 天然全单敌，走通进场-输出-死亡清扫-三 outcome-结算全流程并通过零回归闸，再放开多成员。与 v0.12"先 1 架再编队"同构，且本版的两阶段有真实的中间态（聚合壳 + 单成员是完整可玩版本），不存在 v0.12 实施记录里"1 架与 N 架同一套代码"的塌缩问题——塌缩的只是敌 AI 循环本体，壳层迁移与多敌交互（点选/顺延/killedRef/多 web）是真实的增量。
- **深度门控与段结构天然对齐**：`deadspaceDepth` 按段 +5（1→6→11），双敌解锁深度 6 = 第二段起点、三敌解锁深度 11 = 第三段起点——玩家以"段"为单位感知形态跃迁，intel（FR-5）在每段首次出现新形态时承担教学。
- **zustand SSR 恒读初始态**（v0.8 教训）：本版 store 零 diff，B 组不需要 store 替换插件——MapScreen 纯 props 直喂 fixture 即可（v0.12 B-4 先例），是历版 B 组最轻的一次。
- **BattleScene 无法在 Node 下真实帧驱动**（Canvas/WebGL，v0.11 起的预设口径）：帧循环行为以"模型复演脚本（把 per-member AI 循环与死亡清扫规则按设计决策逐字实现）+ 静态门序核对"验证；其中**零回归复演采用双实现并跑**——同一脚本内并列实现 v0.12 单敌规则与 v0.13 单成员循环，喂同一 rng 流断言逐拍输出相同。真实交互（点选/顺延体感）归 D 组。

## 设计决策

### 1. 编队生成（`src/data/npcs.js`）——门控、权重与 rng 协议定稿

**新导出**：

```js
// 规模档位与出现权重（docs/balance.md "Enemy groups"）。d1–5 恒单敌且
// 零 rng 消耗（确定性结论不掷骰）；深度门控与段边界对齐（段起点 1/6/11…）。
export const GROUP_SIZE_TABLE = [
  { minDepth: 11, weights: [[1, 0.65], [2, 0.20], [3, 0.15]] },
  { minDepth: 6,  weights: [[1, 0.70], [2, 0.30]] },
  { minDepth: 1,  weights: [[1, 1.0]] }
];
// 席位系数：乘在成员 HP/伤害/赏金/SP 上（EWAR 强度与电容从不缩放——既有
// "EWAR strength is not scaled" 惯例的延伸）。
export const SEAT_MULTS = { 1: 1.0, 2: 0.65, 3: 0.5 };

export function rollGroupSize(depth, rng = Math.random)   // d<6 直接 return 1，不调 rng
export function pickWeightedNpc(depth, rng = Math.random, role = null)  // 第三参数向后兼容；role='dps' 时池过滤 NPCS[u.npc].role === 'dps'
```

**多敌占比核对**：d6–10 多敌 30% ∈ [25, 35] ✓；d11+ 多敌 20%+15% = 35% ∈ [25, 35] ✓（PRD 起步锚点带，两个解锁档均落带）。

**`rollEncounter(depth, nodeType, rng)` 聚合输出（rng 顺序锁定协议）**：

```js
// ① size = rollGroupSize(depth, rng)（d<6 零消耗）
// ② seat 1 = pickWeightedNpc(depth, rng, size > 1 ? 'dps' : null)   // 裁定 1：单敌全池
// ③④ seat 2/3 = pickWeightedNpc(depth, rng)                        // 全池（支援可入）
// members[i] = buildEncounter(npc_i, depth, nodeType, SEAT_MULTS[size])
return { members, size, reward: Σ members[i].reward, spReward: Σ members[i].spReward };
```

`buildEncounter(npc, depth, nodeType = 'patrol', seatMult = 1)`：`statMult = depthMult × eliteMult × seatMult`；`reward/spReward` 同乘 `seatMult` 后 round。**seatMult=1 时与 v0.12 输出逐字节相同**（零回归基石，A-2 deep-equal 断言）。`lootTable`/`ewar`/`ai`/`mobility`/`cap_*` 经 `...npc` 展开原样透传，不吃席位系数。`availableNpcs`/`pickWeightedNpc(depth, rng)` 两参调用（runmap 残骸加权）行为零变化。

**前向兼容注记（v0.14）**：Boss 即"编队主位成员"——聚合 shape 的 members 数组 + 逐成员自带 reward/spReward 正是 v0.14 需要的异构编队容器，本版不预建 Boss 字段。

### 3. FR-2 编队强度/奖励预算——验算（写入 balance.md，A-4 逐值断言）

**恒等式**：双敌总 HP/总名义 DPS/总赏金/总 SP = 同深同组合单敌和 ×0.65 = 混合期望的 **1.3×**；三敌 = **1.5×**。

**d6 双敌基准例**（guristas_frigate 主位 + serpentis_scout 二位；depthMult 1.75，seat 0.65 → statMult 1.1375）：

| 成员 | HP 盾/甲/构（缩放后） | 每轮伤害 | 名义 dps | 赏金 | SP | EWAR |
|------|------------------------|----------|----------|------|-----|------|
| Guristas Pirate | 250/182/228（660 raw） | kin 13.65 | 4.55 | 17,063 | 228 | — |
| Serpentis Scout | 216/228/228（672 raw） | th 9.10 + kin 7.96 | 5.33 | 22,181 | 228 | web 45% @ 9 km **全强度** |
| **编队合计** | **1,332 raw** | — | **9.88** | **39,244** | **456** | — |

对照：单敌 d6 同两型期望（26,250 + 34,125)/2 = 30,188 → ×1.3 = 39,244 ✓ 精确成立。

**DPS 红线核对**（红线：编队总名义 DPS ≤ 同深单敌最强档 ×1.5）：dps 角色单体名义 dps 极值 = Brawler 6.0（18 伤/3.0 rof）。最坏双敌 = 双 Brawler：2×6.0×0.65 = 7.8 = **1.30×** ✓；最坏三敌 = 三 Brawler：3×6.0×0.5 = 9.0 = **1.50×**——恰在红线上（≤ 成立），这是席位系数 0.5 的设计定点，不是巧合。

**TTK 落带**（PRD：双敌总 TTK ≈ 单敌 ×1.3–1.6）：HP 恒等给出精确 1.3×；目标切换开销（击毁后重新接近/角速度重建，成员轨道半径差异 1–15 km）加 3–8s/次 → 模型断言带 **[1.25, 1.65]×**。生存性粗核：d6 双 Brawler 入射名义 9.88–13.65 dps × 命中质量 ~0.775 × 玩家抗性 ~30% ≈ 5.3–7.4 有效 dps，Incursus 级 ~2,000 有效 EHP → 被击穿时限 270–380s ≫ 总 TTK ~90–130s，护卫在双敌下有生存空间（PRD 风险 1 的先决核对）。

**无人机损耗联动**（自然推论，写入 balance.md）：每成员独立 25%/6s 转火 roll → 双敌转火窗期望 ×2、每窗毁伤 ×0.65 → 多敌节点无人机损耗 ≈ ×1.3——与编队总强度同倍率，v0.12 损耗锚点按比例自洽，无需改任何 DRONE_* 常量。

**AC5 组合协同验算**（webber+sniper，全部来自既有公式）：玩家 5 km 轨道 315 m/s → 角速度 0.063 rad/s；对 Sniper（tracking 35、optimal 15 km，无距离惩罚）命中 = 0.5^((0.063×40000/(35×42))²) = 0.5^2.94 ≈ **13.0%**；被 webber 减速 50% → 角速度 0.0315 → 命中 0.5^0.735 ≈ **60.1%**。杀掉 webber = Sniper 命中率从 60% 崩回 13%（4.6 倍差）——组合协同零专用逻辑，A-5 以 ±1pp 断言两值。

**帧循环段序**（六段骨架不动，敌侧单敌直写变 per-member 循环；新增唯一一段 = 死亡清扫）：

1. **几何**：per-member 计算 `dist/distMeters/dirTo/angVel`（公式与现状逐字相同，存入成员条目）；HUD 距离取主目标。
2. **玩家模块**：目标解析 = `enemiesRef.current[targetIdxRef.current]`；武器两分支的 `distMeters/sig_radius/angularVelocity/defense/hp` 全部换读主目标成员字段；missile/neut 射程 hold 对主目标距离；web 置主目标 `webbed`、neut 扣主目标 `cap`（裁定 7）。分支结构/门序（reload→弹药门→电容门→付费→结算）逐字不动，`mod.type ===` 集合零新增。
3. **2.5 无人机**：目标 = 主目标成员（`enemyPos/enemyHp/enemy.defense` 三个硬引用换成员字段）；主目标切换/顺延时无人机下一帧自然跟随（每帧读 ref，无缓存）。其余逐行零改动。
4. **2.75 成员死亡清扫（本版唯一新段，插在 2.5 与敌 AI 之间）**：per member `alive && hp.hull <= 0` → `alive = false`；`killedRef.isk += def.reward`、`killedRef.sp += def.spReward`；日志 `${def.name} breaks apart!`；该成员 aggro 释放；若为主目标 → `targetIdxRef` 顺延到席位序首个存活成员（FR-4 自动顺延）+ `setForceRender`。随后**全灭判**：`every(!alive)` → `endBattle('victory')` 并 return——无人机击杀最后一员由同帧同位捕获，且清扫在敌 AI 之前，死亡成员当帧即停火（与现状"胜利判在敌 AI 前"语义逐字一致）。
5. **敌 AI**：`enemiesRef.current.forEach`，跳过 `!alive`；每成员内部依次跑 EWAR 块（cap/floor/powered 逐行照搬，玩家减速**乘法累积**——裁定 8）→ aggro 簿记（对无人机独立 roll，逐字照搬）→ 武器块（cap/floor 门在前、目标解析在后的 v0.12 门序不动：先解析是否锁无人机，否则打玩家；对玩家的 `tSig` 仍为 MWD bloom 后的 `effectiveSig`，对无人机为其自身 sig——全部现状逻辑，只是变量取自成员条目）。
6. **玩家 hull 判 → align 倒计时**：原位不动（同帧竞态仍按"胜利/阵亡先于撤退"解析，死亡清扫属胜利侧、在 align 判之前）。
7. **移动**：玩家不变；per-member `desiredVelocity({orbit, radius: def.ai.orbitRange}) + applyInertia`（逐字照搬进循环）。
8. **电容回充**：玩家不变；per-member 同款曲线，`Infinity` 池跳过。
9. **HUD 节流**：`enemyHud` 变数组 `[{ name, shield, armor, hull, cap, alive }]` + `targetIdx`；`droneHud` 照旧。

**交互**：`setTarget(i)`（守卫：`battleOverRef`、成员存活、i ≠ 当前）——HUD 成员卡 `onClick` 与 3D `ShipMesh onClick`（`e.stopPropagation()`）双入口，独立 handler（`startAlign` 先例，不进 `commandRef`）。**不做分火**（非目标）。

**表现层**：敌方 HUD 面板改成员卡纵列（每卡：名字 + S/A/H 迷你条 + cap 条[有 cap 者] + 死亡 ✕ 置灰；主目标青色描边——size 1 不渲染描边，裁定 14）；3D 每存活成员一个 `ShipMesh`（死亡即隐藏，与现状 `outcome !== 'victory'` 同款条件渲染）+ 主目标脚下扁平圆环标记（`torusGeometry`，positionRef 驱动；**可滑落表现层**，见实施步骤 6）。

## 风险

- **多敌 DPS 叠加压死护卫带（PRD 风险 1，最大项）**：观察信号 = D-1/D-5 与玩家反馈中 d6 双敌节点的死亡率/规避率异常。**单一调参杆 = 席位系数**（0.65/0.5 起步，PRD 自认"是拍的"）；红线 = 编队总名义 DPS ≤ 同深单敌 ×1.5（三敌 0.5 已构造性贴线，系数只能下调不能上调）；解锁深度（6/11）与出现权重是第二、三顺位阀，不与系数同动（一个目标一个旋钮）。A-4 的 TTK/红线断言是发布闸。
- **BattleScene 膨胀（PRD 风险 2）**：敌侧数组化后该文件承载玩家/敌编队/无人机/结算四类逻辑。防线三层：切片闸（步骤 3 的零回归全绿再放量）、停手判据（触碰玩家模块段或 2.5 段内部逻辑 / 模拟态入 React state → 停手）、PRD 授权的 `src/lib/enemyAi.js` 抽取阀（输入成员条目 + 玩家快照、输出动作，纯函数层——仅在停手触发时启用，不预建）。
- **零回归基准脆弱**：单成员路径的字节级等价横跨条目初值（spawn/timer/rot/aggro/cap 回退）、门序、目标解析默认值三层——任何一处手滑都让 FR-3 基准失效。防线：A-8 的双实现并跑（唯一能发现"语义相同但时序漂移"的手段）+ 初值表逐字段断言 + 八文件零 diff。
- **撤退部分结算削弱"全灭或全弃"张力（PRD 风险 4）**：观察信号 = 撤退率（尤其"击毁 N−1 员后必撤"模式）异常升高。回退阀 = banked 部分结算 ×0.5 折扣（PRD 预案，记录不预装）；掉落全弃 + 节点封闭两个既有代价已是第一道闸。
- **多敌节点掉落富矿（裁定 10 的代价）**：双敌模块 EV [0.70, 0.90]/场为单敌两倍。观察信号 = 玩家路径选择系统性偏向多敌节点且体感为"白捡"。候选阀 = 掉落 chance 乘席位系数（记录不预装——先观察，salvage 溢价本是全灭激励的定价）。
- **3D 点选精度（PRD 风险 3）**：小目标/重叠时 raycast 命中不稳。HUD 成员卡是保底路径（PRD 字面"不作为唯一路径"）；观察信号 = D-1 点选失败率；降级 = 保留 HUD 列表、场景点选与目标环随步骤 6 滑落。
- **三敌 HUD/战场读性**：右侧三张成员卡 + 无人机编队条的纵向堆叠在小屏可能溢出。封顶三敌是 PRD 非目标钉死的上限；若 D-4 观察到溢出，压缩成员卡高度（纯表现层，零逻辑改动）。
- **伏击编队无预警（★裁定 5 的余波）**：d6+ 的 25% 伏击可为双敌，玩家无 intel 可读。观察信号 = 伏击节点死亡率对普通双敌节点的显著超出；回退 = 伏击强制单敌（npcs.js 加 `forceSingle` 参数 + runmap 一行，改动面已勘定）。
