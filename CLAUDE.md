## Project Overview

EVE: Rogue is an EVE Online-inspired roguelike browser game: fit a frigate in station, undock into Abyssal Deadspace, fight through nodes for ISK, and go deeper. Built with React 19 + Vite in plain JSX (no TypeScript), Zustand for state, and three.js via @react-three/fiber for the 3D battle scene.

Note: git history contains a deleted Unity/C# prototype and design docs (removed in commit 358920f). The current codebase is web-only; ignore Unity references in old commits.

## Architecture

### View flow (App.jsx)

`App.jsx` is a three-view state machine held in local `useState`: `'station'` â†’ `'map'` â†’ `'space'`. UNDOCK (`startRun`) generates one dive's **5-layer branching segment map** (`generateRunMap`, see Run structure below) and holds it in a second local state, `runState: { map, at, playerHp, cleared, closed, notice } | null` â€” `at` is the last-cleared node id (`null` at the segment's start, when every layer-1 node is selectable), `playerHp` is the HP snapshot carried between nodes (`null` = full), `cleared`/`closed` are id lists driving the map's visited/dead-end styling, and `notice` is the transient wreck-loot popup. **`runState` is deliberately not persisted**: a refresh is a voluntary dock-out â€” per-node ISK/SP/loot settlement already landed in the store, but the map/position/HP are lost and the next UNDOCK re-rolls a fresh segment from the same depth. A third local state, `pendingNodeId`, tracks which node is currently being fought so `runState.at/cleared/closed` only update once BattleScene's outcome is confirmed. StationHub has four tabs: FITTING (FittingWindow, now also the hull-insurance control), MARKET (MarketWindow), INDUSTRY, and SKILLS. UNDOCK is disabled while `activeShip` is null (player was podded).

Selecting a map node branches on its type: `patrol`/`elite`/an ambushed `wreck` enters BattleScene; a `repair` node heals in place (`playerHp` reset to `null`) with no battle; a peaceful `wreck` (no ambush) banks its pre-rolled loot immediately via `addLoot` and shows the `notice` popup â€” neither non-combat node yields ISK/SP. BattleScene's three outcomes (see Combat below) drive distinct transitions: **victory** on a non-final node advances `at`/`cleared` and carries the returned HP snapshot to the next node; victory on the segment's final (always-elite) layer calls `advanceSegment()` (the only place `deadspaceDepth` advances â€” by `SEGMENT_LAYERS`, replacing the old per-victory `advanceDepth`) and returns to station; **retreat** closes the node for the segment (`reachableNodeIds` re-checked â€” an empty result means every path is now dead-ended, so the segment ends early with depth unchanged); **defeat** calls `shipDestroyed()` as before. The header's ABORT & DOCK button is hidden entirely in the `'space'` view (retreat is the only way out of a fight, not a free escape); in the `'map'` view it's a `window.confirm`-gated segment-forfeit action.

### Run structure (src/lib/runmap.js)

`generateRunMap(startDepth, rng?)` is a pure function (rng injected, same testing convention as `rollEncounter`) producing one dive's map: `{ startDepth, layers: node[][] }`, always `SEGMENT_LAYERS` (5) layers of 2â€“3 nodes each. A node's `depth` is `startDepth + layer index` â€” the sole scaling anchor for everything on it (combat stats, wreck loot tier gates, wreck NPC weighting all key off the node's own depth, never the segment's start depth). Layer 0 is always `patrol`; the final layer is always `elite` (2â€“3 side by side); the three middle layers roll `patrol`/`elite`/`wreck`/`repair` at 55/15/20/10% with post-roll distribution fixes (force at least one `repair` into the segment if none rolled, cap at 1 per layer / 2 per segment â€” see `docs/balance.md` "Run structure" for the exact numbers). Combat nodes pre-roll their `encounter` at generation time â€” the "what you scan is what you fight" convention now spans the whole segment. `wreck` nodes pre-roll a salvage pass (`loot`, via `rollLoot` against a depth-weighted NPC's table, patrol-grade) and a 25% `ambush` chance (a pre-rolled patrol-grade `encounter`; non-null replaces the peaceful pickup with a fight â€” the wreck was bait, so a node never pays both). Edges between adjacent layers come from a basic-coverage + gap-fill + non-crossing-widening pass, guaranteeing every node is reachable from layer 0 and every non-final node has an onward path. `reachableNodeIds(map, atNodeId, closedIds?)` and `isFinalLayer(map, nodeId)` are the runtime helpers App uses for node selection and segment-completion checks; `SEGMENT_LAYERS` is also what `gameStore.advanceSegment` adds to `deadspaceDepth`.

### Game state (src/store/gameStore.js)

Single Zustand store holding all persistent player state: ISK, skill points, skills, blueprints, module inventory, `deadspaceDepth`, `ownedShips` (hull id list, duplicates allowed), and `activeShip` (a ship definition plus `fittedModules: { high, mid, low }`, **or `null` while podded**). `fitModule`/`unfitModule` validate against `getEffectiveStats` (see below) computed with the module already added/removed â€” this makes capacity-raising modules like Co-Processor self-consistent (fit it first, then the module it enables fits) and blocks unfitting a module that would leave the remaining fit over PG/CPU. **`unfitModule`'s guard has one exemption (v0.10):** the stacking penalty on repeated Co-Processors can leave an old save's fit already over PG/CPU (a legacy fit that was legal before the penalty existed) â€” for a fit that's *already* over capacity, unfitting is allowed as long as it doesn't *worsen* the overage, otherwise no module could ever be removed from it. A legal (not-yet-over-limit) fit keeps the original guard verbatim: never unfit into a newly-over-capacity state. `fitModule` also enforces slot type, slot count per rack, a module's `maxPerShip` (e.g. Damage Control), and a module's `requiredSkills`. `skillMult(level)`, re-exported from `src/lib/shipStats.js`, defines the skill-bonus convention: level 1 is baseline, +5% per level past 1 â€” used only by the three stat skills (engineering/gunnery/navigation).

**Skills (src/data/skills.js):** `SKILLS` is the display registry (name/desc) for all ten skills â€” three stat skills that feed `skillMult` (default level **1**, the no-bonus baseline) and seven gate skills that only unlock content via `requiredSkills` on ships/modules (default level **0**, since they never feed `skillMult` and starting them at 1 would make a `requiredSkills: { x: 1 }` gate trivially pass at game start). `meetsRequiredSkills(requiredSkills, skills)` / `describeRequiredSkills(requiredSkills)` are the two helpers every gate (fitting, boarding, FittingWindow's lock hints) shares â€” a `requiredSkills` shape is always `{ skillKey: minLevel }`. Training cost for the next level is `1000 Ã— (currentLevel + 1)`, so a gate skill's first level costs 1,000 SP (not free). New skill keys need no save-version bump: `merge`'s existing `{ ...current.skills, ...persisted.skills }` shallow-spread already backfills any key missing from an old save to its default. `skillUnlocks(skillKey)` (v0.8) reverse-scans `SHIPS`/`MODULES` for `requiredSkills` entries and groups matches by required level â€” StationHub's `SkillsPanel` renders this under each gate skill so SP spend has a visible target; stat skills naturally return `[]` since nothing lists them in `requiredSkills`.

Ship boarding is gated the same way: `canBoard(ship, skills)` (private to gameStore.js) wraps `meetsRequiredSkills` and guards `switchShip` and `buyShip`'s auto-board branch â€” you can always *own* a hull (e.g. buy a Catalyst with no Destroyers skill), you just don't get seated in it until qualified. The insurance-granted hull is always Incursus, which has no `requiredSkills`, so the never-soft-lock guarantee is unaffected.

### Effective stats (src/lib/shipStats.js)

`getEffectiveStats(ship, fittedModules, skills)` is the single source of truth for a ship's real stats: base hull values, plus every fitted passive module's `modifiers`, plus skill bonuses (Engineeringâ†’PG/CPU, Navigationâ†’speed, Gunneryâ†’damage). `fittedModules` accepts either a flat array or the `{ high, mid, low }` slot object. Order is fixed: all `add` modifiers apply first, then all `mult` modifiers; resist modifiers are add-only and capped at 90. **Stacking penalty (v0.10):** module `mult` modifiers on the same stat are grouped by bonus (>1) vs. malus (<1) and ranked within each group by magnitude (`|value âˆ’ 1|`, descending); the i-th-ranked modifier keeps `stackingPenalty(i)` of its own bonus/malus (`src/lib/shipStats.js` â€” 100% / 86.9% / 57.1% / 28.3% / â€¦, the classic EVE sequence) before the group multiplies together. Skill multipliers and all `add` modifiers are exempt and always apply in full â€” only module `mult`s on the same stat compete for rank. Returns `{ resources, defense, mobility, damageMult: { hybrid_weapon, missile_weapon }, used: { pg, cpu } }` â€” `used` is what the fitted modules actually consume, compared against `resources.pg/cpu` to validate a fit. `activeModules(fittedModules)`, also exported here, filters out passive modules â€” this is what BattleScene uses to build its module rack, so passive modules never appear as toggleable and never need a branch in the combat loop. FittingWindow, `gameStore.fitModule`/`unfitModule`, and BattleScene's combat init all call `getEffectiveStats` â€” never read `ship.defense`/`resources`/`mobility` directly once a ship is in play, or passive modifiers and skills silently stop applying.

Market/hangar actions: `buyModule`/`sellModule` (sell = 50% of `price`), `buyShip` (auto-boards if podded), `switchShip` (strips the current fit back to the hangar, and voids any hull insurance policy â€” see below), `addLoot(moduleIds)` (combat reward â€” see Combat below â€” appends resolved modules to the hangar, silently dropping unknown ids). `shipDestroyed()` (defeat) destroys the hull and its fitted modules, resets depth to 1; ISK and hangar survive. Free-Incursus soft-lock guarantee: if the player ends up with no hull and less ISK than the cheapest hull, a free Incursus is granted (also applied when loading such a save) â€” the game can never soft-lock.

**Hull insurance (v0.9):** `insurance: { shipId } | null` is a one-time policy bound to the boarded hull, bought via `buyInsurance()` (FittingWindow) for 30% of the hull's price; `shipDestroyed()` pays out 100% of the hull's price (looked up live from `SHIPS`, not stored â€” the save only keeps the id) if the destroyed hull matches the policy, and the payout is added to `isk` **before** the free-Incursus soft-lock check runs (so a payout that alone covers a new hull correctly suppresses the freebie). Fittings are never covered. `switchShip` voids the policy with no refund (no `sellShip` action exists, so "sell voids it" only manifests via `switchShip` in practice). The invariant `insurance === null || insurance.shipId === activeShip.id` is enforced both by the actions above and by `merge` on load.

Saves persist to localStorage via `zustand/middleware` `persist` (key `eve-rogue-save`, version 2 â€” unchanged by v0.9's `insurance` addition, a zero-version-bump field: `partialize` stores `{ shipId }`, `merge` restores it only if the shipId is known and matches the boarded hull, else `null`). The save stores content **ids only** (`partialize`), so data-file balance changes apply to old saves; `merge` rebuilds objects from `MODULES`/`SHIPS`, silently dropping unknown ids, and any unreadable save falls back to a fresh start. `migrate` upgrades v1 saves (backfills `ownedShips`); unknown versions are discarded. Blueprints are static content and intentionally not persisted. **Run state (`runState` in App.jsx â€” the segment map, position, and HP) is never part of the store and is never persisted**, by design (see View flow above). `resetProgress` (station RESET PROGRESS button) wipes back to defaults, including `insurance`.

### Static content (src/data/)

`ships.js` and `modules.js` export plain objects keyed by id, using EVE conventions: ships have `slots` (high/mid/low counts), `resources` (pg/cpu/capacitor), `defense` (shield/armor/hull HP with per-damage-type resists, sig radius), `mobility` (incl. `agility`, the inertia time constant in seconds), and `price` (ISK, consumed by the market); modules have `slot` (which rack), `cost` ({pg, cpu, cap}), `tier` ('T1'/'Meta'/'T2'), `price`, and type-specific `stats`. Passive modules (armor plates, damage amps, shield extenders, Damage Control, Co-Processorâ€¦) instead set `passive: true` and a `modifiers` array of `{ stat, op: 'add'|'mult', value }` consumed by `getEffectiveStats` (see below); optionally `maxPerShip` (Damage Control is limited to 1). Either a ship or a module may carry `requiredSkills` (Catalyst/Cormorant need `destroyers: 1`; every T2 module needs its matching weapon/support skill at 4 or 5 â€” as of v0.8 this includes the two v0.6 T2 hybrid/missile damage amps, `light_neutron_blaster_ii` and `ballistic_control_system_ii`, which originally shipped without a gate) â€” see Game state above for how that's enforced. MarketWindow only lists `tier: 'T1'` modules for purchase â€” Meta and T2 are loot-only (see below) but remain sellable at 50% of `price`. Six hulls exist: Incursus/Kestrel/Merlin/Atron (`class: 'Frigate'`, no skill gate) and Catalyst/Cormorant (`class: 'Destroyer'`, 8/2/3 and 7/4/2 slots respectively, both gated) â€” Catalyst is a close-orbit blaster boat, Cormorant (v0.8) its long-range railgun mirror. `docs/balance.md` records the numeric anchors (EHP/speed/price bands, tier strength ladder, faction resist-signature convention) new ship/module/NPC content should be derived from.

`npcs.js` defines eight enemy archetypes across four factions (two ships each): Guristas (baseline missile boat, railgun sniper â€” both weak to kinetic), Serpentis (close-orbit blaster brawler, EWAR scout â€” both weak to thermal), Sansha's Nation (mid-range "laser" slaver, close-orbit ravager â€” both weak to EM), Angel Cartel (rocket boat with a stasis web, and the fastest hull in the game â€” both weak to explosive). All eight now carry an authored resist signature (weak type 20â€“30 points below the other three, across all three layers, per `docs/balance.md`) â€” the original four (v0.4) were retrofitted in v0.8 to close the gap with the four v0.7 NPCs that launched with one. Every NPC carries a `faction` field (read by MapScreen's intel display) and a `weapon.type` that reuses the two existing branches (`hybrid_weapon`/`missile_weapon`) â€” no new combat-loop type was introduced for any of them, including the speed-focused Angel Hunter, whose distinctiveness comes entirely from its base speed stat plus a small `ai.orbitRange`, not a new AI command. **NPC capacitor (v0.10):** every NPC now carries `cap_capacity`/`cap_recharge`, `weapon.stats.cap_use` (rocket-armed NPCs are `cap_use: 0` â€” cap-free on both sides, so they're structurally immune to neutralizer suppression), and â€” where an `ewar` block exists â€” `activation_time`/`cap_use` on that block too; see Combat below for how the sim spends and gates on it. The three rocket-armed NPCs (`guristas_frigate`, `angel_webber`, `angel_hunter`) also carry `weapon.stats.explosion_radius`/`explosion_velocity` (35/150), calibrated so a stationary player of any hull takes exactly full missile damage. None of this is scaled by depth/elite in `buildEncounter` (same rule as EWAR strength â€” see below). `ENCOUNTER_UNLOCKS` gates archetypes by depth and carries a `weight` per entry; `pickWeightedNpc(depth, rng?)` does a weighted draw over the currently-unlocked pool (so e.g. the depth-1 baseline Guristas frigate stays the single most likely pull even once every archetype has unlocked), and `rollEncounter(depth, nodeType, rng?)` wraps it with `buildEncounter`: HP and damage by depth (`1 + 0.15Ã—(depthâˆ’1)`, elite nodes Ã—1.5), ISK reward (elite Ã—2), and `spReward` (v0.8; same depth/elite scaling, base 200 â€” see `docs/balance.md` "SP economy"); EWAR strength (and now capacitor) is not scaled. `availableNpcs(depth)` still returns the plain unweighted array (unrelated call sites/tests that only need "what's unlocked" aren't affected by the weighting). Encounters are rolled in App's `goToMap` and passed down as props. Each NPC also carries a `lootTable` (`[{ moduleId, chance }]`), transparently spread through `buildEncounter` onto the encounter object; `rollLoot(encounter, depth, nodeType, rng?)` in `src/lib/loot.js` does one independent roll per entry (elite rolls the whole table twice), skipping entries whose module `tier` hasn't unlocked yet per `TIER_UNLOCKS` (T1 always, Meta at depth 4+, T2 at depth 7+) â€” the same table gets richer the deeper a run goes; the two Sansha NPCs' tables now also carry the Meta energy neutralizer (see below). `src/lib/tiers.js` holds the one `TIER_COLORS` lookup shared by MarketWindow, BattleScene's loot display, and FittingWindow's lock hints.

A module's `type` string ('hybrid_weapon', 'missile_weapon', 'shield_repair', 'armor_repair', 'propulsion', 'ewar', 'energy_neut') drives its combat behavior via branches in BattleScene's module-processing loop â€” adding a new active module type means adding a branch there. **Energy neutralizer (v0.10):** `energy_neut` (`small_energy_neutralizer_i`, T1/market; `small_infectious_scoped_energy_neutralizer`, Meta/loot-only via both Sansha tables) burns the target's capacitor by `stats.neut_amount` per `stats.activation_time`-second cycle at heavy self capacitor cost â€” the one new type branch this version adds. Passive modules (`passive: true`) never reach this loop at all â€” `activeModules()` filters them out before `modulesStateRef` is built, so a new passive module is pure data (a `modifiers` entry), zero engine changes.

### Combat (src/components/BattleScene.jsx)

The most complex file. Real-time sim inside a react-three-fiber `<Canvas>`, updated by a `useFrame` loop. Critical pattern: all simulation state lives in **refs** (positions, velocities, capacitor, shield/armor/hull in `playerHp`/`enemyHp`, module cooldown timers in `modulesStateRef`) and is mutated every frame; React state is only synced a few times per second for the HUD, plus a `setForceRender` call when module active-states change. Don't move sim state into React state. `PhysicsUpdater` is a module-level component that runs `stepRef.current()` each frame â€” the step closure is reassigned every render so it sees fresh state, while the component itself never remounts; movement commands reach the sim through `commandRef`.

The pure combat math lives in `src/lib/combat.js` (capacitor recharge curve, turret hit-chance formula, `rollTurretShot` â€” one roll doubles as the 1%-wrecking-shot quality roll, `applyDamage` with per-layer resists and one-layer bleed-through, `missileDamageFactor`, `npcActivationFloor`, movement command vectors, inertia) and is shared by both sides of the sim and by tests.

Implements EVE mechanics: capacitor recharge, turret hit chance (optimal/falloff distance penalty + tracking vs. relative angular velocity â€” symmetric, so both sides reuse `angularVelocity`), wrecking shots, per-layer damage resists, inertia (`agility` as time constant), missile range check (holds the cycle instead of wasting it), and approach / orbit / keep-at-range / stop commands. The player's max HP, capacitor, speed, agility, sig radius, and per-weapon-type damage multiplier are all read from `getEffectiveStats` (computed once per render via `useMemo`, not re-derived in the frame loop) â€” this is where passive-module bonuses and skills reach combat. The enemy comes from the pre-rolled `encounter` prop (or `rollEncounter(depth prop ?? store deadspaceDepth, nodeType)` as a direct-entry fallback) and orbits at `ai.orbitRange`; its weapon branches on `weapon.type` like the player's module loop (turrets roll to hit against the player's **effective signature** â€” MWD `sig_multiplier` bloom applies on top of any static sig modifier from passives like Shield Extender; missiles always hit in range). Damage the enemy deals is resolved against the player's effective (not base) resists, so Damage Control/EANM actually mitigate incoming damage. An enemy `ewar` block webs the player inside its envelope; the player's web halves enemy speed.

**Missile damage factor (v0.10):** `missileDamageFactor(targetSig, targetSpeed, explosionRadius, explosionVelocity)` (`src/lib/combat.js`, `MISSILE_DRF = 0.5`) replaces "missiles always deal full damage in range" with `min(1, (sig/expR) Ã— min(1, expV/v)^drf)` â€” a fast, small-signature target takes measurably less missile damage, and MWD's signature bloom (`sig_multiplier`) restores full damage since it inflates the sig term faster than the capped velocity term can shrink it. Both sides call it: the player's missile branch feeds it the enemy's (unmodified) sig/speed, the enemy's feeds it the player's **effective signature** (`eff.defense.sig_radius Ã— playerSigMult`, the same MWD-bloomed value the enemy's turrets already use) and `playerVel`. Fields default to 1 (full damage) if a weapon has no `explosion_radius`/`explosion_velocity` â€” back-compat for any future weapon that doesn't define them.

**Enemy capacitor & energy neutralizers (v0.10):** the enemy now runs the same `getCapacitorRecharge` curve as the player, in a plain ref (`enemyCapRef`, mounts full every fight, never part of the HP snapshot) that's skipped entirely for the `Infinity` fallback an NPC without `cap_capacity` gets. Its weapon and `ewar` block each carry a `cap_use`; whenever one is due to activate, it fires (and pays) only if `enemyCapRef.current >= npcActivationFloor(enemy.cap_capacity, cap_use)` (`max(cap_use, 10% of capacity)`, `src/lib/combat.js`) â€” below that floor the module is fully suppressed (a `powered` ref flips false/true and logs the transition) until recharge crosses back over it. Rocket-armed NPCs (`cap_use: 0`) never enter this check, so they're immune to neutralizer pressure. The player's `energy_neut` module (BattleScene's module loop) burns `stats.neut_amount` straight off `enemyCapRef` every cycle it's in range of `stats.optimal` â€” the only mechanism that can drive an NPC below its own activation floor, since standing NPC consumption is calibrated (see `docs/balance.md`) to never sag there on its own.

**Outcomes (v0.9): victory / defeat / retreat.** Battle ends when either side's hull reaches 0, or when the ALIGN & WARP retreat timer completes (`battleOverRef` freezes the sim in all three cases). **HP persists across nodes within a segment**: `initialHp` prop (`null` = full) is resolved via `resumeHp(initialHp, eff.defense)` (`src/lib/combat.js`) into both the HUD's initial state and the `playerHp` ref â€” capacitor is never part of this snapshot and always starts full. `depth` prop is the node's own depth (segment start + layer), read instead of the store's `deadspaceDepth` for both the fallback encounter roll and the victory-time `rollLoot` call, so a segment's later layers roll loot at their own (deeper) tier gates rather than the segment's start depth. `introLog` optionally overrides the opening combat-log line (used for the "Ambush!" wreck-fight text); `isFinalNode` swaps the victory button's label. Victory rolls loot via `rollLoot` immediately (into React state, not on button click, so the modal can't be re-rolled by dismissing it) and shows it in the victory modal alongside the bounty and `enemy.spReward`; confirming pays all three (`addIsk` + `addSp` + `addLoot`) and calls `onVictory({ ...playerHp.current })` â€” **BattleScene no longer advances depth itself** (`advanceDepth` is gone); that's now App's job, only on the segment's final node, via `advanceSegment()`. Defeat calls `shipDestroyed()` and pays no SP, same as before. **ALIGN & WARP (retreat):** a TACTICAL COMMANDS button starts an `ALIGN_TIME` (8s) countdown held in `alignRef` (the only new simulation state â€” a plain ref decremented in the frame loop, no new React state driving the sim); it's fully cancellable, and while it counts down the player stays fully vulnerable and modules/enemy AI keep running normally (no safety valve â€” a web makes the 8s lethal up close, which is the intended counterplay). Completion calls `endBattle('retreat')`, which â€” like defeat â€” rolls no loot; the retreat modal's confirm calls `onRetreat({ ...playerHp.current })` with **no store writes** (App alone decides whether the node closes into a dead end). A same-frame race between hull-zero and align-completion resolves in favor of victory/defeat, since those checks run first in the frame step and return early.

**Units convention:** data files use EVE units (meters for ranges, m/s for speed); the 3D scene uses world units where 1 unit = 1 km. Hence `dist * 1000` when comparing scene distance to module optimal/falloff, and `base_speed / 1000` for scene movement speed (0.315 u/s for a 315 m/s frigate). Angular velocity is therefore in real rad/s, which the hit formula's `Ã—40000` factor is calibrated for. Ships spawn 4 km apart; ship meshes are rendered at an exaggerated scale (0.25 group scale) for visibility.

## Current gaps (intentional, MVP in progress)

- Drone bay/bandwidth on ships are unused.
- Run mid-state (segment map, position, HP) is intentionally not persisted (v0.9) â€” a page refresh discards the in-progress dive (banked ISK/SP/loot are unaffected, since those settle into the store per node) rather than being restored. Revisit only if playtesting shows refresh-loss is a frequent frustration, not a rare accident.
- No player-side counter to being neutralized (cap injector/battery) and no NPC-side active neutralizer use (v0.10) â€” deferred until the strength of a player neut build is observed in play.
- No NPC active repair modules (v0.10 gives NPCs a capacitor budget that could support one, but none is wired up yet).
