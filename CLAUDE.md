# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

EVE: Rogue is an EVE Online-inspired roguelike browser game: fit a frigate in station, undock into Abyssal Deadspace, fight through nodes for ISK, and go deeper. Built with React 19 + Vite in plain JSX (no TypeScript), Zustand for state, and three.js via @react-three/fiber for the 3D battle scene.

Note: git history contains a deleted Unity/C# prototype and design docs (removed in commit 358920f). The current codebase is web-only; ignore Unity references in old commits.

## Commands

- `npm run dev` — start Vite dev server
- `npm run build` — production build
- `npm run preview` — serve the production build
- `npm run lint` — oxlint (not ESLint); config in `.oxlintrc.json`

There is no test framework configured.

## Architecture

### View flow (App.jsx)

`App.jsx` is a three-view state machine held in local `useState`: `'station'` → `'map'` → `'space'`. The game loop: StationHub (fit ship) → UNDOCK → MapScreen (pick a deadspace node) → BattleScene (combat) → victory pays ISK + advances depth → back to map, or dock to return to station. StationHub has four tabs: FITTING (FittingWindow), MARKET (MarketWindow), INDUSTRY, and SKILLS. UNDOCK is disabled while `activeShip` is null (player was podded). Entering the map goes through `goToMap()`, which pre-rolls both nodes' encounters (`rollEncounter`) and passes them to MapScreen (intel display) and BattleScene (the actual fight) — what the node shows is what spawns.

### Game state (src/store/gameStore.js)

Single Zustand store holding all persistent player state: ISK, skill points, skills, blueprints, module inventory, `deadspaceDepth`, `ownedShips` (hull id list, duplicates allowed), and `activeShip` (a ship definition plus `fittedModules: { high, mid, low }`, **or `null` while podded**). `fitModule`/`unfitModule` validate against `getEffectiveStats` (see below) computed with the module already added/removed — this makes capacity-raising modules like Co-Processor self-consistent (fit it first, then the module it enables fits) and blocks unfitting a module that would leave the remaining fit over PG/CPU. `fitModule` also enforces slot type, slot count per rack, a module's `maxPerShip` (e.g. Damage Control), and a module's `requiredSkills`. `skillMult(level)`, re-exported from `src/lib/shipStats.js`, defines the skill-bonus convention: level 1 is baseline, +5% per level past 1 — used only by the three stat skills (engineering/gunnery/navigation).

**Skills (src/data/skills.js):** `SKILLS` is the display registry (name/desc) for all ten skills — three stat skills that feed `skillMult` (default level **1**, the no-bonus baseline) and seven gate skills that only unlock content via `requiredSkills` on ships/modules (default level **0**, since they never feed `skillMult` and starting them at 1 would make a `requiredSkills: { x: 1 }` gate trivially pass at game start). `meetsRequiredSkills(requiredSkills, skills)` / `describeRequiredSkills(requiredSkills)` are the two helpers every gate (fitting, boarding, FittingWindow's lock hints) shares — a `requiredSkills` shape is always `{ skillKey: minLevel }`. Training cost for the next level is `1000 × currentLevel`, so a gate skill's first level costs 1,000 SP (not free). New skill keys need no save-version bump: `merge`'s existing `{ ...current.skills, ...persisted.skills }` shallow-spread already backfills any key missing from an old save to its default.

Ship boarding is gated the same way: `canBoard(ship, skills)` (private to gameStore.js) wraps `meetsRequiredSkills` and guards `switchShip` and `buyShip`'s auto-board branch — you can always *own* a hull (e.g. buy a Catalyst with no Destroyers skill), you just don't get seated in it until qualified. The insurance-granted hull is always Incursus, which has no `requiredSkills`, so the never-soft-lock guarantee is unaffected.

### Effective stats (src/lib/shipStats.js)

`getEffectiveStats(ship, fittedModules, skills)` is the single source of truth for a ship's real stats: base hull values, plus every fitted passive module's `modifiers`, plus skill bonuses (Engineering→PG/CPU, Navigation→speed, Gunnery→damage). `fittedModules` accepts either a flat array or the `{ high, mid, low }` slot object. Order is fixed: all `add` modifiers apply first, then all `mult` modifiers (module mults and skill mults combine per-stat before applying, so ordering among them never matters); resist modifiers are add-only and capped at 90. Returns `{ resources, defense, mobility, damageMult: { hybrid_weapon, missile_weapon }, used: { pg, cpu } }` — `used` is what the fitted modules actually consume, compared against `resources.pg/cpu` to validate a fit. `activeModules(fittedModules)`, also exported here, filters out passive modules — this is what BattleScene uses to build its module rack, so passive modules never appear as toggleable and never need a branch in the combat loop. FittingWindow, `gameStore.fitModule`/`unfitModule`, and BattleScene's combat init all call `getEffectiveStats` — never read `ship.defense`/`resources`/`mobility` directly once a ship is in play, or passive modifiers and skills silently stop applying.

Market/hangar actions: `buyModule`/`sellModule` (sell = 50% of `price`), `buyShip` (auto-boards if podded), `switchShip` (strips the current fit back to the hangar), `addLoot(moduleIds)` (combat reward — see Combat below — appends resolved modules to the hangar, silently dropping unknown ids). `shipDestroyed()` (defeat) destroys the hull and its fitted modules, resets depth to 1; ISK and hangar survive. Insurance clause: if the player ends up with no hull and less ISK than the cheapest hull, a free Incursus is granted (also applied when loading such a save) — the game can never soft-lock.

Saves persist to localStorage via `zustand/middleware` `persist` (key `eve-rogue-save`, version 2). The save stores content **ids only** (`partialize`), so data-file balance changes apply to old saves; `merge` rebuilds objects from `MODULES`/`SHIPS`, silently dropping unknown ids, and any unreadable save falls back to a fresh start. `migrate` upgrades v1 saves (backfills `ownedShips`); unknown versions are discarded. Blueprints are static content and intentionally not persisted. `resetProgress` (station RESET PROGRESS button) wipes back to defaults.

### Static content (src/data/)

`ships.js` and `modules.js` export plain objects keyed by id, using EVE conventions: ships have `slots` (high/mid/low counts), `resources` (pg/cpu/capacitor), `defense` (shield/armor/hull HP with per-damage-type resists, sig radius), `mobility` (incl. `agility`, the inertia time constant in seconds), and `price` (ISK, consumed by the market); modules have `slot` (which rack), `cost` ({pg, cpu, cap}), `tier` ('T1'/'Meta'/'T2'), `price`, and type-specific `stats`. Passive modules (armor plates, damage amps, shield extenders, Damage Control, Co-Processor…) instead set `passive: true` and a `modifiers` array of `{ stat, op: 'add'|'mult', value }` consumed by `getEffectiveStats` (see below); optionally `maxPerShip` (Damage Control is limited to 1). Either a ship or a module may carry `requiredSkills` (Catalyst needs `destroyers: 1`; every T2 module needs its matching weapon/support skill at 4 or 5) — see Game state above for how that's enforced. MarketWindow only lists `tier: 'T1'` modules for purchase — Meta and T2 are loot-only (see below) but remain sellable at 50% of `price`. Five hulls exist: Incursus/Kestrel/Merlin/Atron (`class: 'Frigate'`, no skill gate) and Catalyst (`class: 'Destroyer'`, 8/2/3 slots, gated). `docs/balance.md` records the numeric anchors (EHP/speed/price bands, tier strength ladder, faction resist-signature convention) new ship/module/NPC content should be derived from.

`npcs.js` defines eight enemy archetypes across four factions (two ships each): Guristas (baseline missile boat, railgun sniper), Serpentis (close-orbit blaster brawler, EWAR scout), Sansha's Nation (mid-range "laser" slaver, close-orbit ravager — both weak to EM), Angel Cartel (rocket boat with a stasis web, and the fastest hull in the game, weak to explosive). Every NPC carries a `faction` field (read by MapScreen's intel display) and a `weapon.type` that reuses the two existing branches (`hybrid_weapon`/`missile_weapon`) — no new combat-loop type was introduced for any of them, including the speed-focused Angel Hunter, whose distinctiveness comes entirely from its base speed stat plus a small `ai.orbitRange`, not a new AI command. `ENCOUNTER_UNLOCKS` gates archetypes by depth and carries a `weight` per entry; `pickWeightedNpc(depth, rng?)` does a weighted draw over the currently-unlocked pool (so e.g. the depth-1 baseline Guristas frigate stays the single most likely pull even once every archetype has unlocked), and `rollEncounter(depth, nodeType, rng?)` wraps it with `buildEncounter`: HP and damage by depth (`1 + 0.15×(depth−1)`, elite nodes ×1.5), ISK reward (elite ×2); EWAR strength is not scaled. `availableNpcs(depth)` still returns the plain unweighted array (unrelated call sites/tests that only need "what's unlocked" aren't affected by the weighting). Encounters are rolled in App's `goToMap` and passed down as props. Each NPC also carries a `lootTable` (`[{ moduleId, chance }]`), transparently spread through `buildEncounter` onto the encounter object; `rollLoot(encounter, depth, nodeType, rng?)` in `src/lib/loot.js` does one independent roll per entry (elite rolls the whole table twice), skipping entries whose module `tier` hasn't unlocked yet per `TIER_UNLOCKS` (T1 always, Meta at depth 4+, T2 at depth 7+) — the same table gets richer the deeper a run goes. `src/lib/tiers.js` holds the one `TIER_COLORS` lookup shared by MarketWindow, BattleScene's loot display, and FittingWindow's lock hints.

A module's `type` string ('hybrid_weapon', 'missile_weapon', 'shield_repair', 'armor_repair', 'propulsion', 'ewar') drives its combat behavior via branches in BattleScene's module-processing loop — adding a new active module type means adding a branch there. Passive modules (`passive: true`) never reach this loop at all — `activeModules()` filters them out before `modulesStateRef` is built, so a new passive module is pure data (a `modifiers` entry), zero engine changes.

### Combat (src/components/BattleScene.jsx)

The most complex file. Real-time sim inside a react-three-fiber `<Canvas>`, updated by a `useFrame` loop. Critical pattern: all simulation state lives in **refs** (positions, velocities, capacitor, shield/armor/hull in `playerHp`/`enemyHp`, module cooldown timers in `modulesStateRef`) and is mutated every frame; React state is only synced a few times per second for the HUD, plus a `setForceRender` call when module active-states change. Don't move sim state into React state. `PhysicsUpdater` is a module-level component that runs `stepRef.current()` each frame — the step closure is reassigned every render so it sees fresh state, while the component itself never remounts; movement commands reach the sim through `commandRef`.

The pure combat math lives in `src/lib/combat.js` (capacitor recharge curve, turret hit-chance formula, `rollTurretShot` — one roll doubles as the 1%-wrecking-shot quality roll, `applyDamage` with per-layer resists, movement command vectors, inertia) and is shared by both sides of the sim and by tests.

Implements EVE mechanics: capacitor recharge, turret hit chance (optimal/falloff distance penalty + tracking vs. relative angular velocity — symmetric, so both sides reuse `angularVelocity`), wrecking shots, per-layer damage resists, inertia (`agility` as time constant), missile range check (holds the cycle instead of wasting it), and approach / orbit / keep-at-range / stop commands. The player's max HP, capacitor, speed, agility, sig radius, and per-weapon-type damage multiplier are all read from `getEffectiveStats` (computed once per render via `useMemo`, not re-derived in the frame loop) — this is where passive-module bonuses and skills reach combat. The enemy comes from the pre-rolled `encounter` prop and orbits at `ai.orbitRange`; its weapon branches on `weapon.type` like the player's module loop (turrets roll to hit against the player's **effective signature** — MWD `sig_multiplier` bloom applies on top of any static sig modifier from passives like Shield Extender; missiles always hit in range). Damage the enemy deals is resolved against the player's effective (not base) resists, so Damage Control/EANM actually mitigate incoming damage. An enemy `ewar` block webs the player inside its envelope; the player's web halves enemy speed. Battle ends when either side's hull reaches 0 (`battleOverRef` freezes the sim). Victory rolls loot via `rollLoot` immediately (into React state, not on button click, so the modal can't be re-rolled by dismissing it) and shows it in the victory modal alongside the bounty; confirming pays both (`addIsk` + `addLoot`) and advances depth. Defeat calls `shipDestroyed()`.

**Units convention:** data files use EVE units (meters for ranges, m/s for speed); the 3D scene uses world units where 1 unit = 1 km. Hence `dist * 1000` when comparing scene distance to module optimal/falloff, and `base_speed / 1000` for scene movement speed (0.315 u/s for a 315 m/s frigate). Angular velocity is therefore in real rad/s, which the hit formula's `×40000` factor is calibrated for. Ships spawn 4 km apart; ship meshes are rendered at an exaggerated scale (0.25 group scale) for visibility.

### Styling

EVE-style dark sci-fi theme defined as CSS custom properties in `src/index.css` (`--color-*`, `--font-display` = Orbitron for headers/numbers, Inter for body). Components mostly use inline styles referencing those variables, with a few dedicated CSS files (e.g. `FittingWindow.css`). Global `button`, heading, and glass-panel styles come from `index.css`.

## Current gaps (intentional, MVP in progress)

- Missile `explosion_radius` / `explosion_velocity` stats exist but are unused (missiles simply always hit in range); the enemy has no capacitor simulation.
- No stacking penalty on repeated `mult` modifiers (e.g. two Magnetic Field Stabilizers apply the full ×1.10 each, ×1.21 combined) — intentional for now, revisit if it makes stacking pure-damage passives a dominant strategy.
- Drone bay/bandwidth on ships are unused.
- Damage does not bleed through layers within a volley (a volley that breaks a layer wastes its overflow).
- Only the 4 newest NPCs (Sansha ×2, Serpentis Scout, Angel Hunter) have a deliberately-authored weak-damage-type resist signature (see `docs/balance.md`); the 4 v0.4 NPCs keep their original generic layer resists and aren't retrofitted.
- Catalyst is the only destroyer hull; a second (Cormorant) was deferred pending balance feedback on the first.
