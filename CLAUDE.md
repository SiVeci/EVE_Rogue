# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

EVE: Rogue is an EVE Online-inspired roguelike browser game: fit a frigate in station, undock into Abyssal Deadspace, fight through nodes for ISK, and go deeper. Built with React 19 + Vite in plain JSX (no TypeScript), Zustand for state, and three.js via @react-three/fiber for the 3D battle scene.

Note: git history contains a deleted Unity/C# prototype and design docs (removed in commit 358920f). The current codebase is web-only; ignore Unity references in old commits.

## Commands

- `npm run dev` — start Vite dev server
- `npm run build` — production build
- `npm run preview` — serve the production build
- `npm run lint` — oxlint (not ESLint); config in `.oxlintrc.json`

There is no test framework configured.

## Architecture

### View flow (App.jsx)

`App.jsx` is a three-view state machine held in local `useState`: `'station'` → `'map'` → `'space'`. UNDOCK (`startRun`) generates one dive's **5-layer branching segment map** (`generateRunMap`, see Run structure below) and holds it in a second local state, `runState: { map, at, playerHp, cleared, closed, notice } | null` — `at` is the last-cleared node id (`null` at the segment's start, when every layer-1 node is selectable), `playerHp` is the HP snapshot carried between nodes (`null` = full), `cleared`/`closed` are id lists driving the map's visited/dead-end styling, and `notice` is the transient wreck-loot popup. **`runState` is deliberately not persisted**: a refresh is a voluntary dock-out — per-node ISK/SP/loot settlement already landed in the store, but the map/position/HP are lost and the next UNDOCK re-rolls a fresh segment from the same depth. A third local state, `pendingNodeId`, tracks which node is currently being fought so `runState.at/cleared/closed` only update once BattleScene's outcome is confirmed. StationHub has four tabs: FITTING (FittingWindow, now also the hull-insurance control), MARKET (MarketWindow), INDUSTRY, and SKILLS. UNDOCK is disabled while `activeShip` is null (player was podded).

Selecting a map node branches on its type: `patrol`/`elite`/an ambushed `wreck` enters BattleScene; a `repair` node heals in place (`playerHp` reset to `null`) with no battle; a peaceful `wreck` (no ambush) banks its pre-rolled loot immediately via `addLoot` and shows the `notice` popup — neither non-combat node yields ISK/SP. BattleScene's three outcomes (see Combat below) drive distinct transitions: **victory** on a non-final node advances `at`/`cleared` and carries the returned HP snapshot to the next node; victory on the segment's final (always-elite) layer calls `advanceSegment()` (the only place `deadspaceDepth` advances — by `SEGMENT_LAYERS`, replacing the old per-victory `advanceDepth`) and returns to station; **retreat** closes the node for the segment (`reachableNodeIds` re-checked — an empty result means every path is now dead-ended, so the segment ends early with depth unchanged); **defeat** calls `shipDestroyed()` as before. **v0.11 revises v0.9's "retreat writes nothing to the store" invariant to "retreat writes only ammo consumption"**: all three outcomes now call `settleBattleAmmo` at their confirm button (see Combat below) — bounty/SP/salvage are still forfeit on retreat, only the fired rounds are deducted from cargo either way. The header's ABORT & DOCK button is hidden entirely in the `'space'` view (retreat is the only way out of a fight, not a free escape); in the `'map'` view it's a `window.confirm`-gated segment-forfeit action.

### Run structure (src/lib/runmap.js)

`generateRunMap(startDepth, rng?)` is a pure function (rng injected, same testing convention as `rollEncounter`) producing one dive's map: `{ startDepth, layers: node[][] }`, always `SEGMENT_LAYERS` (5) layers of 2–3 nodes each. A node's `depth` is `startDepth + layer index` — the sole scaling anchor for everything on it (combat stats, wreck loot tier gates, wreck NPC weighting all key off the node's own depth, never the segment's start depth). Layer 0 is always `patrol`; the final layer is always `elite` (2–3 side by side); the three middle layers roll `patrol`/`elite`/`wreck`/`repair` at 55/15/20/10% with post-roll distribution fixes (force at least one `repair` into the segment if none rolled, cap at 1 per layer / 2 per segment — see `docs/balance.md` "Run structure" for the exact numbers). Combat nodes pre-roll their `encounter` at generation time — the "what you scan is what you fight" convention now spans the whole segment. `wreck` nodes pre-roll a salvage pass (`loot`, via `rollLoot` against a depth-weighted NPC's table, patrol-grade) and a 25% `ambush` chance (a pre-rolled patrol-grade `encounter`; non-null replaces the peaceful pickup with a fight — the wreck was bait, so a node never pays both). Edges between adjacent layers come from a basic-coverage + gap-fill + non-crossing-widening pass, guaranteeing every node is reachable from layer 0 and every non-final node has an onward path. `reachableNodeIds(map, atNodeId, closedIds?)` and `isFinalLayer(map, nodeId)` are the runtime helpers App uses for node selection and segment-completion checks; `SEGMENT_LAYERS` is also what `gameStore.advanceSegment` adds to `deadspaceDepth`.

### Game state (src/store/gameStore.js)

Single Zustand store holding all persistent player state: ISK, skill points, skills, module inventory, `cargo` (v0.11 — see below), `droneHangar` (v0.12 — see below), `deadspaceDepth`, `ownedShips` (hull id list, duplicates allowed), and `activeShip` (a ship definition plus `fittedModules: { high, mid, low }`, `ammo` — an array parallel to `fittedModules.high` — and `drones`, a `droneId[]` loadout manifest, **or `null` while podded**). Blueprints (`src/data/blueprints.js`, v0.11) are static content read directly by `manufacture`/StationHub — they are not store state and never were persisted, so moving them out of `initialState` is a zero-save-impact refactor. `fitModule`/`unfitModule` validate against `getEffectiveStats` (see below) computed with the module already added/removed — this makes capacity-raising modules like Co-Processor self-consistent (fit it first, then the module it enables fits) and blocks unfitting a module that would leave the remaining fit over PG/CPU. **`unfitModule`'s guard has one exemption (v0.10):** the stacking penalty on repeated Co-Processors can leave an old save's fit already over PG/CPU (a legacy fit that was legal before the penalty existed) — for a fit that's *already* over capacity, unfitting is allowed as long as it doesn't *worsen* the overage, otherwise no module could ever be removed from it. A legal (not-yet-over-limit) fit keeps the original guard verbatim: never unfit into a newly-over-capacity state. `fitModule` also enforces slot type, slot count per rack, a module's `maxPerShip` (e.g. Damage Control), and a module's `requiredSkills`. `skillMult(level)`, re-exported from `src/lib/shipStats.js`, defines the skill-bonus convention: level 1 is baseline, +5% per level past 1 — used only by the three stat skills (engineering/gunnery/navigation).

**Skills (src/data/skills.js):** `SKILLS` is the display registry (name/desc) for all eleven skills — three stat skills that feed `skillMult` (default level **1**, the no-bonus baseline) and eight gate skills that only unlock content via `requiredSkills` on ships/modules (default level **0**, since they never feed `skillMult` and starting them at 1 would make a `requiredSkills: { x: 1 }` gate trivially pass at game start). `meetsRequiredSkills(requiredSkills, skills)` / `describeRequiredSkills(requiredSkills)` are the two helpers every gate (fitting, boarding, FittingWindow's lock hints) shares — a `requiredSkills` shape is always `{ skillKey: minLevel }`. Training cost for the next level is `1000 × (currentLevel + 1)`, so a gate skill's first level costs 1,000 SP (not free). New skill keys need no save-version bump: `merge`'s existing `{ ...current.skills, ...persisted.skills }` shallow-spread already backfills any key missing from an old save to its default. `skillUnlocks(skillKey)` (v0.8) reverse-scans `SHIPS`/`MODULES` for `requiredSkills` entries and groups matches by required level — StationHub's `SkillsPanel` renders this under each gate skill so SP spend has a visible target; stat skills naturally return `[]` since nothing lists them in `requiredSkills`. **`drones` (v0.12) is a third consumption shape**: a gate skill (default 0, same training curve) whose *level itself* is read as a number — `maxDronesInSpace(ship, skills)` (`src/lib/drones.js`) takes `min(floor(ship.drone_bandwidth / 5), skills.drones)` — rather than only gating a boolean `requiredSkills` check. It still also gates Algos' `requiredSkills: { destroyers: 1, drones: 1 }` boarding check the normal way, so `skillUnlocks('drones')` still surfaces `Lv 1 → Algos` in `SkillsPanel` for free.

Ship boarding is gated the same way: `canBoard(ship, skills)` (private to gameStore.js) wraps `meetsRequiredSkills` and guards `switchShip` and `buyShip`'s auto-board branch — you can always *own* a hull (e.g. buy a Catalyst with no Destroyers skill), you just don't get seated in it until qualified. The insurance-granted hull is always Incursus, which has no `requiredSkills`, so the never-soft-lock guarantee is unaffected.

### Effective stats (src/lib/shipStats.js)

`getEffectiveStats(ship, fittedModules, skills)` is the single source of truth for a ship's real stats: base hull values, plus every fitted passive module's `modifiers`, plus skill bonuses (Engineering→PG/CPU, Navigation→speed, Gunnery→damage). `fittedModules` accepts either a flat array or the `{ high, mid, low }` slot object. Order is fixed: all `add` modifiers apply first, then all `mult` modifiers; resist modifiers are add-only and capped at 90. **Stacking penalty (v0.10):** module `mult` modifiers on the same stat are grouped by bonus (>1) vs. malus (<1) and ranked within each group by magnitude (`|value − 1|`, descending); the i-th-ranked modifier keeps `stackingPenalty(i)` of its own bonus/malus (`src/lib/shipStats.js` — 100% / 86.9% / 57.1% / 28.3% / …, the classic EVE sequence) before the group multiplies together. Skill multipliers and all `add` modifiers are exempt and always apply in full — only module `mult`s on the same stat compete for rank. Returns `{ resources, defense, mobility, damageMult: { hybrid_weapon, missile_weapon }, used: { pg, cpu } }` — `used` is what the fitted modules actually consume, compared against `resources.pg/cpu` to validate a fit. `activeModules(fittedModules)`, also exported here, filters out passive modules — this is what BattleScene uses to build its module rack, so passive modules never appear as toggleable and never need a branch in the combat loop. FittingWindow, `gameStore.fitModule`/`unfitModule`, and BattleScene's combat init all call `getEffectiveStats` — never read `ship.defense`/`resources`/`mobility` directly once a ship is in play, or passive modifiers and skills silently stop applying.

Market/hangar actions: `buyModule`/`sellModule` (sell = 50% of `price`), `buyShip` (auto-boards if podded), `switchShip` (strips the current fit back to the hangar, and voids any hull insurance policy — see below), `addLoot(moduleIds)` (combat reward — see Combat below — appends resolved modules to the hangar, silently dropping unknown ids). `shipDestroyed()` (defeat) destroys the hull and its fitted modules, resets depth to 1; ISK and hangar survive. Free-Incursus soft-lock guarantee: if the player ends up with no hull and less ISK than the cheapest hull, a free Incursus is granted (also applied when loading such a save) — the game can never soft-lock.

**Hull insurance (v0.9):** `insurance: { shipId } | null` is a one-time policy bound to the boarded hull, bought via `buyInsurance()` (FittingWindow) for 30% of the hull's price; `shipDestroyed()` pays out 100% of the hull's price (looked up live from `SHIPS`, not stored — the save only keeps the id) if the destroyed hull matches the policy, and the payout is added to `isk` **before** the free-Incursus soft-lock check runs (so a payout that alone covers a new hull correctly suppresses the freebie). Fittings are never covered. `switchShip` voids the policy with no refund (no `sellShip` action exists, so "sell voids it" only manifests via `switchShip` in practice). The invariant `insurance === null || insurance.shipId === activeShip.id` is enforced both by the actions above and by `merge` on load.

**Ammo cargo (v0.11):** `cargo: { [ammoId]: qty }` is a stacked-quantity commodity store, deliberately separate from `inventory` (module inventory is a per-item array; ammo trades and stores as a count) — see `src/data/ammo.js`/`src/lib/ammo.js` below and the Combat section for how it's spent. `activeShip.ammo` is an array index-aligned with `fittedModules.high` (`ammo.length === fittedModules.high.length` is the invariant): a weapon slot holds its assigned ammo id (auto-set to the family default by `fitModule`, via `defaultAmmoIdFor`), a non-weapon high-slot module holds `null`. `fitModule`/`unfitModule` push/splice `ammo` in lockstep with `fittedModules.high`; `switchShip` resets it to `[]` (via `freshShip`, which always seeds `ammo: []`). Market actions: `buyAmmo`/`sellAmmo` trade whole `AMMO_LOT` lots/stacks (mirroring `buyModule`/`sellModule`'s silent-failure pattern); `setWeaponAmmo(highIndex, ammoId)` is FittingWindow's sole write path, validated against the weapon's `ammoFamily`. `addLoot` now accepts a **mixed-shape array**: string module ids (unchanged since v0.6) route to `inventory`, `{ ammoId, qty }` objects (v0.11, from `rollLoot` — see Static content) route to `cargo`. `settleBattleAmmo({ consumed, assignments })` is BattleScene's battle-end ammo settlement (see Combat below) — called by all three outcomes; the free-Incursus soft-lock guarantee below extends to ammo via `topUpAmmoSafety` (tops each family default up to `AMMO_SAFETY_QTY`, both in `shipDestroyed` and `merge`'s equivalent branch).

**Drone hangar and loadout (v0.12):** `droneHangar: { [droneId]: qty }` is a station-side stacked stock, deliberately its **own** key rather than folded into `cargo` — `merge`'s per-key validation checks `AMMO[id]`, so a shared key would silently wipe drone ids on load (see the plan's decision log). `activeShip.drones: droneId[]` is the boarded hull's loadout manifest (array order = launch priority — no index-alignment invariant like `ammo`, it's just a flat list); `freshShip` always seeds `drones: []`. `buyDrone`/`sellDrone` trade one unit at a time (module granularity, unlike ammo's whole-lot trade); `loadDrone`/`unloadDrone` move a drone between `droneHangar` and the manifest, gated by `canLoadDrone` (`src/lib/drones.js` — the ship's `drone_bay` m³ budget) and hangar stock. **`switchShip` strips the manifest back to `droneHangar` by id count** — the easiest v0.12 touchpoint to miss (skipping it evaporates drones instead of banking them), mirrored after the existing fitted-module strip-to-inventory. `settleBattleDrones({ lost })` (BattleScene's battle-end drone-loss settlement, called by victory/retreat only — see Combat below) splices destroyed ids out of the manifest; **`shipDestroyed()` needs zero new code for the defeat case** — discarding `activeShip` already destroys the manifest with the hull, while `droneHangar` (station stock, untouched by that action) survives, exactly like `cargo` survives death. The free-Incursus/ammo-safety soft-lock grant does **not** extend to drones (no "must have a drone to fight" pressure the way ammo has "must have ammo to fire") — `freshShip`'s `drones: []` is the only effect. `addLoot`'s mixed-shape array gets a third route: `{ droneId, qty }` objects (from `rollLoot`, tier-gated like modules — see Static content) go to `droneHangar`.

Saves persist to localStorage via `zustand/middleware` `persist` (key `eve-rogue-save`, **version 3 as of v0.11, unchanged by v0.12** — `SAVE_VERSION` bumped for `cargo`/`activeShip.ammo`; v0.9's `insurance` addition and v0.12's `droneHangar`/`activeShip.drones` addition were both zero-version-bump fields, since "missing key defaults to empty" is the entire migration for a pure-increment feature nobody is gifted). The save stores content **ids only** (`partialize`), so data-file balance changes apply to old saves; `merge` rebuilds objects from `MODULES`/`SHIPS`/`AMMO`/`DRONES`, silently dropping unknown ids, and any unreadable save falls back to a fresh start. `migrate` chains v1 and v2 saves through the same v3 upgrade path (v1 gets its existing `ownedShips` backfill first): both are gifted `STARTER_CARGO` and have any already-fitted weapons auto-assigned their family default ammo; unknown versions are discarded. `merge` then re-validates `cargo` (per-key `AMMO[id]` + finite non-negative qty check, unknown ids dropped) and re-aligns `activeShip.ammo` by walking the **original, unfiltered** `fitted.high` id sequence in lockstep with the persisted ammo array — an unknown module id drops its ammo entry too (keeping alignment with `fromIds`' own silent drop), a surviving weapon's invalid/mismatched-family ammo id self-heals to its default, and a save missing the `ammo` key entirely rebuilds it from scratch. `merge` mirrors the same per-key validation for `droneHangar` (against `DRONES[id]` instead of `AMMO[id]`) and filters `activeShip.drones` to known ids, then truncates from the tail against the boarded hull's own `drone_bay` if a hand-edited save overloaded it. **Run state (`runState` in App.jsx — the segment map, position, and HP) is never part of the store and is never persisted**, by design (see View flow above). `resetProgress` (station RESET PROGRESS button) wipes back to defaults, including `insurance`, `cargo` (reset to `STARTER_CARGO`) and `droneHangar` (reset to `{}` — no starter gift, see above).

### Static content (src/data/)

`ships.js` and `modules.js` export plain objects keyed by id, using EVE conventions: ships have `slots` (high/mid/low counts), `resources` (pg/cpu/capacitor), `defense` (shield/armor/hull HP with per-damage-type resists, sig radius), `mobility` (incl. `agility`, the inertia time constant in seconds), `price` (ISK, consumed by the market), and `drone_bandwidth`/`drone_bay` (Mbit / m³, see the Drones paragraph below — every ship carries these fields, most at 0/0). Modules have `slot` (which rack), `cost` ({pg, cpu, cap}), `tier` ('T1'/'Meta'/'T2'), `price`, and type-specific `stats`. Passive modules (armor plates, damage amps, shield extenders, Damage Control, Co-Processor…) instead set `passive: true` and a `modifiers` array of `{ stat, op: 'add'|'mult', value }` consumed by `getEffectiveStats` (see below); optionally `maxPerShip` (Damage Control is limited to 1). Either a ship or a module may carry `requiredSkills` (Catalyst/Cormorant/Algos need `destroyers: 1`; Algos additionally needs `drones: 1`; every T2 module needs its matching weapon/support skill at 4 or 5 — as of v0.8 this includes the two v0.6 T2 hybrid/missile damage amps, `light_neutron_blaster_ii` and `ballistic_control_system_ii`, which originally shipped without a gate) — see Game state above for how that's enforced. MarketWindow only lists `tier: 'T1'` modules for purchase — Meta and T2 are loot-only (see below) but remain sellable at 50% of `price`. Eight hulls exist: Incursus/Kestrel/Merlin/Atron/Tristan (`class: 'Frigate'`, all but Tristan ungated) and Catalyst/Cormorant/Algos (`class: 'Destroyer'`, 8/2/3, 7/4/2, and 4/4/4 slots respectively, all gated) — Catalyst is a close-orbit blaster boat, Cormorant (v0.8) its long-range railgun mirror, Tristan/Algos (v0.12) trade high slots for a `drone_bay` (see Drones below). `docs/balance.md` records the numeric anchors (EHP/speed/price bands, tier strength ladder, faction resist-signature convention) new ship/module/NPC content should be derived from.

**Ammunition (`src/data/ammo.js`, v0.11):** all 10 weapon modules (6 hybrid turrets, 2 rocket launchers, 2 light missile launchers) carry an `ammoFamily` string (`'hybrid_s'` / `'rocket_s'` / `'light_missile'`); `AMMO` holds 11 T1-only round definitions keyed by id, each shaped either `{ charge: { damage_mult, optimal_mult } }` (3 hybrid charges — Antimatter/Lead/Iridium) or `{ warhead: { damage_type, explosion_radius, explosion_velocity } }` (4 damage-type warheads × 2 missile families — Mjolnir/Inferno/Scourge/Nova). Ammo is **not a module**: no `cost`/`modifiers`, never enters `inventory` or `fittedModules`. `AMMO_LOT` (100) is the market/loot trade unit; `DEFAULT_AMMO` maps each family to its fit-time default (Lead / Scourge Rocket / Scourge Light Missile); `STARTER_CARGO` and `AMMO_SAFETY_QTY` back the new-save gift and soft-lock top-up (see Game state above). `src/lib/ammo.js` holds the pure helpers everything else shares: `applyAmmoToWeapon` (fire-time correction, see Combat below), `compatibleAmmo` (family filter, the FittingWindow/BattleScene selector data source), `defaultAmmoIdFor`, and `ammoReadiness` (UNDOCK warning + FittingWindow red-text shared logic). **`src/data/blueprints.js`** (v0.11) holds `BLUEPRINTS`: the original 2 module blueprints plus 11 ammo blueprints (`produces` an `AMMO` id, `qty: 200`, `cost` = exactly 75% of market per-round price × qty) — `manufacture` (gameStore) routes by whether `produces` resolves in `MODULES` (→ inventory) or `AMMO` (→ cargo). Drone blueprints are out of scope for v0.12 (market + loot only, see below) — `manufacture`'s two-way route is untouched.

**Drones (`src/data/drones.js`, v0.12):** `DRONES` holds 12 light scout drone definitions keyed by id — 4 T1 (Hobgoblin/Hornet/Warrior/Acolyte, one per damage type, the same "one frame, damage type is the only combat difference" convention as v0.11's four missile warheads) plus 8 loot-only tier variants (4 'Integrated' Meta at ×1.10 dmg/HP, 4 'II' T2 at ×1.20, gated into `rollLoot` by the same `TIER_UNLOCKS` a module uses). A drone is **not a module** (no `cost`/`modifiers`, never enters `inventory`/`fittedModules`) and **not a ship** but reuses both shapes: `defense` is a three-layer `{ shield, armor, hull }` block (all-zero resists on T1/Meta/T2) so `applyDamage` works unmodified, and `weapon` is a turret-shaped `{ damage, rof, optimal, falloff, tracking }` block so `rollTurretShot` works unmodified — this is why drones needed zero new entries in `combat.js`. Also on each entry: `volume`/`bandwidth` (the `drone_bay`/`drone_bandwidth` budgets a ship spends to carry/field it), `sig_radius`/`speed`/`orbit_range`/`orbit_speed`/`agility` (its own movement stats, distinct from the ship's). `src/lib/drones.js` holds the three pure limit helpers (mirrors `src/lib/ammo.js`'s layer/convention): `maxDronesInSpace(ship, skills)` = `min(floor(drone_bandwidth / 5), skills.drones)`, `droneBayUsed(droneIds)`, `canLoadDrone(ship, bayIds, droneId)` — shared by gameStore (`loadDrone`), FittingWindow (bay UI), and BattleScene (launch cap), so the limit matrix is defined exactly once.

`npcs.js` defines ten enemy archetypes across five factions (two ships each): Guristas (baseline missile boat, railgun sniper — both weak to kinetic), Serpentis (close-orbit blaster brawler, EWAR scout — both weak to thermal), Sansha's Nation (mid-range "laser" slaver, close-orbit ravager — both weak to EM), Angel Cartel (rocket boat with a stasis web, and the fastest hull in the game — both weak to explosive), and Rogue Drones (v0.12 FR-6 — a mid-range blaster swarm and a missile swarm, both weak to EM, deliberately reusing Sansha's exact resist shape since a fifth faction necessarily repeats one of the four damage types). All ten now carry an authored resist signature (weak type 20–30 points below the other three, across all three layers, per `docs/balance.md`) — the original four (v0.4) were retrofitted in v0.8 to close the gap with the four v0.7 NPCs that launched with one. Every NPC carries a `faction` field (read by MapScreen's intel display) and a `weapon.type` that reuses the two existing branches (`hybrid_weapon`/`missile_weapon`) — no new combat-loop type was introduced for any of them, including the speed-focused Angel Hunter and the Rogue Drones pair, whose distinctiveness comes entirely from base stats (speed, sig, orbit range), not a new AI command. **NPC capacitor (v0.10):** every NPC now carries `cap_capacity`/`cap_recharge`, `weapon.stats.cap_use` (rocket-armed NPCs are `cap_use: 0` — cap-free on both sides, so they're structurally immune to neutralizer suppression; this now includes `rogue_alvum`), and — where an `ewar` block exists — `activation_time`/`cap_use` on that block too; see Combat below for how the sim spends and gates on it. The four rocket-armed NPCs (`guristas_frigate`, `angel_webber`, `angel_hunter`, `rogue_alvum`) also carry `weapon.stats.explosion_radius`/`explosion_velocity` (35/150), calibrated so a stationary player of any hull takes exactly full missile damage — the same fields, unmodified, are what BattleScene's enemy-fire-switching target resolution feeds a locked-on drone (see Combat below) instead of the player. None of this is scaled by depth/elite in `buildEncounter` (same rule as EWAR strength — see below). `ENCOUNTER_UNLOCKS` gates archetypes by depth and carries a `weight` per entry; `pickWeightedNpc(depth, rng?)` does a weighted draw over the currently-unlocked pool (so e.g. the depth-1 baseline Guristas frigate stays the single most likely pull even once every archetype has unlocked), and `rollEncounter(depth, nodeType, rng?)` wraps it with `buildEncounter`: HP and damage by depth (`1 + 0.15×(depth−1)`, elite nodes ×1.5), ISK reward (elite ×2), and `spReward` (v0.8; same depth/elite scaling, base 200 — see `docs/balance.md` "SP economy"); EWAR strength (and now capacitor) is not scaled. `availableNpcs(depth)` still returns the plain unweighted array (unrelated call sites/tests that only need "what's unlocked" aren't affected by the weighting). Encounters are rolled in App's `goToMap` and passed down as props. Each NPC also carries a `lootTable`, transparently spread through `buildEncounter` onto the encounter object; `rollLoot(encounter, depth, nodeType, rng?)` in `src/lib/loot.js` does one independent roll per entry (elite rolls the whole table twice) and now handles **three entry shapes**: `{ moduleId, chance }` (unchanged since v0.6 — a hit pushes the bare string id, skipping entries whose module `tier` hasn't unlocked yet per `TIER_UNLOCKS`: T1 always, Meta at depth 4+, T2 at depth 7+), `{ ammoId, qty, chance }` (v0.11 — a hit pushes `{ ammoId, qty }`; ammo is all T1 and never tier-gated), and `{ droneId, qty, chance }` (v0.12 — a hit pushes `{ droneId, qty }`; drones ARE tier-gated, same rule as modules, since Meta/T2 drones are a real strength ladder rather than a flavor swap). Every NPC table now carries exactly one ammo bundle entry (50–80 rounds at 15–20%, the faction's counter-ammo — e.g. both Sansha NPCs drop Mjolnir) alongside its unchanged module entries; the two Sansha NPCs' tables also still carry the Meta energy neutralizer. The two Rogue Drones tables each carry two drone bundle entries (one T1, one Meta/T2 — "drones drop drones") plus their own counter-ammo bundle, at the same 0.35–0.45 module-EV band as every other table (drone/ammo bundles are a supplement, not counted in that band, per the v0.11 convention). `rollLoot`'s output is therefore a **mixed array** of strings, `{ ammoId, qty }` and `{ droneId, qty }` objects — `gameStore.addLoot` and the three loot-display call sites (BattleScene's victory modal, MapScreen's `NoticeOverlay`, App's peaceful-wreck path) all branch on shape to route/render each. `src/lib/tiers.js` holds the one `TIER_COLORS` lookup shared by MarketWindow, BattleScene's loot display, and FittingWindow's lock hints.

A module's `type` string ('hybrid_weapon', 'missile_weapon', 'shield_repair', 'armor_repair', 'propulsion', 'ewar', 'energy_neut') drives its combat behavior via branches in BattleScene's module-processing loop — adding a new active module type means adding a branch there. **Energy neutralizer (v0.10):** `energy_neut` (`small_energy_neutralizer_i`, T1/market; `small_infectious_scoped_energy_neutralizer`, Meta/loot-only via both Sansha tables) burns the target's capacitor by `stats.neut_amount` per `stats.activation_time`-second cycle at heavy self capacitor cost — the one new type branch that version added. Passive modules (`passive: true`) never reach this loop at all — `activeModules()` filters them out before `modulesStateRef` is built, so a new passive module is pure data (a `modifiers` entry), zero engine changes. **Ammunition (v0.11) and Drones (v0.12) both add no new module type or branch**: ammo is a correction applied to the existing `hybrid_weapon`/`missile_weapon` branches at fire time, and a drone is not a module at all — it never enters `modulesStateRef`/`activeModules()`, it gets its own array (`dronesRef`) and its own frame-loop stage (see Combat below). This module-type-branch set (seven entries) is a standing audit invariant across both versions' plans: grep it and it hasn't grown since v0.10.

### Combat (src/components/BattleScene.jsx)

The most complex file. Real-time sim inside a react-three-fiber `<Canvas>`, updated by a `useFrame` loop. Critical pattern: all simulation state lives in **refs** (positions, velocities, capacitor, shield/armor/hull in `playerHp`/`enemyHp`, module cooldown timers in `modulesStateRef`) and is mutated every frame; React state is only synced a few times per second for the HUD, plus a `setForceRender` call when module active-states change. Don't move sim state into React state. `PhysicsUpdater` is a module-level component that runs `stepRef.current()` each frame — the step closure is reassigned every render so it sees fresh state, while the component itself never remounts; movement commands reach the sim through `commandRef`.

The pure combat math lives in `src/lib/combat.js` (capacitor recharge curve, turret hit-chance formula, `rollTurretShot` — one roll doubles as the 1%-wrecking-shot quality roll, `applyDamage` with per-layer resists and one-layer bleed-through, `missileDamageFactor`, `npcActivationFloor`, movement command vectors, inertia) and is shared by both sides of the sim and by tests.

Implements EVE mechanics: capacitor recharge, turret hit chance (optimal/falloff distance penalty + tracking vs. relative angular velocity — symmetric, so both sides reuse `angularVelocity`), wrecking shots, per-layer damage resists, inertia (`agility` as time constant), missile range check (holds the cycle instead of wasting it), and approach / orbit / keep-at-range / stop commands. The player's max HP, capacitor, speed, agility, sig radius, and per-weapon-type damage multiplier are all read from `getEffectiveStats` (computed once per render via `useMemo`, not re-derived in the frame loop) — this is where passive-module bonuses and skills reach combat. The enemy comes from the pre-rolled `encounter` prop (or `rollEncounter(depth prop ?? store deadspaceDepth, nodeType)` as a direct-entry fallback) and orbits at `ai.orbitRange`; its weapon branches on `weapon.type` like the player's module loop (turrets roll to hit against the player's **effective signature** — MWD `sig_multiplier` bloom applies on top of any static sig modifier from passives like Shield Extender; missiles always hit in range). Damage the enemy deals is resolved against the player's effective (not base) resists, so Damage Control/EANM actually mitigate incoming damage. An enemy `ewar` block webs the player inside its envelope; the player's web halves enemy speed.

**Missile damage factor (v0.10):** `missileDamageFactor(targetSig, targetSpeed, explosionRadius, explosionVelocity)` (`src/lib/combat.js`, `MISSILE_DRF = 0.5`) replaces "missiles always deal full damage in range" with `min(1, (sig/expR) × min(1, expV/v)^drf)` — a fast, small-signature target takes measurably less missile damage, and MWD's signature bloom (`sig_multiplier`) restores full damage since it inflates the sig term faster than the capped velocity term can shrink it. Both sides call it: the player's missile branch feeds it the enemy's (unmodified) sig/speed, the enemy's feeds it the player's **effective signature** (`eff.defense.sig_radius × playerSigMult`, the same MWD-bloomed value the enemy's turrets already use) and `playerVel`. Fields default to 1 (full damage) if a weapon has no `explosion_radius`/`explosion_velocity` — back-compat for any future weapon that doesn't define them.

**Enemy capacitor & energy neutralizers (v0.10):** the enemy now runs the same `getCapacitorRecharge` curve as the player, in a plain ref (`enemyCapRef`, mounts full every fight, never part of the HP snapshot) that's skipped entirely for the `Infinity` fallback an NPC without `cap_capacity` gets. Its weapon and `ewar` block each carry a `cap_use`; whenever one is due to activate, it fires (and pays) only if `enemyCapRef.current >= npcActivationFloor(enemy.cap_capacity, cap_use)` (`max(cap_use, 10% of capacity)`, `src/lib/combat.js`) — below that floor the module is fully suppressed (a `powered` ref flips false/true and logs the transition) until recharge crosses back over it. Rocket-armed NPCs (`cap_use: 0`) never enter this check, so they're immune to neutralizer pressure. The player's `energy_neut` module (BattleScene's module loop) burns `stats.neut_amount` straight off `enemyCapRef` every cycle it's in range of `stats.optimal` — the only mechanism that can drive an NPC below its own activation floor, since standing NPC consumption is calibrated (see `docs/balance.md`) to never sag there on its own.

**Ammunition (v0.11):** `modulesStateRef`'s high-slot entries are now built with their original index preserved (`activeModules()` alone flattens high/mid/low and loses it), so a weapon carries `{ ammoId, highIndex, reload }` alongside the usual `{ active, timer }` — `ammoId` seeds from `activeShip.ammo[highIndex]`. `ammoPoolRef` snapshots the whole `cargo` store on mount (undocking carries the hangar's entire stock, no magazine/capacity concept) and is shared across weapons on the same round type; `ammoUsedRef` accumulates total fired per ammo id for battle-end settlement. Every frame, `mod.reload` ticks down **unconditionally**, before the `!mod.active` early-return — this is what stops a "reload, toggle off, toggle back on" exploit from zeroing the countdown, since `toggleModule` never touches `reload`. A weapon's fire gate (after the missile/neut range holds, before the capacitor check) is: `reload > 0` → retry in 0.5s; no `ammoId` or an empty pool → deactivate with an "Out of ammunition!" log line and `setForceRender` (mirrors the capacitor-depletion auto-shutoff immediately below it). A shot that clears the gate computes `fireStats = applyAmmoToWeapon(mod.stats, AMMO[mod.ammoId])` (`src/lib/ammo.js` — hybrid charges multiply `damage`/`optimal`; missile warheads replace `damage` with a single type at the same total plus their own `explosion_radius`/`velocity`) and feeds `fireStats` into `rollTurretShot`/`missileDamageFactor`/`applyDamage` in place of `mod.stats`, decrementing `ammoPoolRef`/incrementing `ammoUsedRef` by one regardless of hit or miss; the missile range hold still reads the launcher's own unmodified `mod.stats.range`. Reloading (`changeAmmo`, wired to a `<select>` under each weapon in the module rack) sets `mod.reload = RELOAD_TIME` (10s) — no ammo cost, pure time. `toggleModule` gets a matching pre-activation gate (no ammo assigned/loaded → refuse with a log line, same shape as its existing capacitor check). Ammo corrections are applied **downstream of `getEffectiveStats`**, so they never enter the v0.10 stacking-penalty pipeline and never touch `damageMult`.

**Drones (v0.12):** the engine's first multi-entity dimension, deliberately scoped to a single new stage plus one target-resolution addition inside the existing enemy weapon block — the frame loop's other six stages, `combat.js`, and `shipStats.js` are all zero-diff. `dronesRef` (a plain array ref, mirrors `modulesStateRef`'s "all sim state lives in refs" rule) is rebuilt from `activeShip.drones` every mount — each entry `{ droneId, def, hp, pos, vel, state: 'bay'|'out'|'returning', fireTimer, alive }` always starts full HP and `'bay'` (cross-node persistence is by survival only, not HP — the strategic drone resource is count, not health; a repair node's `playerHp` reset knows nothing about drones and doesn't need to). `enemyAggroRef` (`{ droneIdx, timer, checkTimer }`) tracks the enemy's current fire-switch target. `launchDrones()`/`recallDrones()` are full-squadron commands living beside `commandRef`, mirroring `startAlign`/`cancelAlign`'s "independent entry point" pattern rather than going through it — LAUNCH tops the in-space count up to `maxDronesInSpace(activeShip, skills)` (topping up, not launch-all, is what makes Algos' bay-40 backup drones meaningful after a mid-fight loss), RECALL flips every `'out'` drone to `'returning'`. **Stage 2.5** (inserted between the player module `forEach` and the victory check, so a drone kill that breaks the enemy's hull is caught by the SAME unmodified victory check) drives each non-bay drone: approach at `speed` outside `DRONE_ENGAGE_RANGE` (1.5 km), else orbit at `orbit_range`/`orbit_speed` (the engage slowdown is what makes a drone hittable at all) via the same `desiredVelocity`/`applyInertia` the player/enemy use; fires via the same `rollTurretShot`/`applyDamage` the player's turrets use, with **no `damageMult`** (drones don't consume gunnery or damage-amp bonuses — their growth axes are count and tier, not a percent pipeline) and its own relative-angular-velocity calc (same tangent·relVel/dist formula as stage 1, evaluated drone-vs-enemy). A `'returning'` drone flies straight at the player and self-transitions to `'bay'` inside `DRONE_DOCK_RANGE`. **Enemy fire switching** is bookkept every frame just before the enemy weapon block (`DRONE_AGGRO_PERIOD` 6s cadence, `DRONE_AGGRO_CHANCE` 25% redirect roll, `DRONE_AGGRO_DURATION` 10s lock, auto-released early if the target dies or docks) and resolved as **target substitution inside that block, not a new one**: the existing cap/activation-floor gate runs first unmodified, and only once a shot clears it does target resolution pick the locked drone's own distance/`sig_radius`/velocity (defaulting to the player's, byte-identical to pre-v0.12 behavior when no drone is locked) before calling the same `rollTurretShot`/`missileDamageFactor`/`applyDamage`. A drone killed this way flips `alive = false` and immediately releases the aggro lock. The squadron HUD bar (`droneHud` state, synced at the same 0.15s throttle as the rest of the HUD via `syncHud`, plus immediately on launch/recall/dock/kill) renders nothing when the manifest is empty — a zero-drone fight's UI is pixel-identical to pre-v0.12.

**Outcomes (v0.9, ammo settlement added v0.11, drone settlement added v0.12): victory / defeat / retreat.** Battle ends when either side's hull reaches 0, or when the ALIGN & WARP retreat timer completes (`battleOverRef` freezes the sim in all three cases). **HP persists across nodes within a segment**: `initialHp` prop (`null` = full) is resolved via `resumeHp(initialHp, eff.defense)` (`src/lib/combat.js`) into both the HUD's initial state and the `playerHp` ref — capacitor is never part of this snapshot and always starts full. `depth` prop is the node's own depth (segment start + layer), read instead of the store's `deadspaceDepth` for both the fallback encounter roll and the victory-time `rollLoot` call, so a segment's later layers roll loot at their own (deeper) tier gates rather than the segment's start depth. `introLog` optionally overrides the opening combat-log line (used for the "Ambush!" wreck-fight text); `isFinalNode` swaps the victory button's label. Victory rolls loot via `rollLoot` immediately (into React state, not on button click, so the modal can't be re-rolled by dismissing it) and shows it in the victory modal alongside the bounty, `enemy.spReward`, (v0.11) an "Expended: ..." ammo-consumption line, and (v0.12) a "Drones lost: ..." line (read directly from `dronesRef.current` at render time — safe because it's frozen once `battleOverRef` flips true, same reasoning as reading `ammoUsedRef.current` directly below — hidden entirely when nothing died, mirroring the ammo line); confirming pays all five (`addIsk` + `addSp` + `addLoot` + `settleBattleAmmo({ consumed: ammoUsedRef.current, assignments: buildAmmoAssignments() })` + `settleBattleDrones({ lost: dronesRef.current.filter(d => !d.alive).map(d => d.droneId) })`) and calls `onVictory({ ...playerHp.current })` — **BattleScene no longer advances depth itself** (`advanceDepth` is gone); that's now App's job, only on the segment's final node, via `advanceSegment()`. Defeat calls `settleBattleAmmo({ consumed: ammoUsedRef.current })` (consumption only — no `assignments`, since the destroyed hull's fit has nothing left to reassign) then `shipDestroyed()`, still paying no SP — **and never calls `settleBattleDrones`**: the destroyed hull's entire manifest (bay + in-space) is lost with it via `shipDestroyed()`'s existing `activeShip` discard, so there's nothing left to individually settle (see Game state above). **ALIGN & WARP (retreat):** a TACTICAL COMMANDS button starts an `ALIGN_TIME` (8s) countdown held in `alignRef` (the only new simulation state — a plain ref decremented in the frame loop, no new React state driving the sim); it's fully cancellable, and while it counts down the player stays fully vulnerable and modules/enemy AI keep running normally (no safety valve — a web makes the 8s lethal up close, which is the intended counterplay); drones keep fighting and can keep dying during the countdown too, no special-casing. Completion calls `endBattle('retreat')`, which — like defeat — rolls no loot; the retreat modal's confirm calls `settleBattleAmmo({ consumed, assignments })` then `settleBattleDrones({ lost })` then `onRetreat({ ...playerHp.current })` — **v0.9's "no store writes" is now "no store writes beyond ammo consumption and drone losses"** (App still alone decides whether the node closes into a dead end; bounty/SP/salvage stay forfeit). Surviving drones — including any still `'returning'` mid-transit — all count as recovered on retreat with no distance/time condition; only `alive === false` counts as lost. A same-frame race between hull-zero and align-completion resolves in favor of victory/defeat, since those checks run first in the frame step and return early.

**Units convention:** data files use EVE units (meters for ranges, m/s for speed); the 3D scene uses world units where 1 unit = 1 km. Hence `dist * 1000` when comparing scene distance to module optimal/falloff, and `base_speed / 1000` for scene movement speed (0.315 u/s for a 315 m/s frigate). Angular velocity is therefore in real rad/s, which the hit formula's `×40000` factor is calibrated for. Ships spawn 4 km apart; ship meshes are rendered at an exaggerated scale (0.25 group scale) for visibility.

## Current gaps (intentional, MVP in progress)

- Run mid-state (segment map, position, HP) is intentionally not persisted (v0.9) — a page refresh discards the in-progress dive (banked ISK/SP/loot are unaffected, since those settle into the store per node) rather than being restored. Revisit only if playtesting shows refresh-loss is a frequent frustration, not a rare accident.
- No player-side counter to being neutralized (cap injector/battery) and no NPC-side active neutralizer use (v0.10) — deferred until the strength of a player neut build is observed in play.
- No NPC active repair modules (v0.10 gives NPCs a capacitor budget that could support one, but none is wired up yet).
- NPCs never consume ammo (v0.11) — enemy weapons still read `enemy.weapon.stats` directly and always have unlimited rounds; this is an explicit PRD non-goal, not an oversight.
- No cargo hold volume/m³ cap (v0.11) — a run carries the hangar's entire ammo stock automatically, with no capacity tradeoff. Deferred until the core load-ammo-and-fight loop has been observed in play.
- No T2 or faction ammo variants (v0.11) — every round is T1; a second damage/range tradeoff axis (EVE's Javelin/Rage-style ammo) is explicitly out of scope for this version.
- Ammo only moves in/out of cargo at the station (market buy/sell, INDUSTRY manufacture, loot) or via BattleScene's in-battle reload select — there is no mid-dive resupply node.
- No enemy drones and no multi-enemy encounters (v0.12) — the enemy side stays single-entity; multi-entity engine experience was deliberately banked on the player side first.
- No medium/heavy/sentry/repair/EWAR drones (v0.12) — every drone is a light scout frame; the four T1/Meta/T2 damage-type variants share one frame on purpose.
- No single-drone micro (v0.12) — LAUNCH/RECALL are full-squadron commands, no focus-fire/spread/passive-orbit per-drone control.
- No drone-boosting modules (v0.12) — damage amplifiers etc. for drones are deferred until a drone build's shape is observed in play, same "answer-type module" convention v0.4's non-goals used.
- Drones consume no ammo and drone damage takes no gunnery/damage-amp multiplier (v0.12, both explicit non-goals) — a drone's growth axes are count (Drones skill) and tier (T1/Meta/T2), not a percent pipeline.
- Drone HP resets to full between nodes (v0.12) — the strategic drone resource is count, not HP (see Combat below); a candidate valve (carry hull-layer drone damage across nodes) is recorded in `docs/balance.md`/the v0.12 plan but not implemented.
