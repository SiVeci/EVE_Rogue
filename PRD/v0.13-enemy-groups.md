# PRD v0.13 — 多敌人遭遇与敌方编队

## 概述

敌侧从单实体扩展为 **1–3 实体编队**：深层节点可遭遇多敌组合（狙击手 + 近战、EWAR 支援 + 输出），节点 intel 完整展示编队构成，玩家获得目标选择与切换能力。这是 v0.12 玩家侧多实体引擎（`dronesRef` 数组迭代、转火目标解析、编队 HUD）在敌侧的对称落地，也是 Boss 节点（v0.14）的引擎前置。

## 背景与动机

- v0.12 非目标明文记录"敌方使用无人机、多敌人编队——敌侧保持单实体；多实体引擎经验先在玩家侧攒"。经验现已攒完：帧循环数组迭代、多实体命中结算、转火 bookkeeping、编队 HUD 全部有玩家侧先例，敌侧扩展不需要发明任何新原语。
- 深度缩放目前只有一根轴：`depthMult = 1 + 0.15×(depth−1)` 数值乘子——越深只是"同一个敌人更硬更痛"，形态不变。编队规模是第二根深度轴：数值膨胀换成遭遇形态变化，深层的"更难"变得可阅读、可决策。
- EVE 经典遭遇形态（webber 支援 + 主力输出、狙击掩护 + 近战逼近）在单实体下无法表达；v0.7 建成的十个 NPC 原型两两组合即是现成的内容池——**零新 NPC 数据也能产出新遭遇**。
- 依赖：v0.12（多实体引擎与转火模式）；v0.10 的敌方电容/激活地板天然按成员独立运行，无需改动。

## 功能需求

- **FR-1 编队遭遇生成**：`rollEncounter` 输出从单 NPC 扩展为 `{ members: npc[], ... }` 聚合对象；编队规模按深度门控（d1–5 恒单敌——现状带零变化；d6+ 解锁双敌、d11+ 解锁三敌，出现权重入 balance.md，起步锚点：解锁深度上多敌占比 ~25–35%）。成员组合从当深已解锁池加权抽取，**支援型（EWAR/狙击）只出现在编队第二、三位**，主位始终是输出型。
- **FR-2 编队奖励与强度预算**：编队总赏金/SP/掉落 = 各成员求和，但成员按"编队席位系数"减配生成（双敌各 ×0.65、三敌各 ×0.5，乘在 HP/伤害上，赏金/SP 同系数）——编队总强度 ≈ 同深单敌 ×1.3/×1.5，总奖励同比例放大。防止"双敌 = 双倍难度同倍奖励"的负激励。系数与总强度带入 balance.md。
- **FR-3 战斗引擎多敌**：`enemyHp`/`enemyCapRef`/AI 状态等全部数组化（镜像 `dronesRef` 模式）；每个成员独立跑现有全套 AI（orbitRange 环绕、武器分支、电容 floor 门控、EWAR、对无人机转火）。**单成员编队与 v0.12 现状逐拍零回归**是本版的基准测试（同 v0.12"空数组零回归"惯例）。
- **FR-4 目标选择**：玩家有唯一主目标（`targetRef`）：武器与无人机全部结算到主目标；HUD 敌方列表（成员名/HP 条/派系徽标）点击切换，3D 场景点选敌舰同效；主目标死亡自动顺延下一个存活成员。不做分火。
- **FR-5 编队 intel**："所见即所遇"约定延伸到编队：MapScreen 节点显示完整成员列表（各自派系、伤害类型徽标、武器射程、赏金）——多敌节点的规避/配置决策必须在出击前可做。
- **FR-6 击杀结算**：胜利 = 全灭，结算总奖励（沿用一次性 victory modal）。撤退时**已击毁成员的赏金/SP 保留入账**（击毁即锁定，撤退确认时随弹药/无人机结算一并写入），掉落仍只在全灭后 roll、撤退全弃——部分击杀后撤退不再是全损，但"salvage 需要打完"保住了全灭的独特激励。

## 非目标

- Boss 节点与守门人（v0.14）。
- 敌方增援/波次刷新（编队成员在节点生成时定死——"所见即所遇"不许变卦）。
- 玩家分火/多目标同时锁定（唯一主目标）。
- 敌方协同 AI（集火指挥、阵型、互相修理）——各成员独立 AI 即可，协同性由组合设计表达。
- 四实体以上编队（HUD 与战场读性封顶三敌）。

## 技术方案要点

- `buildEncounter`/`rollEncounter`（`src/data/npcs.js`）输出 shape 变更是本版最大接口面：App `goToMap` 预掷、MapScreen intel、BattleScene 全部调用点同步迁移；`runmap.js` 预掷的 `encounter`/`ambush` 字段透传不变。建议聚合对象保留 `reward`/`spReward` 总和字段，减少调用点求和逻辑。
- BattleScene：敌侧 refs 数组化 + 帧循环 `forEach`（镜像 stage 2.5 的无人机迭代）；胜利判定从"敌 hull 0"改为"全员 hull 0"；`enemyAggroRef` 转火 bookkeeping 每成员独立。撤退/阵亡路径不变。
- 击杀锁定：新增 `killedRef` 累积已击毁成员的赏金/SP，三种 outcome 的确认按钮统一走既有结算惯例（victory 全额、retreat 按 `killedRef`、defeat 全弃）——与 `settleBattleAmmo`/`settleBattleDrones` 同位置、同模式。
- 存档零影响：encounter 从不持久化（runState 本就不持久化），store 无新 key。
- 无人机交互免费：敌方成员对无人机的转火（v0.12）逐成员独立跑即可；玩家无人机跟随主目标切换。
- 验证：Node 测试覆盖编队生成深度门控/席位系数/奖励求和、多敌 TTK 落带（席位系数下双敌总 TTK ≈ 单敌 ×1.3–1.6）、单成员逐拍零回归、撤退部分结算。

## 验收标准

- [ ] d1–5 遭遇恒为单敌且与 v0.12 现状逐拍零回归；d6+ 出现双敌节点且 intel 预先可见全部成员。
- [ ] 双敌战斗：点击 HUD 列表/敌舰切换主目标，武器与无人机火力跟随；击毁一员后自动顺延。
- [ ] 全灭结算总赏金/SP/掉落 = 成员求和；双敌总奖励 ≈ 同深单敌 ×1.3（席位系数成立）。
- [ ] 击毁 1/2 后撤退：已击毁成员赏金/SP 入账、掉落全弃、节点封闭——v0.9 撤退语义仅此一处放宽。
- [ ] webber + 狙击组合：web 成员先死后玩家恢复机动、狙击命中率随角速度回升下降（组合协同来自既有公式，无专用逻辑）。
- [ ] 敌方成员对无人机转火逐成员独立发生；被中和的成员武器停转不影响其僚机。

## 风险与开放问题

- **多敌 DPS 叠加是坦克带杀手**：席位系数是总阀门，起步 ×0.65/×0.5 是拍的——若护卫级在双敌下无生存空间，先调系数再考虑改编队解锁深度。红线：编队总 DPS ≤ 同深单敌 ×1.5。
- **BattleScene 复杂度**：敌侧数组化后该文件承载四类实体逻辑（玩家/敌编队/无人机/结算）——若本版实施中膨胀失控，把敌 AI 步进抽到 `src/lib/enemyAi.js` 纯函数层（输入实体状态、输出动作），作为本版内允许的顺路重构。
- **3D 点选目标的命中精度**（小目标、重叠）：HUD 列表是保底交互，场景点选做增强，不作为唯一路径。
- 撤退部分结算放宽是否削弱"全灭或全弃"的张力——若实测撤退率异常升高，回退方案是部分结算打折（×0.5）而非取消。
