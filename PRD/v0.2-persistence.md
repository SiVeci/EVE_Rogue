# PRD v0.2 — 存档与持久化

## 概述

将玩家状态持久化到 localStorage，使进度在刷新/关闭浏览器后保留。这是后续所有版本的地基：没有存档，任何超过一局的游玩、测试和平衡反馈都不可能。

## 背景与动机

- 当前刷新页面即清空全部进度（CLAUDE.md 中列为已知缺口）。
- 全部玩家状态已集中在单一 Zustand store（`src/store/gameStore.js`），持久化的技术条件成熟。
- 后续版本（市场、掉落）会频繁调整模块数值，存档格式必须一开始就为"数据定义变更"做好准备。

## 目标

- 刷新页面后：ISK、SP、技能等级、蓝图、机库库存、当前船体及已装配模块、`deadspaceDepth` 全部保留。
- 模块/舰船数值在版本更新后自动生效于旧存档（存档不冻结旧数值）。
- 存档损坏或版本不兼容时优雅降级为新开局，不白屏。

## 非目标

- 云存档、多存档位、导入导出。
- 手动存档按钮（自动持久化即可）。

## 功能需求

- **FR-1 自动持久化**：使用 `zustand/middleware` 的 `persist`，写入 localStorage（建议 key：`eve-rogue-save`）。所有会改变玩家状态的 action 之后自动落盘。
- **FR-2 按 id 序列化**：`inventory` 与 `activeShip.fittedModules` 当前存放完整模块对象。序列化时必须只存 **id 列表**（允许重复 id 表示多件同款），`activeShip` 只存 `shipId` + 三个槽位的 id 数组；反序列化时从 `MODULES` / `SHIPS` 重建对象。这保证平衡性调整对旧存档立即生效，也避免存档里残留已删除字段。
- **FR-3 存档版本与迁移**：存档带 `version` 字段，使用 persist 的 `migrate` 钩子。未知版本或迁移失败 → 丢弃存档并以初始状态开局（可在控制台警告）。
- **FR-4 无效引用清理**：反序列化时若某 id 在 `MODULES`/`SHIPS` 中已不存在，静默丢弃该件（而非崩溃）。
- **FR-5 重置入口**：StationHub 内提供"Reset Progress"按钮（带二次确认），清除存档并回到初始状态。开发期也需要它来快速回归测试。

## 技术方案要点

- `persist` 使用 `partialize` 只持久化玩家数据字段；`skillMult` 等派生逻辑不入档。
- 序列化/反序列化转换放在 persist 的 `storage`/`merge` 或自定义 `serialize` 层，store 运行时结构（对象引用）保持现状，**不改动** FittingWindow / BattleScene 的读取方式，控制改动面。
- 受影响文件：`src/store/gameStore.js`（主要）、`src/components/StationHub.jsx`（重置按钮）。

## 验收标准

- [ ] 装配模块 → 打两场 → 刷新页面：ISK / 深度 / 装配 / 库存完全一致。
- [ ] 手改 localStorage 为乱码后刷新：正常新开局，无报错弹窗。
- [ ] 修改某模块的 `damage` 数值后刷新旧存档：战斗使用新数值。
- [ ] Reset Progress 后回到初始状态（50,000 ISK、深度 1、初始库存）。
- [ ] `npm run lint` 通过。

## 风险与开放问题

- 重复模块以 id 重复表示，未来若模块出现"实例状态"（损耗、改装）需升级为实例对象 + 版本迁移——本版不做，但 `version` 字段为此留门。
